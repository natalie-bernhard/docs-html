安全性故障诊断

如果要查找与安全性相关的问题的解决方案，那么以下信息可帮助确定常见问题的原因以及关联的错误消息。

  • 对 ACME 证书进行故障诊断

  • LDAP 故障诊断

    • 对针对 LDAP 服务器的 Kerberos 认证进行故障诊断

  • 对 SSO 进行故障诊断

  • 对 SSL 和 TLS 进行故障诊断

  • 其他故障诊断问题

对 ACME 证书进行故障诊断

如果迂到 ACME 认证中心 (CA) 证书问题，那么可以参考 Open Liberty 服务器中的错误消息或 HTTP 错误消息来调试问题。 以下信息可帮助确定与 ACME CA 证书关联的常见问题和错误消息的原因。

要帮助对 ACME CA 证书的问题进行故障诊断，您可以使用以下跟踪规范来启用跟踪:

ACMECA=all

以下部分描述了使用 ACME CA 帐户和连接时可能迂到的常见问题。 有关更多信息，请参阅 使用 ACME 自动证书管理 和 ACME 支持 功能。

证书请求超时

如果证书请求超时，那么可以使用 challengePollTimeout 和 orderPollTimeout 属性来设置较长的超时值。

您在更新请求中接收到 HTTP 代码 429 消息

为了防止直接证书更新请求过多以及可能对服务器产生的负面影响，将在一小段时间内阻止证书更新请求。 此窗口到期后，可以发出新请求。 429 消息指示何时可以发出新请求。

您接收到指示超出速率限制的消息

某些 CA (例如 LetsEncrypt) 对请求新证书实施速率限制。 如果要在短时间内测试并请求多个证书，请使用相应的测试服务器。 例如， LetsEncrypt 提供了具有更高速率限制的登台服务器。

当证书未到期时，将在启动时更新该证书

以下条件可能会导致未到期的证书在启动时自动更新:

  • 证书标记为已撤销

  • 证书到期日期在 renewBeforeExpiration 属性设置的窗口内。

  • 目录 URI ，域或其他帐户信息已更改，需要新的证书。

  • 已使用 --clean 选项启动服务器，并且已除去有关证书的历史信息。

授权质询失败，显示 CWPKI2001E 消息

如果服务器未能访存证书，那么您可能会看到类似以下示例的错误消息:

CWPKI0804E: SSL 证书创建错误。 错误为: CWPKI2001E: 位于 http://my-configured-ca.com/directory URI 的 ACME 认证中心响应 mydomainname.com 域的授权质询失败。 提问状态为 INVALID。  错误为 "正在访存 http://mydomainname.com/.well-known/acme-challenge/FXCFcGCv4Ov2ofJ2i-PgMsO1kECwKB0XfTzsPjNIXBs: 连接已拒绝"。

如果看到此消息，请验证 CA 是否可访问所提供的域名。 查看日志并确认预期的域名或 IP 地址用于 acme-challenge Web 应用程序。 在日志中查找以下消息:

CWWKT0016I: Web 应用程序可用 (default_host): http://mydomainname.com:80/.well-known/acme-challenge/

要配置用于 Web 应用程序的主机名，请在 server.xml 文件中添加或更新 httpEndpoint 配置的 host 属性。

访存证书失败后，密钥库会生成错误

如果服务器无法访存证书，那么将创建空密钥库。 在较低版本的 Java 中，空密钥库可能会导致异常。 此错误的示例包括以下消息:

CWPKI2030E: ACME 服务无法将缺省别名下的证书安装到 defaultKeyStore 密钥库中。 错误为 "配置中不存在密钥库 [defaultKeyStore]"。` `
CWWKS9582E: 未在 10 秒内解析具有 defaultOrb 标识的 orb 元素所需的 [defaultSSLConfig] sslRef 属性。 因此，将不会启动应用程序。 确保包含 keyStore 元素并且正确配置了安全套接字层 (SSL)。 如果 sslRef 为 defaultSSLConfig，请添加标识值为 `defaultKeyStore ` 的 keyStore 元素和密码。

要在访存初始证书失败后绕过此错误，请除去空密钥库。

在撤销检查期间接收到 CWPKI2058W 警告消息

运行 ACME CA 服务器的容器化版本时，可能无法访问证书中定义的 OCSP 响应程序 URL。 您可以通过在 "acmeRevocationChecker" 元素中指定 "ocspResponderUrl" 属性来覆盖证书中的 OCSP 响应程序 URL。 如果未配置此 URL ，那么在撤销检查期间可能会发生以下警告:

CWPKI2058W: 证书撤销状态检查已忽略的软故障。 撤销检查可能未完成。 失败的有: "[java.security.cert.CertPathValidatorException: 由于网络错误无法确定撤销状态， java.security.cert.CertPathValidatorException: 由于网络错误无法确定撤销状态]"

如果看到此网络错误警告，并且您正在使用测试 CA 服务器运行，那么可以添加定制 ocspResponderUrl URL。 如果测试 CA 不支持撤销测试，那么可以通过在 acmeRevocationChecker 元素 false上设置 enabled 属性，如以下示例中所示:

<acmeCA>
   ...
   <acmeRevocationChecker enabled="false" />
</acmeCA>

LDAP 故障诊断

您收到一条 FFDC1015I 错误消息，指示无法访问已配置的 LDAP 服务器

如果无法访问已配置的 LDAP 服务器，那么可能会在 messages.log 文件中看到类似于以下示例的错误消息:

FFDC1015I: 已创建 FFDC 事件：javax.naming.ServiceUnavailableException：myldapserver.mycompany.com：636；套接字已关闭 com.ibm.ws.security.registry.ldap.internal.LdapRegistry 298

如果看到此消息，请检查 LDAP 服务器以验证其是否正在运行，以及是否可以在 Open Liberty 服务器的主机上访问 LDAP 服务器的 IP 地址。

与 LDAP 服务器的连接失败，发生 SSL 握手错误。

如果在 LDAP 服务器上启用 SSL 而不将 LDAP 服务器的签署者证书复制到信任库中，那么与 LDAP 服务器的连接将因 SSL 握手错误而失败。 信任库在 server.xml 文件的 LDAPSSLSettings 元素中引用。 您可能会看到以下消息：

javax.naming.CommunicationException：简单绑定失败：myldapserver.mycompany.com：636 [根异常是 javax.net.ssl.SSLHandshakeException：com.ibm.jsse2.util.g：PKIX 路径构建失败：java.security.cert.CertPathBuilderException：找不到所请求目标的有效认证路径]

确保将签署者证书从 LDAP 服务器复制到信任库文件中。

对于 LDAP 或联合存储库，登录，认证或授权速度较慢。

缓慢登录有多种原因。 请查看以下列表以了解一些常见原因。

  • 如果将引荐的 LDAP 配置设置为 follow，那么将对 LDAP 请求执行任何对其他 LDAP 实例的引用。 此情况可能会导致联系一个或多个 LDAP 服务器，具体取决于您的 LDAP 配置，因此将花费更多时间。 例如， LDAP1 服务器可能遵循对 LDAP2 服务器的引用，这可能遵循对 LDAP3 服务器的引用。 如果不需要从其他 LDAP 服务器获取更多信息，请将引荐设置为 ignore。

  • 一个 LDAP 服务器可能迂到可能未显示在 JVM 日志中的问题和错误。 当 LDAP 服务器与引用的 LDAP 服务器进行对话时，示例包括 TCP 读超时或 DNS 问题。 要诊断这些情况，您可以捕获包以了解正在进行的调用数，以及是否存在因 LDAP 服务器正遵循引荐而引起的任何延迟或错误。 防火墙或其他软件会关闭与 LDAP 的连接。

  • 对于联合存储库，将检查所有存储库和注册表以确保唯一用户在域中。 如果存储库或注册表正在缓慢响应，那么即使用户不在运行缓慢的注册表中，每个调用也会缓慢。 确保联合存储库中的所有参与基本条目都在迅速响应。

访问 LDAP 服务器时偶尔发生连接异常。 例如：java.net.SocketException: Connection reset

防火墙或其他软件会关闭与 LDAP 的连接。 缺省情况下，将维护 LDAP 连接池以提高性能。 如果以远程方式关闭已缓存的连接，那么会在上下文池中建立新的连接并将新连接重新放回其中。 此过程可能导致延迟，并可能导致在 JVM 日志中记录错误。 例如，您可能会收到以下错误:

java.net.SocketException: Connection reset

要避免这种情况，请将上下文池超时设置为小于远程连接关闭时间。 例如，如果防火墙在 10 分钟后关闭连接，那么可以将连接池超时设置为 9 分钟。 在遇到失败的连接时不会创建新的连接，而是检查池中连接是否到期，然后创建新的连接，跳过失败步骤。

对 LDAP 服务器的 Kerberos 认证进行故障诊断

如果在向 LDAP 服务器进行 Kerberos 认证时迂到问题，请参阅来自 Open Liberty 服务器的错误消息或 HTTP 错误消息以调试问题。 以下信息可帮助确定与 LDAP 服务器的 Kerberos 认证相关联的常见问题和错误消息的原因。

为联合用户注册表配置 Kerberos 时，性能很慢

如果联合用户注册表中的一个或多个用户注册表不可用，那么对 federatedRepository 元素启用 allowOpIfRepoDown 属性可帮助避免失败。 但是，如果在具有 krb5TicketCache 属性的 ccache 文件中指定了 Kerberos 凭证，那么此配置可能会导致整体性能降低。 当 Kerberos 凭证位于 ccache 文件中时， Open Liberty 会尝试自动更新接近到期时间或已到期的凭证。 此自动更新尝试可能会导致性能下降。

要避免此问题，可以在具有 kerberos 元素的 keytab 文件中指定 Kerberos 凭证。 keytab 文件中的凭证不会到期，因此无需自动更新。 有关更多信息，请参阅 Open Liberty的Kerberos 认证。

即使未启用Kerberos 的注册表可用，用户也无法登录

如果为 Open Liberty 服务器配置了多个用户注册表，那么所有基本用户注册表，定制用户注册表和 LDAP 用户注册表都将组合到单个联合用户注册表中。 缺省情况下，服务器必须在所有参与用户注册表中成功搜索该用户，以验证该用户在联合用户注册表中是否唯一。 如果联合注册表中启用了 Kerberos的 LDAP 服务器使用 Kerberos 凭单高速缓存来保存用户凭证，并且凭证到期，那么 LDAP 注册表的搜索将失败。 要解决此问题，请更新 Kerberos 凭单高速缓存。 例如，您可以使用 Java kinit 工具来更新 Kerberos 凭单高速缓存。

要避免在用户注册表不可用时发生故障，请在 server.xml 文件中配置 allowOpIfRepoDown 属性。 在 federatedRepository 元素的 primaryRealm 子元素上将 allowOpIfRepoDown 属性设置为 true ，如以下示例中所示:

<federatedRepository>
        <primaryRealm name="FederatedRealm" allowOpIfRepoDown="true">
            <participatingBaseEntry name="o=SampleBasicRealm"/>
            <participatingBaseEntry name="o=ibm,c=us"/>
        </primaryRealm>
</federatedRepository>

有关更多信息，请参阅 Federated User Registry 功能部件。

Kerberos 主体名称不在 Kerberos 凭单高速缓存文件中

如果在 Kerberos 凭单高速缓存文件中找不到 Kerberos 主体名称，那么 Open Liberty 会记录 CWIML 消息类型。 由于以下原因，可能会出现缺少 Kerberos 主体名称的情况:

  • 没有为 Kerberos 主体名称生成凭证，这将导致不正确的 Kerberos 配置。

  • Kerberos 凭单高速缓存包含已到期的凭证。

在任一情况下，更新 Kerberos 凭单高速缓存以解决问题。 例如，您可以使用 Java kinit 工具来更新 Kerberos 凭单高速缓存。

根据 Java SDK 的类型，您可以接收的消息类似于下列其中一个示例:

CWIML4507E: Kerberos 登录失败，返回 user1@SAMPLE.COM Kerberos 主体和 C:\krb5\krb5-user1.cc Kerberos 凭证高速缓存 (ccache)。 javax.security.auth.login.LoginException: 无法从用户获取密码

CWIML4520E: 无法完成 LDAP 操作。 处理期间发生 LDAP 命名异常 javax.naming.AuthenticationException: GSSAPI [Root 异常为 javax.security.sasl.SaslException: GSS 启动失败 [由 GSSException 导致: 未提供有效凭证 (机制级别: 凭单已到期 (32))]]。

CWIML4520E: 无法完成 LDAP 操作。 LDAP 命名异常 javax.naming.NamingException: CWIML4507E: Kerberos 登录失败，返回 user1@SAMPLE.COM Kerberos 主体和 C:\krb5\krb5-user1.cc Kerberos 凭证高速缓存 (ccache)。 javax.security.auth.login.LoginException: 无法从用户获取密码

要查看 Kerberos 主体用户的到期时间，请运行 Java klist 工具。

有关更多信息，请参阅 为 LDAP 服务器配置 Kerberos 认证。

对 SSO 进行故障诊断

单点登录 (SSO) 不起作用，因为服务器不共享全球标准时间和用户注册表

不同的 Open Liberty 服务器共享 webAppSecurity 元素的 LTPA 密钥，密码和 ssoCookieName 属性。 如果这些服务器不共享全球标准时间和用户注册表，那么 SSO 可能无法工作。 确保这些服务器共享全球标准时间和用户注册表。

系统会提示您再次输入凭证信息

如果令牌到期，那么 SSO 可能不起作用。 此外，如果在您以不一致的方式更改用户注册表之后从 Web 浏览器发送 cookie ，那么 SSO 可能会失败。 例如，修改域或除去 cookie 表示的用户。 系统可能会提示您再次输入凭证信息。 确保令牌未到期，并且您对用户注册表进行一致的更改。

对 SSL 和 TLS 进行故障诊断

在广泛采用 TLS 协议之前， SSL 是保护 Web 通信的标准协议。 随着时间的推移，针对 SSL 识别了许多安全漏洞，并且该漏洞不再被广泛使用。 但是，由于历史原因，术语 "SSL" 通常用于引用加密网络连接，即使在使用 TLS 时也是如此。 在 Open Liberty中，在功能部件和配置名称中仍使用术语 SSL ，即使 TLS 是底层协议也是如此。 实际上， SSL 现在是 TLS 的同义词。

您收到 CWWKS9105E 消息，指示无法确定 TLS 端口以进行重定向

如果尝试访问重定向到不可用的 TLS 端口的应用程序，那么可能会看到以下消息

 CWWKS9105E: 无法确定用于自动重定向的 TLS 端口。

由于缺少 TLS 配置或 TLS 配置定义中存在一些错误，因此端口可能不可用。 请检查 TLS 配置是否存在于 server.xml 文件中并且是否正确。

在没有标识字段的配置中存在密钥库元素，并为您提供 FFDC1015I 错误

当配置中存在没有标识字段的密钥库元素时，您可能会收到以下错误

FFDC1015I: 在 ffdc_12.04.18_16.09.42.0.log 中创建了 FFDC 事件：“java.lang.IllegalArgumentException：配置未知、不完整：缺少标识字段 com.ibm.ws.config.internal.cm.ManagedServiceFactoryTracker aSyncReadNupdate。
尝试读取配置并更新 ManagedServiceFactory 时抛出异常。 异常 = java.lang.IllegalArgumentException：配置未知、不完整：缺少标识字段”

当配置中存在没有标识字段的密钥库元素时，会发生此错误。 如果使用最低 TLS 配置，请将 ID 字段设置为 defaultKeyStore。

其他故障诊断问题

您收到一条 SESN0008E 消息，表明匿名用户尝试访问已认证用户拥有的会话。

当未经认证的用户尝试访问由已认证的用户创建的会话时，您可能会看到以下消息:

SESN0008E: 已认证为匿名的用户尝试访问由其拥有的会话
用户:LdapRegistry/cn=steven ， o=myCompany， c = US。

当您使用 JSP (Jakarta Server Page) (例如 login.jsp 文件) 进行表单登录并且浏览器发送的 SSO 令牌到期时，可能会发生此错误。 然后，系统会提示用户使用为 form-login 配置的 login.jsp 页面重新登录。 缺省情况下， JSP 页面尝试获取最初由令牌已到期的用户创建的会话。 因此，当您访问此会话时，不会认证用户，也不会建立凭证，这将导致此错误。

要避免此错误，请重新启动那个启动新会话的浏览器，或者将 login.jsp 文件配置成缺省情况下不创建会话。 如果选择更新 login.jsp 文件，请设置 <%@ page session="false" %>。

您收到 CWWKS9104A 消息，指示未授予用户对任何必需角色的访问权

当用户无权访问应用程序所需的角色时，您可能会看到以下消息:

CWWKS9104A: 对 {2} 调用 {1} 时，授权用户 {0} 失败。 用户不是
已授予对任何必需角色的访问权: {3}。

确保将该用户或该用户所属的组映射到错误消息中提及的至少一个角色。 ibm-application-bnd.xmi/xml 文件中所定义的用户到角色映射优先于 server.xml 文件中所定义的映射。 检查两个资源以确保定义了正确的映射。 有关更多信息，请参阅 授权。

用户授权失败

如果用户授权失败，您可能会看到以下消息:

CWWKS9104A: 对用户 {0} 授权失败。

如果对同一上下文根指定了 application 和 webApplication，那么可能发生此错误。 如果发生冲突，那么所定义的最新配置被忽略，并导致意外错误，例如，CWWKS9104A。

应用程序安装导致意外的安全行为

如果使用应用程序元素和 dropins 文件夹在 server.xml 文件中指定应用程序，那么将尝试安装应用程序两次，并且 server.xml 文件中的安全性配置可能生效，也可能不生效。

您可能会看到以下消息：

CWWKZ0013E: 无法启动名为 {0} 的两个应用程序。

此消息可能后跟意外的安全行为和错误消息，例如 CWWKS9104A。 要解决此问题，必须从 dropins 文件夹中除去应用程序并将其复制到另一个目录。 如果必须将其保留在 dropins 文件夹中，那么必须在 server.xml 文件中使用以下代码来禁用应用程序监视:

<applicationMonitor dropinsEnabled="false"/>
