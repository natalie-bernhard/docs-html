日志和跟踪配置

Open Liberty 具有统一的日志记录组件，用于处理由应用程序和运行时编写的消息，并提供首次故障数据捕获 (FFDC) 功能。 应用程序使用 System.out， System.err或 java.util.logging.Logger 流写入的日志记录数据将组合到服务器日志中。

服务器具有以下三个主日志文件:

  • console.log -此文件由 server start 命令创建。 它包含来自底层 JVM 进程的重定向标准输出和标准错误流。 此控制台输出已针对人员可读性进行格式化，并且缺少一些对于自动日志分析有用的信息。 console.log 文件会在服务器重新启动时重置。

  • messages.log - 此文件包含由日志记录组件写入或捕获的所有消息。 写入此文件的所有消息都包含其他信息，例如消息时间戳记和写入消息的线程的标识。 此文件适用于自动日志分析。 此文件不包含由 JVM 进程直接写入的消息。

  • trace.log -仅当启用跟踪时，才会创建此文件。 它包含 messages.log 文件的所有内容以及任何已启用的跟踪。 此文件不包含由 JVM 进程直接写入的消息。

日志记录配置

可以通过服务器配置来控制日志记录组件。 可以通过 logging 元素在 server.xml 文件中完全配置日志记录组件。 但是，在处理 server.xml 文件之前会初始化日志记录，因此通过 server.xml 文件配置日志记录可能会导致早期日志条目使用与后续日志条目不同的日志配置。 为了避免此问题，您可以在 boostrap.properties 文件中提供许多日志记录配置，在某些情况下，还可以使用环境变量来提供日志记录配置。 避免在 bootstrap.properties 和 server.xml 文件中为同一配置属性指定不同的值。 有关日志记录配置设置的更多信息，请参阅 按源列出的配置设置 表。

日志文件存储管理

通过将进程 stdout 和 stderr 流重定向到文件来创建 console.log 文件。 因此， Open Liberty 无法提供与 messages.log 文件相同的管理级别，例如日志回滚。 如果您关注 console.log 文件的大小，那么可以将控制台日志设置为 OFF 并改为使用 messages.log 文件。 日志记录组件写入或捕获的所有消息都将写入 messages.log 文件，并且您可以在消息日志中配置文件回滚。

但是，当您将控制台日志级别设置为 OFF时，来自底层 JVM 的消息仍将写入 console.log 文件。 同一原则适用于写入本机标准输出和标准错误流的任何消息。 Open Liberty 日志记录组件不会捕获或控制 JVM 进程直接写入的消息的目标，例如 -verbose:gc 输出。 这些消息将根据活动 JVM 设置路由到文件系统，并且不会显示在 messages.log 文件中。

要阻止日志记录组件写入控制台日志，并将 messages.log 文件配置为在 100 Mb 上滚动三次，请使用以下配置:

com.ibm.ws.logging.max.file.size=100
com.ibm.ws.logging.max.files=3
com.ibm.ws.logging.console.log.level=OFF
com.ibm.ws.logging.copy.system.streams=false

日志格式

三个 Open Liberty 主日志文件中的每个主日志文件都具有缺省日志格式以及可在日志记录配置中指定的若干备用格式。 您为每个日志选择的格式取决于应用程序和基础结构的需求。 例如，如果消息由日志分析平台使用，那么您可以选择以 JSON 格式输出消息日志。 或者，如果您在所有日志文件中需要一致的日志格式，那么可以为所有三个日志选择 TBASIC 格式。

  • console.log 文件的缺省格式为 DEV。

  • messages.log 文件的缺省格式为 SIMPLE。

  • trace.log 文件的缺省格式为 ENHANCED。

TBASIC 日志格式

TBASIC 日志记录格式在 Open Liberty 控制台，消息和跟踪日志文件中提供一致的日志格式。 跟踪日志存在 BASIC 日志记录格式。 但是，跟踪日志与控制台日志和消息日志的跟踪日志之间的输出不同。 TBASIC 格式为跟踪日志，控制台日志和消息日志提供与跟踪日志的 BASIC 选项匹配的一致格式选项。 TBASIC 格式充当 BASIC 选项的别名。

可以在 bootstrap.properties 文件中指定 TBASIC 格式，如以下示例中所示:

com.ibm.ws.logging.message.format= tbasic
com.ibm.ws.logging.console.format= tbasic
com.ibm.ws.logging.trace.format= tbasic

您还可以将消息和控制台日志的 TBASIC 日志格式指定为 server.env 文件中的环境变量，如以下示例中所示:

WLP_LOGGING_MESSAGE_FORMAT=TBASIC
WLP_LOGGING_CONSOLE_FORMAT=TBASIC

JSON 日志记录

您可以通过生成 JSON 格式的日志来简化日志解析。 JSON 是一种自我描述格式，许多日志分析工具可以使用此格式，而无需特定于格式的解析指令。 您可以通过编辑 bootstrap.properties 文件或指定环境变量来配置 Open Liberty 日志以生成 JSON 格式的日志。 以下两个示例显示了其中每个选项的配置:

  • 在 bootstrap.properties 文件中配置 JSON 日志记录:

    com.ibm.ws.logging.message.format=json
com.ibm.ws.logging.message.source=message,trace,accessLog,ffdc,audit

  • 使用 server.env 文件中的环境变量配置 JSON 日志记录:

    WLP_LOGGING_MESSAGE_FORMAT=json
WLP_LOGGING_MESSAGE_SOURCE=message,trace,accessLog,ffdc,audit

可配置的 JSON 字段名称

当日志为 JSON 格式时，您可以使用 jsonFieldMappings 属性将缺省字段名称替换为新的字段名称。 如果同一日志记录配置中的其他服务器使用的字段名称与 Open Liberty 缺省名称不同，那么可能需要替换缺省字段名称。 例如， Open Liberty 消息由 message 字段名称引用，但另一个容器中的消息可能位于名为 log的字段中。 在这种情况下，消息的两个不同可视化显示在仪表板上的日志中。 如果修改 Open Liberty 输出字段名称以使其与其他日志匹配，那么可以在同一可视化中查看这些字段。 以下示例显示了用于重命名 JSON 字段的样本配置。

  • 要配置新的字段名称，可以在 server.env 文件中包含以下环境变量:

    WLP_LOGGING_JSON_FIELD_MAPPINGS=loglevel:level

    在此示例中， loglevel 字段名称将替换为 level 字段名称。

  • 要配置特定源的字段名称，可以在 server.env 文件中包含以下环境变量:

    WLP_LOGGING_JSON_FIELD_MAPPINGS=message:message:log

    在此示例中， message 字段名称将替换为消息日志中的 log 字段名称。

要从日志中省略字段，请指定字段名称而不进行替换，如以下示例中所示:

WLP_LOGGING_JSON_FIELD_MAPPINGS=defaultField名称:

要重命名或者省略多个字段，请指定字段名称映射的逗号分隔列表。

有关缺省 JSON 字段名称的完整列表，请参阅 JSON 日志事件参考列表。

有关更多信息，请参阅 可配置的 JSON 日志字段名称。

定制消息和跟踪 JSON 字段

您可以将定制字段添加到 JSON 格式的消息和跟踪输出，以收集有关特定问题或事件的信息。 例如，如果要检查来自特定用户的请求，可以添加定制字段以按该用户的标识过滤应用程序日志。 您可以为会话标识添加另一个字段，以便可以分析和过滤特定会话的应用程序日志。

Open Liberty LogRecordContext API 可以将定制字段添加到日志和跟踪记录。 此 API 通过指定扩展的值来添加字段。 要使用 LogRecordContext API ，请首先导入 com.ibm.websphere.logging.hpel.LogRecordContext 类。 以下示例显示如何向 JSON 日志添加不同类型的定制字段。

要向应用程序日志添加字符串值字段，可以在应用程序中包含以下行:

LogRecordContext.addExtension("userName","bob");

新指定的字段将添加到在执行 addExtension 方法的同一线程上创建的日志和跟踪条目。 在此示例中，将为 bob 用户标识添加名为 userName 的定制字段。

要添加具有布尔值，浮点值，整数值或长整型值的字段，扩展名称必须包含后缀 _bool， _float， _int或 _long，如以下示例中所示:

LogRecordContext.addExtension("extensionName_bool","true");
LogRecordContext.addExtension("extensionName_int","112233");
LogRecordContext.addExtension("extensionName_float","1.2");
LogRecordContext.addExtension("extensionName_long","132");

当您指定这些后缀以添加非字符串值时，生成的 JSON 字段值不会括在日志中的引号内。 日志中只有字符串值的 JSON 字段值括在引号内。

要从日志中除去定制字段，请使用以下方法:

LogRecordContext.removeExtension(extensionName);

除去扩展后，在同一线程上生成的后续日志和跟踪的 JSON 输出不包含该字段。

可选择的 HTTP 访问日志 JSON 字段

如果日志为 JSON 格式并且指定了 accessLog 源，那么可以将缺省 HTTP 访问日志 JSON 字段替换为另一组字段。 您可以使用 jsonAccessLogFields 属性来指定访问日志是使用缺省字段还是使用 logFormat 属性指定的一组字段。 在 httpEndpoint 元素的 accessLogging 元素的 logFormat 属性中指定替换字段。

通过此配置，您可以接收在 JSON 日志中未提供的信息，例如远程用户标识，请求头等。 日志分析工具 (例如 Elastic 堆栈) 可以使用这些日志来监视服务器。 有关 HTTP 访问日志格式选项的更多信息，请参阅 HTTP 访问日志记录。

以下示例显示了 server.xml 文件中用于将缺省 HTTP 访问日志字段替换为 logFormat 属性指定的字段的配置。 必须启用 HTTP 访问日志记录 才能接收 JSON 访问日志。

<httpEndpoint httpPort="9080" httpsPort="9443" id="defaultHttpEndpoint">
    <accessLogging logFormat='%R{W} %u %{my_cookie}C %s'/>
</httpEndpoint>
<logging jsonAccessLogFields="logFormat"/>

容器映像日志记录配置

在容器化环境中，可以禁用消息日志，并使用环境变量将控制台输出格式化为 JSON ，如以下示例中所示:

WLP_LOGGING_MESSAGE_FORMAT=json
WLP_LOGGING_MESSAGE_SOURCE=
WLP_LOGGING_CONSOLE_FORMAT=json
WLP_LOGGING_CONSOLE_LOGLEVEL=info
WLP_LOGGING_CONSOLE_SOURCE=message,trace,accessLog,ffdc,audit

通过使用 -e 标志来设置环境变量，可以在运行 docker run 命令时指定此配置:

docker run -e "WLP_LOGGING_CONSOLE_SOURCE=message,trace,accessLog,ffdc"
           -e "WLP_LOGGING_CONSOLE_FORMAT=json"
           -e "WLP_LOGGING_CONSOLE_LOGLEVEL=info"
           -e "WLP_LOGGING_MESSAGE_FORMAT=json"
           -e "WLP_LOGGING_MESSAGE_SOURCE=" open-liberty

如果使用 Podman 来管理容器，请运行以下命令:

podman run -e "WLP_LOGGING_CONSOLE_SOURCE=message,trace,accessLog,ffdc"
           -e "WLP_LOGGING_CONSOLE_FORMAT=json"
           -e "WLP_LOGGING_CONSOLE_LOGLEVEL=info"
           -e "WLP_LOGGING_MESSAGE_FORMAT=json"
           -e "WLP_LOGGING_MESSAGE_SOURCE=" open-liberty

二进制日志记录

Liberty 具有高性能二进制日志格式选项，可减少写入跟踪文件所需的资源。 为了获得最佳性能，在配置二进制日志记录时，请将控制台日志级别设置为 OFF ，并 改为使用消息日志。 您可以在 bootstrap.properties 文件中配置二进制日志记录，如以下示例中所示:

websphere.log.provider=binaryLogging-1.0
com.ibm.ws.logging.console.log.level=OFF
com.ibm.ws.logging.copy.system.streams=false

但是，当您将控制台日志级别设置为 OFF时，来自底层 JVM 的其他消息 (写入本机标准输出和标准错误流) 仍会写入 console.log 文件。

可以使用 binaryLog 命令将二进制日志转换为文本文件，如名为 defaultServer的服务器的以下示例中所示:

基于时间的日志回滚

您可以通过指定日志回滚开始时间和日志回滚时间间隔持续时间来启用 Liberty 消息和跟踪日志的基于时间的定期回滚。 指定的回滚开始时间是日志首次回滚的时间。 回滚时间间隔持续时间是连续日志回滚之间的时间间隔。

例如，回滚开始时间为午夜，回滚时间间隔为 1 天的服务器每天在午夜回滚一次消息和跟踪日志。

您可以通过编辑 bootstrap.properties 文件，指定环境变量或使用服务器配置属性来配置 Open Liberty 消息和跟踪日志的回滚。 以下示例显示了其中每个选项的配置，日志回滚开始时间为午夜，日志回滚时间间隔为 1 天。

  • 以下示例在 bootstrap.properties 文件中配置基于时间的日志回滚。

    com.ibm.ws.logging.rollover.start.time=00:00
com.ibm.ws.logging.rollover.interval=1m

  • 以下示例使用 server.env 文件中的环境变量配置基于时间的日志回滚。

    WLP_LOGGING_ROLLOVER_START_TIME=00:00
WLP_LOGGING_ROLLOVER_INTERVAL=1m

  • 以下示例使用 server.xml 文件中的服务器配置属性配置基于时间的日志回滚。

    <logging rolloverStartTime="00:00" rolloverInterval="1d"/>

配置设置 (按源)

下表显示了等效的 server.xml 文件， bootstrap.properties 文件和环境变量配置以及简要描述。 首先显示应用于控制台日志的设置，然后显示应用于消息日志的设置，然后显示跟踪日志。 全局应用或应用于多个日志类型的设置在表中最后出现。 有关更多信息，请参阅 日志记录元素。

日志记录配置设置
服务器 XML 属性 引导程序属性 环境变量 描述

consoleFormat

com.ibm.ws.logging.console.format

WLP_LOGGING_CONSOLE_FORMAT

此设置指定控制台的必需格式。 有效值为 DEV， SIMPLE， JSON和 TBASIC 格式。 缺省情况下， consoleFormat 设置为 DEV。 有关更多信息，请参阅 日志格式。

consoleLogLevel

com.ibm.ws.logging.console.log.level

WLP_LOGGING_CONSOLE_LOGLEVEL

此设置控制转至控制台的消息的详细程度。 有效值为 INFO、AUDIT、WARNING、ERROR 和 OFF。 缺省值为 AUDIT。 如果与 Eclipse 开发者工具配合使用，那么必须将其设置为缺省值。

consoleSource

com.ibm.ws.logging.console.source

WLP_LOGGING_CONSOLE_SOURCE

此设置指定路由到控制台的源的逗号分隔列表。 仅当控制台格式设置为 json时才适用。 有效值为 message， trace， accessLog， ffdc和 audit。 缺省情况下， consoleSource 设置为 message。 要使用 audit 源，请启用 Liberty 审计 功能部件。 要使用 accessLog 源，需要配置 HTTP 访问日志记录。

copySystemStreams

com.ibm.ws.logging.copy.system.streams

如果此设置设置为 true，那么会将写入 System.out 和 System.err 流的消息复制到进程 stdout 和 stderr 流中，从而显示在 console.log 文件中。 如果此设置设置为 false，那么会将这些消息写入已配置的日志 (例如 messages.log 文件或 trace.log 文件) ，但这些消息不会复制到 stdout 和 stderr 中，并且不会显示在 console.log中。 缺省值为 true。

messageFileName

com.ibm.ws.logging.message.file.name

此设置指定消息日志文件的名称。 消息日志文件的缺省名称为 messages.log。 此文件始终存在，并且除了包含 System.out 和 System.err 流外，还包含 INFO 和其他 (AUDIT ， WARNING ， ERROR ， FAILURE) 消息。 此日志还包含时间戳记和发出消息的线程标识。 如果日志文件已翻转，那么先前日志文件的名称的格式为 messages_timestamp.log

messageFormat

com.ibm.ws.logging.message.format

WLP_LOGGING_MESSAGE_FORMAT

此设置指定 messages.log 文件的必需格式。 有效值为 SIMPLE， JSON或 TBASIC 格式。 缺省情况下， messageFormat 设置为 SIMPLE。 有关更多信息，请参阅 日志格式。

messageSource

com.ibm.ws.logging.message.source

WLP_LOGGING_MESSAGE_SOURCE

此设置指定路由到 messages.log 文件的以逗号分隔的源的列表。 仅当消息格式设置为 json`时，此设置才适用。 有效值为 message， trace， accessLog， ffdc和 audit。 缺省情况下， messageSource 设置为 message。 要使用 audit 源，请启用 Liberty audit-1.0 功能部件。 要使用 accessLog 源，您需要已配置 httpAccess日志记录。

suppressSensitiveTrace

此属性设置为 true时，将阻止在日志和跟踪文件中显示潜在的敏感信息。 服务器跟踪可在跟踪非类型化数据（例如，通过网络连接接收到的字节数）时公开敏感数据。 缺省值为 false。

traceFileName

com.ibm.ws.logging.trace.file.name

此设置指定跟踪日志文件的名称。 仅当启用了附加或详细跟踪时，才会创建 trace.log 文件。 stdout 识别为特殊值，并促使将跟踪定向到原始标准输出流。

traceFormat

com.ibm.ws.logging.trace.format

此设置控制跟踪日志的格式。 Liberty 的缺省格式为 ENHANCED。 您还可以使用 BASIC， TBASIC和 ADVANCED 格式。 有关更多信息，请参阅 日志格式。

traceSpecification

com.ibm.ws.logging.trace.specification

此设置用于选择性地启用跟踪。 日志详细信息级别规范采用以下格式:

component = level

组件指定为其设置级别的日志源。 组件可以是记录器名称，跟踪组或类名。 该级别通过使用下列其中一个级别来指定要为该组件输出的跟踪级别:

off， fatal， severe， warning， audit， info， config， detail， fine， finer， finest和 all。 将跟踪组件设置为 off， fatal， severe， warning， audit或 info 级别与将其设置为 info 级别的效果相同，无法禁用更高级别的 info 。

您可以提供多个以冒号分隔的日志详细信息级别规范。

星号 * 充当通配符，用于根据前缀匹配多个组件。 例如：

- * 指定在应用程序服务器中运行的所有可跟踪代码，包括产品系统代码和客户代码。

- com.ibm.ws.* 指定包名以 com.ibm.ws开头的所有类。

- com.ibm.ws.classloading.AppClassLoader 仅指定 AppClassLoader 类。

仅当一个或多个跟踪组件设置为 config 级别或更低级别时，才会创建 trace.log 文件并将日志写入其中。 有关日志记录级别的更多信息，请参阅 跟踪日志的有效日志详细信息级别。

com.ibm.ws.logging.newLogsOnStart

如果此设置在 Open Liberty 启动时设置为 true ，那么将翻转任何现有 messages.log 或 trace.log 文件，并将日志记录写入新的 messages.log 或 trace.log 文件。 如果此设置设置为 false ，那么仅当 messages.log 或 trace.log 文件达到 maxFileSize 属性指定的大小时，才会刷新这些文件。 缺省值为 true。 无法使用 server.xml 文件中的 logging 元素提供此设置，因为仅在服务器引导期间处理此设置。

hideMessage

com.ibm.ws.logging.hideMessage

您可以使用此设置来配置要从 console.log 和 messages.log 文件中隐藏的消息密钥。 当消息处于隐藏状态时，它们将重定向到 trace.log 文件。

isoDateFormat

com.ibm.ws.logging.isoDateFormat

此设置指定是否在日志文件中使用 ISO-8601 格式的日期。 缺省值为 false。

如果此设置设置为 true ，那么将在 messages.log 文件， trace.log 文件和 FFDC 日志中使用 ISO-8601 格式。 格式为 yyyy-MM-dd’T’HH:mm:ss.SSSZ。

如果指定值 false，那么会根据系统中设置的缺省语言环境设置日期和时间格式。 如果找不到缺省语言环境，那么格式为 dd/MMM/yyyy HH:mm:ss:SSS z。

maxFfdc年龄

com.ibm.ws.logging.max.ffdc.age

WLP_LOGGING_MAX_FFDC_AGE

使用此设置可根据生存期启用 FFDC 文件清除。 此设置指定在删除 FFDC 文件之前的最大文件寿命。 每天午夜都会发生清除。 指定一个正整数，后跟一个时间单位，可以是天 (d) ，小时 (h) 或分钟 (m)。 例如，将 2 天指定为 2d。 可以将多个值包括在单个条目中。 例如， 2d6h 相当于 2 天和 6 小时。

appsWriteJSON

com.ibm.ws.logging.apps.write.json

WLP_LOGGING_APPS_WRITE_JSON

当消息日志或控制台为 JSON 格式时，此设置允许应用程序将 JSON 格式的消息写入这些目标，而无需修改。

jsonAccessLogFields

com.ibm.ws.json.access.log.fields

WLP_LOGGING_JSON_ACCESS_LOG_FIELDS

当日志为 JSON 格式时，可以使用此设置将缺省 HTTP 访问日志 JSON 字段替换为由 accesLogging 元素的 logFormat 属性指定的字段。

jsonField映射

com.ibm.ws.logging.json.field.mappings

WLP_LOGGING_JSON_FIELD_MAPPstrings

当日志为 JSON 格式时，请使用此设置将缺省字段名称替换为新的字段名称，或者从日志中省略字段。 有关更多信息，请参阅 可配置的 JSON 字段名称

logDirectory

com.ibm.ws.logging.log.directory

LOG_DIR

可以使用此设置为所有日志文件 (不包括 console.log 文件) 设置目录，但包括 FFDC。 缺省值为 WLP_OUTPUT_DIR/serverName/logs。 建议不要在 server.xml 文件中设置 logDirectory ，因为这可能会导致在读取 server.xml 文件之前将某些日志数据写入缺省位置。

maxFiles

com.ibm.ws.logging.max.files

此设置指定保留每个日志文件的数量。 此设置也适用于 FFDC 的异常摘要日志数。 因此，如果此数字为 10，那么 ffdc/ 目录中可能有 10 个消息日志， 10 个跟踪日志和 10 个异常摘要。 缺省情况下，该值为 2。 控制台日志不会滚动，因此此设置不适用于 console.log 文件。

maxFileSize

com.ibm.ws.logging.max.file.size

此设置指定日志文件在滚动之前可以达到的最大大小 (MB)。 将值设置为 0 将禁用日志滚动。 缺省值为 20。 console.log 不会滚动，因此此设置不适用。

stackTraceSingleEntry

com.ibm.ws.logging.stackTraceSingleEntry

WLP_LOGGING_STACK_TRACE_SINGLE_ENTRY

当此属性设置为 true时，发出到标准输出和标准错误的堆栈跟踪将合并到单个日志事件中。 缺省情况下，此值设置为 false ，并且堆栈跟踪的每一行都将作为单个日志事件发出。

rolloverStart时间

com.ibm.ws.logging.rollover.start.time

WLP_LOGGING_ROLLOVER_START_TIME

单独使用此设置，或者将此设置与 rolloverInterval 属性配合使用，以对 messages.log 和 trace.log 文件启用基于时间的日志回滚。 此设置指定日志首次翻转的调度时间。 rolloverInterval 设置持续时间从 rolloverStartTime 属性指定的时间开始。 有效值遵循 24 小时 ISO-8601 日期/时间格式 HH:MM ，其中 00:00 表示午夜。 需要填充零。 如果指定了 rolloverInterval 属性，那么 rolloverStartTime 属性的缺省值为 00:00 (午夜)。 console.log 不会滚动，因此此设置不适用。

rolloverInterval

com.ibm.ws.logging.rollover.interval

WLP_LOGGING_ROLLOVER_INTERVAL

单独使用此设置，或者将此设置与 rolloverStartTime 属性配合使用，以对 messages.log 和 trace.log 文件启用基于时间的日志回滚。 此设置指定日志回滚之间的时间间隔 (以分钟为单位) (如果未指定时间单位)。 请指定一个正整数，后跟一个时间单位，可以是天 (d)，小时 (h) 或分钟 (m)。 例如，将 5 小时指定为 5h。 可以将多个值包括在单个条目中。 例如，1d5h 相当于 1 天零 5 个小时。 如果指定了 rolloverStartTime 属性，那么 rolloverInterval 属性的缺省值为 1 天。 console.log 不会滚动，因此此设置不适用。

跟踪日志详细信息级别

您可以在 server.xml 文件中为日志记录元素指定 traceSpecification 属性，以选择性地启用跟踪并设置服务器配置中特定组件的跟踪日志详细信息级别。 以下示例指定两组组件的跟踪规范。

<logging traceSpecification="com.myco.mypackage1.*=fine:com.myco.mypackage2.*=finest"/>

在此示例中，两个详细信息级别规范以冒号 (:) 分隔。

com.myco.mypackage1.*=fine 值适用于包名以 com.myco.mypackage1开头的所有类。

com.myco.mypackage2.*=finest 值适用于包名以 com.myco.mypackage2开头的所有类。

下表列出了可以在 traceSpecification 属性中指定的有效日志详细信息级别。 您可以为任何组件指定空值，以禁用该组件的所有跟踪。 未指定的任何组件都将初始化为缺省状态 *=info。

将跟踪组件设置为 off， fatal， severe， warning， audit或 info 级别与将其设置为 info 级别的效果相同，无法禁用更高级别的 info 。 仅当一个或多个跟踪组件设置为 config 级别或更低级别时，才会创建 trace.log 文件并将日志写入其中。

跟踪日志的有效日志详细信息级别
日志记录级别 内容或显着性

关闭

日志记录已关闭。

致命

任务无法继续，并且组件、应用程序和服务器无法工作。

严重

任务无法继续，但是组件、应用程序和服务器仍可工作。 此级别也可以表明即将发生的不可恢复的错误。

警告

可能发生错误或即将发生错误。 此级别还可以指示渐进故障。 例如，资源的潜在泄漏。

审计

影响服务器状态或资源的重大事件

info

概述总体任务进度的一般信息

config

配置更改或状态

详细信息

详细说明子任务进度的一般信息

精细

常规跟踪

更精细

包含方法入口，出口和返回值的详细跟踪

最精细

更详细的跟踪，包含调试问题所需的所有详细信息

全部

记录所有事件。 如果创建定制级别，那么 all 将包含这些级别，并且可以提供比最详细的跟踪。
