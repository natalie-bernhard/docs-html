ネットワーク・セキュリティーの強化

Open Liberty でベスト・プラクティスを採用して、潜在的な攻撃に対して実動用のネットワークを強化します。 無許可ユーザーがネットワーク・アクティビティーにアクセスしてトラフィックを変更し、ネットワーク・リソースを盗むと、ネットワーク攻撃が発生します。

ファイアウォール

実稼働環境はファイアウォールの内側で実行する必要があります。 ファイアウォールは、ネットワーク・トラフィックをモニターし、セキュリティー・ルールに基づいて特定のトラフィックをブロックすることで、セキュリティーへの侵入を防止します。 ファイアウォールを使用して、実動 Open Liberty サーバーが DMZ で実行されないように保護します。 DMZ は、プロキシー・サーバーと Web サーバーがパブリック・ネットワーキング・ゾーンとプライベート・ネットワーキング・ゾーンの間で実行される中間ゾーンです。 通常、ネットワーク・セットアップには、2 つのファイアウォール (周辺ファイアウォールとバックエンド・ファイアウォール) が含まれます。 パブリック・ゾーンと DMZ の間には境界ファイアウォールが存在し、DMZ とプライベート・ゾーンの間にはバックエンド・ファイアウォールが存在します。 バックエンド・ファイアウォール (ネットワーク全体の設計) は、入口トラフィックを信頼できるアドレスに制限し、出口トラフィックをアプリケーションが必要とする最小アドレスに制限します。

要求速度制限

速度制限は、個々のユーザーが不均衡なサーバー・リソースを消費するのを防止し、サービス妨害攻撃を防止するのにも役立ちます。 レート制限を適用するようにリバース・プロキシー・サーバーまたは Web アプリケーション・ファイアウォール (WAF) を構成します。

コンテナー環境での セキュリティー

オンプレミスまたはクラウドのいずれかで実行されるコンテナー環境では、アプリケーションを複数のノードに分散させることができます。 もう 1 つのベスト・プラクティスは、ノードのクラスター全体でワークロードを相互に分離することです。 Open Liberty がコンテナー環境にデプロイされている場合、ユーザー、チーム、またはアプリケーションに基づいてネットワーク・トラフィックをセグメント化します。 名前空間によって割り当てられたコンテナーまたはポッドのコレクションは、IP およびポートの分離を提供する機能を提供します。 デフォルトでは、異なる名前空間を持つポッドが対話することを許可しません。 コンテナー環境でのネットワーク・セキュリティーについて詳しくは、 コンテナー・セキュリティーの 10 層のネットワーク分離に関する情報を参照してください。

TLS 接続

トランスポート・セキュリティー機能は、HTTP 接続に対して TLS を有効にし、TLS を使用してアウトバウンド・トランスポートを構成する機能を追加します。 実動目的では、ネットワーク通信を回避し、ネットワーク侵入のリスクを軽減するために、TLS 接続が不可欠です。

TLS は、鍵ペアと署名付き X.509 証明書を使用して、HTTP、IIOP、LDAP、 IBM MQ、 JDBC、および Open Liberty メッセージング・エンジン、 J2C、および SOAP トラフィック間のメッセージング・トラフィックを保護します。 デフォルトの TLS 構成は簡単に構成できますが、自己署名証明書を生成するため、実動での使用には適していません。 自動生成された証明書は開発目的には便利ですが、実動で使用される証明書は、Verisign などの信頼できる認証局によって署名される必要があります。 ユーザー認証に SSO を使用する場合、すべての SSO エンドポイント URL は HTTPS プロトコルを使用する必要があり、クライアントは TLS サーバー証明書を検査する必要があります。

Open Libertyでは、入口プロキシーは構成可能であり、デフォルトでは信頼できるプロキシーはありません。 プロキシーによってトラフィックが着信するときにトランスポート・レベルで保護するには、必要な Open Liberty サーバーに、信頼できるプロキシーのみがアクセスできるようにします。 可能な限り、非 TLS ポートを無効にしてください。 TLS 対応ポートのみを公開するには、 server.xml ファイルを慎重に検討する必要があります。これは、一部の機能では非 TLS ポートが自動的に有効になるためです。 以下の例に示すように、 server.xml ファイルで非 TLS ポートを -1 の値に設定することで、それらのポートを無効にします。

<httpEndpoint id="defaultHttpEndpoint" httpPort="-1" httpsPort="9445" />

相互認証

実稼働環境では、可能な限り相互 TLS 認証を使用してください。 相互認証 (双方向認証とも呼ばれる) は、クライアントの証明書ベースの認証です。 認証方式は、アプリケーションごとに構成可能です。 例えば、一方のアプリケーションはクライアント認証を使用し、他方のアプリケーションは基本認証を使用する場合があります。 Open Liberty には、クライアント証明書認証用に以下の 2 つのモードがあります。

  • Supported: クライアントは TLS 証明書を提示する場合がありますが、必須ではありません。 クライアントが信頼できない証明書を提示した場合、その証明書は無視されます。 証明書が信頼できる場合は、アプリケーション認証に使用できます。 サポートされるモードの場合は、TLS 構成で clientAuthenticationSupported 属性 を true の値に設定します。

  • 必須: クライアントは、信頼できる TLS 証明書を提示する必要があります。 証明書が送信されない場合、または信頼できない証明書が送信された場合、接続は拒否されます。 必須モードの場合は、 clientAuthentication 属性 を TLS 構成内の true の値に設定します。

プロキシーとともに相互認証を使用する場合は、 trustedHeaderOrigin 属性または trustedSensitiveHeaderOrigin 属性を使用して httpDispatcher 構成エレメント を指定することを検討してください。 この構成エレメントは、以下の例に示すように、指定された IP アドレスを使用してプロキシーがクライアントの代わりに証明書の詳細を渡すことを許可します。

<httpDispatcher trustedSensitiveHeaderOrigin="10.20.30.40, 10.20.50.60" />

Open Liberty デプロイメントにプロキシーが含まれていない場合は、 server.xml ファイルで以下の属性を設定します。

<webContainer trusted="false" /> or
<httpDispatcher trustedHeaderOrigin="none" />

webContainer 構成エレメントの trusted 属性は、アプリケーション・サーバーが Web サーバー・プラグインからのインバウンド・プライベート・ヘッダーを使用できるようにします。 デフォルトでは、 httpDispatcher 構成エレメントの trustedHeaderOrigin 属性は * の値に設定されるため、すべての機密でないプライベート・ヘッダーがすべてのホストから信頼されます。 ただし、 webContainer 構成エレメントの trusted 属性が falseの値に設定されている場合、 trustedHeaderOrigin 属性と trustedSensitiveHeaderOrigin 属性の構成はオーバーライドされます。 これらの構成はオーバーライドされるため、インバウンド・プライベート・ヘッダーは処理されません。これにより、アプリケーション・サーバーに直接接続して Web サーバーをバイパスできるアタッカーからネットワークが保護されます。

LDAP サーバー

Open Liberty サーバーから LDAP サーバーへのリンクが TLS によって保護されていることを確認します。 これらの LDAP 接続を TLS で保護すると、ネットワーク構成が実動用に強化されるように通信が暗号化されます。 TLS を使用して LDAP 接続を保護するには、フィーチャー構成で Lightweight Directory Access Protocol Over Secure Socket Links (LDAPS) プロトコルを有効にします。 LDAPS の有効化および構成について詳しくは、 LDAP ユーザー・レジストリー 機能を参照してください。

LTPA のベスト・プラクティス

Lightweight Third-Party Authentication (LTPA) は、認証に Cookie またはバイナリー・トークンを使用するシングル・サインオン (SSO) テクノロジーです。 LTPA を使用する場合、ネットワークを強化するためにいくつかの手段を取ることができます。 デフォルトでは、LTPA Cookie は HTTPS 接続または HTTP 接続を介して送信されます。 実動目的では、LTPA トークンを保持する Cookie を保護する必要があります。 この Cookie が盗まれた場合、トークンの有効期限が切れるまで、侵入者は認証済みユーザーの代わりに行動することができます。 Cookie は secure 属性をサポートします。この属性は、HTTPS 接続のみを介して Cookie を送信するようにブラウザーに指示します。 server.xml ファイルで以下の属性を設定して、LTPA Cookie が TLS 接続を介して送信されることを要求します。

<webAppSecurity ssoRequiresSSL="true" />

ベスト・プラクティスとして、LTPA 鍵は定期的に置き換えてください。 LTPA 鍵を定期的に置き換えると、不良アクターがファイルのコピーにアクセスしてその内容を暗号化解除できるリスク・ベクトルが削減されます。 鍵を置き換えるには、 ltpa.keys ファイルを削除し、サーバーが自動的に鍵を再生成するのを待ちます。 次に、新しい鍵ファイルを、鍵を共有する他のサーバーのファイル・システムにコピーします。

実稼働環境では、LTPA の Cookie は HTTP 要求にのみ使用する必要があり、 JavaScript アプリケーションなどの他のアプリケーションには使用しないでください。 以下の例は、LTPA Cookie の使用を HTTP 接続のみに制限するようにブラウザーに指示します。

<webAppSecurity httpOnlyCookies="true" />

また、 cookieHttpOnly 属性を使用して、クロスサイト・スクリプティング攻撃を防止することもできます。 値が trueに設定されている場合、この属性は、セッション Cookie に HttpOnly フィールドを含める必要があることを指定します。 HttpOnly フィールドをサポートするブラウザーでは、クライアント・サイドのスクリプトは Cookie にアクセスできません。

<httpSession cookieHttpOnly="true" />

LTPA によって保護されているアプリケーションからユーザーがログアウトすると、クライアント・サイドで LTPA トークン値が破棄されます。 ユーザーのログアウト後に同じサーバー上で LTPA トークンを再使用できないようにするには、 trackLoggedOutSSOCookies 属性を trueに設定します。 この属性は、ログアウト後に同じサーバー上でトークンを再利用できないように、サーバー上でログアウトされた LTPA トークンを追跡するかどうかを指定します。

<webAppSecurity trackLoggedOutSSOCookies="true" />

仮想ホスト

Open Liberty 管理トラフィックをアプリケーション要求と分けて保持するには、サーバーのデフォルト・ホスト以外の仮想ホストで実行するようにアプリケーションを構成します。 さらに、仮想ホストの別名を定義する場合は、別名ホスト名にワイルドカード値 (*) を使用しないでください。 別名ホスト名の値は、ワイルドカードを使用するのではなく、管理者がサーバーで処理することを予期している特定のホストにのみ一致する必要があります。 server.xml ファイルで virtualHost 構成エレメントを指定して、仮想ホストを定義します。

<virtualHost id="secure-app-host">
    <hostAlias>example_host:9080</hostAlias>
</virtualHost>

詳しくは、 仮想ホストを使用した着信アプリケーション要求の分離を参照してください。

のウェルカム・ページとヘッダー

実動の場合は、 Open Liberty のウェルカム・ページを無効にすることができます。 ウェルカム・ページはデフォルトで有効になっており、 / ルート・コンテキストにアクセスすると、 Open Liberty ウェルカム・ページが表示されます。 このウェルカム・ページを無効にするには、 server.xml ファイルで enableWelcomePage 属性を false の値に設定します。

<httpDispatcher enableWelcomePage="false" />

Open Liberty サーバー・ヘッダーもデフォルトで有効になっています。 これらのヘッダーはデフォルトで有効になっているため、状況によっては、実行中のサーバー実装に関する情報がブラウザーまたは Web クライアントに返されることがあります。 removeServerHeader 属性を true の値に設定すると、HTTP ヘッダーからサーバー実装情報が削除されます。

<httpOptions removeServerHeader="true" />

Open Liberty サーバーが稼働していることを明らかにしたくない場合は、X-Powered-By ヘッダーを無効にすることができます。 disableXPoweredBy 属性を値 true に設定すると、X-Powered-By ヘッダーが無効になり、HTTP 応答でヘッダーが送信されなくなります。

<webContainer disableXPoweredBy="true" />

セッション・オーバーフロー

HTTP セッション・オーバーフローを無効にすることにより、メモリー内セッションを使用するアプリケーション用に作成できるセッションの数を制限します。 セッションを制限することで、攻撃者がすべての JVM メモリーが使い尽くされるまで継続的に新規セッションを生成するサービス妨害攻撃を防ぐことができます。 allowOverflow 属性を falseの値に設定することで、HTTP セッション・オーバーフローを無効にすることができます。

<httpSession allowOverflow="false" maxInMemorySessionCount="1000" alwaysEncodeURL="true" cookieSecure="true" cookieHttpOnly="true" />

JMX コネクター

IBM HTTP Serverを使用する場合、以下の項目を削除またはコメント化することにより、Web サーバー・プラグイン内のリモート管理サービス用の Open Liberty JMX コネクターへのアクセスを保護できます。

<Uri AffinityCookie="JSESSIONID" AffinityURLIdentifier="jsessionid" Name="/ibm/api/*" />
<Uri AffinityCookie="JSESSIONID" AffinityURLIdentifier="jsessionid" Name="/IBMJMXConnectorREST/*" />
