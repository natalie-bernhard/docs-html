仮想ホストを使用した着信アプリケーション要求の分離

仮想ホストは、単一サーバーが複数のドメイン・ネームおよびポート構成をホストできるようにする構成エンティティーです。 Open Liberty サーバーで仮想ホストを構成して、着信アプリケーション要求を相互に分離したり、メトリックや JMX 通信などの管理トラフィックから分離したりすることができます。

例えば、会社が同じサーバー上に 2 つのアプリケーション (1 つは顧客用、もう 1 つは従業員用) を維持しているとします。 仮想ホストを使用することで、アプリケーションごとに異なるホスト名とポート番号の構成を指定できます。 また、各ホストの別名を指定して、カスタマー・アプリケーションが customer.example.com URL からアクセスされ、従業員アプリケーションが employee.example.com URL からアクセスされるようにすることもできます。 2 つのアプリケーションは同じサーバー上で実行され、多くの同じ構成リソースを共有しますが、各アプリケーションの着信要求は他のアプリケーションから分離されます。 ある仮想ホストに関連付けられているインバウンド要求は、それらの仮想ホストが物理マシンを共有している場合でも、他の仮想ホストに関連付けられているアプリケーションにアクセスすることはできません。

また、 デフォルトの仮想ホストによって処理されるメトリックや JMX 通信などの管理トラフィックから着信アプリケーション要求を分離するように、アプリケーションの専用仮想ホストを構成することもできます。

仮想ホスト別名

各仮想ホストには、ID 値と、それが認識されている 1 つ以上のホスト別名のリストがあります。 ホスト別名は、リソースを要求するために使用される TCP/IP ホスト名とポート番号の組み合わせです (例: yourHostName:80)。 ポート番号が指定されていない場合、デフォルトのポート番号は 80です。

仮想ホスト構成では、仮想ホスト別名構成のホスト名にワイルドカード値 (*) を使用できます。 ポート別の別名にワイルドカード項目を使用して、特定のポート上のすべての有効なホスト名とアドレスの組み合わせが特定の仮想ホストにマップされるように指定することができます。 ただし、セキュリティーが強化された実稼働環境では、ワイルドカードのホスト名値は推奨されません。 詳しくは、 実稼働環境での仮想ホストおよびアプリケーション・セキュリティーを参照してください。

仮想ホストのデフォルトの別名は、セキュアでない外部ポートを使用する *:80です。 このデフォルトは、別名構成を指定しないすべての仮想ホスト ( default_host のデフォルト仮想ホストを含む) に使用されます。

仮想ホスト別名の構成により、仮想ホストがセキュア・ポートを使用するかどうか、およびポートが内部か外部かが決まります。

  • *:9080 という形式の別名では、セキュアではない内部ポートが使用されます。

  • *:9443 という形式の別名では、セキュアな内部ポートが使用されます。

  • *:443 という形式の別名では、セキュアな外部ポートが使用されます。

要求が、定義された仮想ホストの別名にマップできない別名を使用してリソースを指定した場合、結果は 404 エラーになります。

デフォルトの仮想ホスト

Open Liberty には、 default_hostという 1 つのデフォルト仮想ホストが用意されています。これは、サーバーを初めて始動するときに自動的に構成されます。 特に同じサーバー上のリソースを互いに分離したい場合を除き、デフォルトの仮想ホスト以外の仮想ホストは必要ない可能性があります。 ただし、デフォルトの仮想ホストは、JMX 通信およびその他のメトリック・プロセスにも使用されます。 このトラフィックをアプリケーションへの着信要求から分離するために、アプリケーション用に別の仮想ホストを構成することができます。

デフォルトの仮想ホストのホスト別名は、 *:80 および *:9080として構成されます。ここで、port 80 は HTTP サーバー・ポート、port 9080 は HTTP トランスポートのポートです。 デフォルトの仮想ホストには、マシンの IP アドレス、短縮ホスト名、完全修飾ホスト名などの共通別名が含まれます。 これらの別名の 1 つは、サーブレットなどのリソースにアクセスするためのパスの最初の部分で構成されます。 例えば、 http://localhost:80/myServlet 要求では localhost:80 別名が使用されます。

実稼働環境での 仮想ホストおよびアプリケーション・セキュリティー

ネットワーク・セキュリティー構成の強化の一環として、仮想ホストを使用して、アプリケーションへの着信要求を制限することができます。 Open Liberty 管理トラフィックをアプリケーション要求と分けて保持するには、サーバーのデフォルト・ホスト以外の仮想ホストで実行するようにアプリケーションを構成します。 さらに、セキュリティーを強化した実稼働環境のベスト・プラクティスは、別名ホスト名にワイルドカード値 (*) を使用しないようにすることです。 別名ホスト名の値は、ワイルドカードを使用するのではなく、管理者がサーバーで処理することを予期している特定のホストにのみ一致する必要があります。 詳しくは、 ネットワーク・セキュリティー強化: 仮想ホストを参照してください。

仮想ホスト構成

Open Libertyでは、 server.xml ファイルで virtualHost 構成エレメントの属性を指定することによって、仮想ホストを定義したり、デフォルト仮想ホストの構成を調整したりすることができます。

同じサーバー上のアプリケーションを分離します。

以下の例は、仮想ホスティングの一般的なユース・ケースを示しています。2 つのアプリケーションが同じサーバー上の異なるポートで実行されるように構成します。 この構成では、2 つの HTTP エンドポイントと 2 つの仮想ホストを定義します。 別名構成は、 application-1 仮想ホストを defaultHttpEndpoint エンドポイントに関連付け、 application-2 仮想ホストを alternateEndpoint エンドポイントに関連付けます。

<httpEndpoint id="defaultHttpEndpoint" host="*" httpPort="9080" />
<httpEndpoint id="alternateEndpoint" host="*" httpPort="9081" />

<virtualHost id="application-1">
    <hostAlias>example_host:9080</hostAlias>
</virtualHost>

<virtualHost id="application-2">
    <hostAlias>localhost:9081</hostAlias>
</virtualHost>

<enterpriseApplication location="myApp.ear" name="App1"/>
<webApplication location="myApp2.war" name="App2" />

さらに、 application-2 仮想ホストの仮想ホスト構成では、このホスト上のアプリケーションが localhost インターフェースでのみ使用可能であることが指定されています。 この構成は、開発またはテストの目的で、アプリケーションが実行されているコンピューターからのトラフィックのみを受け入れるようにする場合に役立ちます。

defaultHttpEndpoint HTTP エンドポイントは、ポート 9080上のすべてのインターフェースを公開するように構成されています。 alternateEndpoint HTTP エンドポイントは、ポート 9081 ですべてのインターフェースを公開するように構成されています。 App1 アプリケーションに、 virtual-host name="application-1"を指定する ibm-web-bnd.xml ファイルを持つ WAR ファイルがある場合、このアプリケーションには your_host_name:9080/app1_context_root エンドポイントでのみアクセスできます。 webApplication エレメントで構成されている App2 アプリケーションに、 virtual-host name="application-2"を指定する ibm-web-bnd.xml ファイルがある場合、このアプリケーションには localhost:9081/app2_context_root エンドポイントでのみアクセスできます。

仮想ホストを指定していない 3 番目のアプリケーションが同じサーバーにデプロイされている場合、デフォルトの仮想ホストにマップされるプロキシー要求によってそのアプリケーションにアクセスできます。 例えば、ポート 80 は、どの hostAlias 属性によっても定義されていません。 そのポート上のプロキシーに対して要求が行われると、その要求は default_host 仮想ホストにルーティングされます。

要求されたホストまたはポートに基づいてアプリケーションを分離します

Open Libertyのデフォルトの仮想ホストである default_hostは、JMX 通信などの管理トラフィックに使用されます。 着信アプリケーション要求を管理トラフィックから分離するために、アプリケーション用に別個の仮想ホストを構成することができます。

以下の例は、 server.xml ファイル内の virtualHost エレメントおよび hostAlias 属性構成を示しています。

<virtualHost id="proxiedRequests">
    <hostAlias>external.host.name:80</hostAlias>
    <hostAlias>external.host.name:443</hostAlias>
</virtualHost>

要求がプロキシーからのものである場合、この構成は、プロキシー・ホストおよびポートに対して行われたすべての要求を「proxiedRequests」仮想ホストにルーティングします。

アプリケーションをこの仮想ホストに関連付けるには、アプリケーション・コードでその仮想ホストへの参照を追加する必要があります。 以下の例に示すように、アプリケーションの WAR ファイルの ibm-web-bnd.xml ファイルまたは ibm-web-bnd.xmi ファイルに virtual-host エレメントを追加します。

<?xml version="1.0" encoding="UTF-8"?>
<web-bnd
    xmlns="http://websphere.ibm.com/xml/ns/javaee"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://websphere.ibm.com/xmk/ns/javaee http://websphere.ibm.com/xml/ns/javaee/ibm-web-bnd_1_0.xsd"
    version="1.0" />

    <virtual-host name="proxiedRequests" />

</web-bnd>

発信元エンドポイントに基づいてアプリケーションへのアクセスを制限します

defaultHttpEndpoint HTTP エンドポイントを使用しているシステム・アプリケーションへのアクセスを制限するために、仮想ホスト構成で新規エンドポイントを定義し、 allowFromEndpointRef 属性を指定することができます。 この属性を指定すると、仮想ホストは指定されたエンドポイントからの要求のみを受け入れます。 仮想ホスト別名は、 Host HTTP ヘッダーに基づいて要求をフィルターに掛けることができますが、 allowFromEndpointRef 属性は、システム・ネットワーク・インターフェース層で制約を指定することができます。

以下の例では、 localHostOnly HTTP エンドポイントは、ポート 9081 および 9444 が localhost インターフェースでのみ公開されることを指定しています。 default_host は、 allowFromEndpointRef 属性によってこのエンドポイントに制限されます。

<httpEndpoint id="localHostOnly" host="localhost" httpPort="9081" httpsPort="9444"/>

<virtualHost id="default_host" allowFromEndpointRef="localHostOnly">
    <hostAlias>*:9081</hostAlias>
    <hostAlias>*:9444</hostAlias>
</virtualHost>

</virtualHost id="proxiedRequests">
    <hostAlias>*:9080</hostAlias>
    <hostAlias>*:9443</hostAlias>
    <hostAlias>external.host.name:80</hostAlias>
    <hostAlias>external.host.name:443</hostAlias>
</virtualHost>

この構成では、 default_host 仮想ホストは、 localHostOnly エンドポイントからも発信される localhost:9081 および localhost:9444 でのみ送信される要求を受け入れるようになりました。 ポート 9081 および 9444 に対するその他の要求はすべて拒否されます。 例えば、 localhost:9081 を参照する Host ヘッダーを持つ defaultHttpEndpoint からの要求は拒否されます。

proxiedRequests 仮想ホストは、 defaultHttpEndpoint HTTP エンドポイントによって使用されるポートである、ポート 9080 または 9443へのすべての要求を受け入れます。 また、プロキシーおよびポート 80 または 443からの外部ホスト名を参照するホスト・ヘッダーを持つ要求も受け入れます。
