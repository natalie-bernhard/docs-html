Open Liberty での FIPS 準拠アプリケーションの実行

連邦情報処理標準 (FIPS) 140-2 は、暗号モジュールに関する米国政府のセキュリティー標準です。 FIPS 準拠は基礎となる Java 仮想マシン (JVM) によって決定されますが、FIPS 準拠の JVM で Open Liberty を実行できるようにすることができます。

FIPS の有効化は、多くのユーザーにとって重要です。特に、米国政府機関で作業する場合は重要です。 FIPS 準拠の JVM で Open Liberty サーバーを実行すると、アプリケーションが Java セキュリティー・ライブラリーまたは API を使用する場合に、FIPS 認証暗号方式のみが使用されます。 Open Liberty の FIPS 準拠の JVM オプションは、 IBM SDK, Java Technology Edition と IBM Semeru Runtimesの両方に含まれています。

IBM SDK, Java Technology Edition を使用して Liberty で FIPS を使用可能にする方法について詳しくは、 WebSphere Liberty 資料の「 Setting up Liberty for FIPS compliance 」を参照してください。 構成は、 WebSphere Liberty と Open Libertyの両方で同じです。

IBM Semeru Runtimes 上の FIPS for Open Liberty を使用可能にする

Java 11 および Java 17 用のバージョン 17.0.4 以降では、 IBM Semeru Runtime Certified Edition または FIPS モードの Open Edition をバージョン 11.0.16 以降で使用可能にすることができます。 Java 11 および 17 での FIPS with Semeru Runtimes のサポートは、 x86 プラットフォーム上の Red Hat Enterprise Linux (RHEL) 8 でのみ使用可能です。 IBM Semeru Runtimes は、オペレーティング・システムの基礎となる Network Security Services (NSS) FIPS 140-2 認証に依存するため、RHEL 8 オペレーティング・システムは FIPS モードで実行する必要があります。 IBM Semeru Runtimes で FIPS モードで Open Liberty を実行するには、 Open Liberty バージョン 22.0.0.8 以降をお勧めします。 FIPS モードでは、Semeru Runtimes は、JKS や PKCS#12などのファイル・ベースの鍵ストアをサポートしていません。 ファイル・ベースの鍵ストア内の証明書は、NSS データベースにインポートする必要があります。 Open Liberty は、NSS データベースに証明書を作成しません。

以下のステップを実行して、FIPS モードの Semeru Runtimes で実行するように Open Liberty サーバーを構成し、鍵と証明書を NSS データベースに追加します。

  1. RHEL オペレーティング・システムが FIPS モードでインストールされていることを確認します。
    RHEL オペレーティング・システムが FIPS モードでインストールされていない場合は、FIPS モードに切り替える必要があります。 RHEL オペレーティング・システムの FIPS 状況を有効化または確認する方法について詳しくは、RHEL 資料の FIPS モードへのシステムの切り替え を参照してください。

  2. システム・プロパティーを指定して JVM の FIPS モードを有効にし、オプションでデバッグ・トレースを有効にします。
    -Dsemeru.fips=true プロパティーは、JVM が FIPS 認証済み暗号方式のみを使用すること、および TLS プロトコルと SSL プロトコルが FIPS 認証済みアルゴリズムのみを使用することを指定します。 オプションの -Djava.security.debug=semerufips プロパティーは、デバッグ・トレースを有効にします。 以下の例に示すように、これらのプロパティーを Open Liberty サーバー構成ディレクトリー内の jvm.options ファイルに 1 行に 1 つずつ追加します。

    -Dsemeru.fips=true
-Djava.security.debug=semerufips

  3. PKCS#11 鍵ストアの読み取りに必要な NSS ライブラリーを含む Liberty 構成ファイルを作成します。
    このファイルは、Liberty からアクセス可能な場所になければならず、以下の情報が含まれている必要があります。

    名前 = NSS-FIPS
ライブラリー = /usr/lib64/libsoftokn3.so
スロット = 3
showInfo = true

    このファイルは、ステップ 5 の鍵ストア location 構成属性で tmp/pkcs11cfg.cfgとして参照されます。

  4. 鍵と証明書を NSS データベースにインポートします。
    FIPS モードでは、Semeru Runtimes は、JKS や PKCS#12などのファイル・ベースの鍵ストアをサポートしていません。 ファイル・ベースの鍵ストア内の証明書は、NSS データベースにインポートする必要があります。
    NSS pk12util および certutil コマンドを使用して、NSS データベース内の鍵および証明書をインポートおよび管理することができます。

    • 鍵ストアから NSS データベースに鍵をインポートするには、 pk12util コマンドを使用します。 以下の例では、 key.p12 は鍵ストア・ファイル、 Liberty は鍵ストア・パスワードです。

      pk12util -I key.p12 -W Liberty -d /etc/pki/nssdb

    • 以下の例に示すように、同じ方法でトラステッド証明書をインポートできます。ここで、 trust.p12 は証明書項目を含むファイルです。

      pk12util -I trust.p12 -W Liberty -d /etc/pki/nssdb

    • 信頼された証明書は、信頼された認証局 (CA) としてマークされ、クライアント証明書とサーバー証明書の両方が完全に信頼されている必要があります。 以下の例に示すように、 certutil コマンドを実行して CA を適用します。ここで、 -t 引数は、 CT 値を使用して完全な信頼を指定します。

      certutil -M -n trustCert -t "CT, CT, CT" -d /etc/pki/nssdb

    • また、以下の例に示すように、 certutil コマンドを使用して NSS データベースの内容を調べることもできます。

      certutil -L -d /etc/pki/nssdb

  5. 鍵と証明書をインポートした NSS データベースを参照する keystore 項目を server.xml ファイルに作成します。
    以下の server.xml の例は、Semeru Runtimes で FIPS モードで Open Liberty を実行するための鍵ストア構成を示しています。 この例では、 location=“/tmp/pkcs11cfg.cfg” には、ステップ 3 で作成した Liberty 構成ファイルへのパスを指定します。

    <featureManager>
    <feature>transportSecurity-1.0</feature>
</featureManager>

    <ssl id=“defaultSSLConfig"
         keyStoreRef="defaultKeyStore"
         sslProtocol=“TLSv1.2” />

    <keyStore id="defaultKeyStore" password="Liberty"
         location=“/tmp/pkcs11cfg.cfg” type=“PKCS11”
         fileBased=“false” provider=“SunPKCS11-NSS-FIPS” />

    この例では、鍵ストアの type 属性は PKCS11に設定されていますが、 PKCS11-NSS-FIPS も有効な値です。 この構成は、ファイル・ベースの鍵ストアの代わりに NSS PKCS#11-based 鍵ストアを使用するように Open Liberty に指示します。

これで、 Open Liberty サーバーを FIPS モードで開始できるようになりました。

FIPS モードでの Semeru Runtimes について詳しくは、 IBM Semeru Runtimesの FIPS 認証暗号方式を参照してください。 Open Liberty TLS 構成について詳しくは、 TLS を使用したセキュア通信 および トランスポート・セキュリティー ・フィーチャーを参照してください。
