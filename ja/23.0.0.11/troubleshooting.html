セキュリティーのトラブルシューティング

セキュリティー関連の問題に対する解決策を見つけるには、以下の情報を参考にして、一般的な問題の原因および関連するエラー・メッセージを判別してください。

  • ACME 証明書のトラブルシューティング

  • LDAP のトラブルシューティング

    • LDAP サーバーに対する Kerberos 認証のトラブルシューティング

  • SSO のトラブルシューティング

  • SSL および TLS のトラブルシューティング

  • その他のトラブルシューティングの問題

ACME 証明書のトラブルシューティング

ACME 認証局 (CA) 証明書で問題が発生した場合は、 Open Liberty サーバーからのエラー・メッセージまたは HTTP エラー・メッセージを参照して、問題をデバッグすることができます。 以下の情報は、ACME CA 証明書に関連する一般的な問題およびエラー・メッセージの原因を判別するのに役立ちます。

ACME CA 証明書に関する問題のトラブルシューティングに役立てるために、以下のトレース仕様を使用してトレースを有効にすることができます。

ACMECA= すべて

以下のセクションでは、ACME CA アカウントおよび接続で発生する可能性のある一般的な問題について説明します。 詳しくは、 Automated certificate management with ACME および ACME Support フィーチャーを参照してください。

認証要求がタイムアウトになる

認証要求がタイムアウトになった場合は、 challengePollTimeout プロパティーと orderPollTimeout プロパティーを使用して、より長いタイムアウト値を設定できます。

更新要求で HTTP コード 429 メッセージを受け取りました

即時証明書更新要求の数が多すぎることや、サーバーに悪影響が及ぶ可能性があることを防ぐために、証明書更新要求は短時間ブロックされます。 このウィンドウの有効期限が切れると、新しい要求を行うことができます。 429 メッセージは、いつ新しい要求を行うことができるかを示します。

レート制限を超えたことを示すメッセージを受け取りました

LetsEncryptなどの一部の CA は、新規証明書の要求に対してレート制限を適用します。 短時間で複数の証明書をテストして要求する場合は、適切なテスト・サーバーを使用してください。 例えば、 LetsEncrypt は、より高いレート制限を持つステージング・サーバーを提供します。

証明書は、有効期限が切れていない場合、始動時に更新されます。

以下の条件により、有効期限が切れていない証明書が始動時に自動的に更新される可能性があります。

  • 証明書に取り消し済みのマークが付けられている

  • 証明書の有効期限は、 renewBeforeExpiration プロパティーで設定されたウィンドウ内です。

  • ディレクトリー URI、ドメイン、またはその他のアカウント情報が変更されたため、新しい証明書が必要です。

  • サーバーが --clean オプションを指定して始動され、証明書の履歴情報が削除されました。

許可チャレンジが CWPKI2001E メッセージで失敗する

サーバーが証明書の取り出しに失敗した場合、以下の例のようなエラー・メッセージが表示されることがあります。

CWPKI0804E: SSL 証明書作成エラー。 エラーは次のとおりです。 CWPKI2001E: http://my-configured-ca.com/directory URI の ACME 認証局は、 mydomainname.com ドメインの許可チャレンジが失敗したと応答しました。 ユーザー確認のための質問のステータスが無効です。  エラーは「Fetching http://mydomainname.com/.well-known/acme-challenge/FXCFcGCv4Ov2ofJ2i-PgMsO1kECwKB0XfTzsPjNIXBs: 接続が拒否されました」です。

このメッセージが表示された場合は、指定されたドメイン・ネームが CA によってアクセス可能であることを確認してください。 ログを確認し、予期されるドメイン・ネームまたは IP アドレスが acme-challenge Web アプリケーションに使用されていることを確認します。 ログで以下のメッセージを探します。

CWWKT0016I: 使用可能な Web アプリケーション (default_host): http://mydomainname.com:80/.well-known/acme-challenge/

Web アプリケーションに使用するホスト名を構成するには、 server.xml ファイルで httpEndpoint 構成の host 属性を追加または更新します。

証明書の取り出しに失敗した後、鍵ストアでエラーが生成される

サーバーが証明書をフェッチできない場合は、空の鍵ストアが作成されます。 古いバージョンの Java では、空の鍵ストアが原因で例外が発生する可能性があります。 このエラーの例としては、以下のメッセージがあります。

CWPKI2030E: ACME サービスは、デフォルトの別名の下の証明書を defaultKeyストア鍵ストアにインストールできませんでした。 エラーは「鍵ストア [defaultKeyStore] が構成に存在しません」です。` ` `
CWWKS9582E: defaultOrb ID を持つ orb エレメントが必要とする [defaultSSLConfig] sslRef 属性は、10 秒以内に解決されませんでした。 このため、アプリケーションは始動しません。 keyStore エレメントを組み込んだこと、および Secure Sockets Layer (SSL) が正しく構成されていることを確認します。 sslRef が defaultSSLConfigの場合、ID 値「defaultKeyStore」とパスワードを指定して keyStore エレメントを追加します。

初期証明書の取り出しに失敗した後にこのエラーを回避するには、空の鍵ストアを削除します。

失効検査中に CWPKI2058W 警告メッセージを受け取る

コンテナー化されたバージョンの ACME CA サーバーを実行する場合、証明書に定義されている OCSP レスポンダー URL に到達できない可能性があります。 「acmeRevocationChecker」エレメントで「ocspResponderUrl」属性を指定することにより、証明書内の OCSP 応答側 URL をオーバーライドすることができます。 この URL が構成されていない場合、失効検査中に以下の警告が発生する可能性があります。

CWPKI2058W: 証明書失効状況検査でソフト障害が無視されました。 失効検査が不完全である可能性があります。 失敗: '[java.security.cert.CertPathValidatorException: ネットワーク・エラーが原因で失効状況を判別できません。 java.security.cert.CertPathValidatorException: ネットワーク・エラーが原因で失効状況を判別できません]'

このネットワーク・エラー警告が表示され、テスト CA サーバーで実行している場合は、カスタム ocspResponderUrl URL を追加できます。 テスト CA が失効テストをサポートしていない場合は、以下の例に示すように、 acmeRevocationChecker エレメント falseに enabled 属性を設定することによってテスト CA をサポートできます。

<acmeCA>
   ...
   <acmeRevocationチェッカーの有効化 =「false」/>
</acmeCA>

LDAP のトラブルシューティング

構成された LDAP サーバーに到達できないことを示す FFDC1015I エラー・メッセージを受け取りました。

構成された LDAP サーバーに到達できない場合は、以下の例のようなエラー・メッセージが messages.log ファイルに表示される可能性があります。

FFDC1015I: 発生事象が作成されました: javax.naming.ServiceUnavailableException:
myldapserver.mycompany.com:636; socket closed com.ibm.ws.security.registry.ldap.internal.LdapRegistry
298

このメッセージが表示された場合は、LDAP サーバーが稼働していること、および LDAP サーバーの IP アドレスに Open Liberty サーバーのホスト・マシンからアクセスできることを確認してください。

LDAP サーバーとの接続が SSL ハンドシェーク・エラーで失敗します。

LDAP サーバーの署名者証明書をトラストストアにコピーせずに LDAP サーバーで SSL を有効にすると、LDAP サーバーとの接続が SSL ハンドシェーク・エラーで失敗します。 トラストストアは、 server.xml ファイルの LDAPSSLSettings エレメントで参照されます。 以下のメッセージが表示される場合があります。

The javax.naming.CommunicationException: simple bind failed: myldapserver.mycompany.com:636
[Root exception is javax.net.ssl.SSLHandshakeException: com.ibm.jsse2.util.g:
PKIX path building failed: java.security.cert.CertPathBuilderException:
unable to find valid certification path to requested target]

必ず、署名者証明書を LDAP サーバーからトラストストア・ファイルにコピーしてください。

LDAP または統合リポジトリーのログイン、認証、または許可が遅い。

ログインに時間がかかる理由はいくつか考えられます。 以下のリストで、いくつかの一般的な原因を確認してください。

  • 参照に関する LDAP 構成が follow に設定されている場合、LDAP 要求で、その他の LDAP インスタンスへの参照があると、それらがすべてフォローされます。 この状況では、LDAP 構成によっては 1 つ以上の LDAP サーバーに接続する可能性があるため、追加の時間がかかることがあります。 例えば、 LDAP1 サーバーは、 LDAP3 サーバーへの参照の後に続く LDAP2 サーバーへの参照を追跡する場合があります。 他の LDAP サーバーから追加で情報を取得する必要がない場合は、参照を ignore に設定してください。

  • 1 つの LDAP サーバーで問題が発生しており、JVM ログに表示されない可能性があるエラーが発生している可能性があります。 そのような例には、TCP 読み取りタイムアウトや、LDAP サーバーが参照先 LDAP サーバーに問い合わせる際の DNS の問題などがあります。 このような状態を診断するには、パケットをキャプチャーして、発生している呼び出しの数を確認したり、参照をフォローしている LDAP サーバーが原因の遅延やエラーが存在していないかどうかを確認したりします。 ファイアウォールまたはその他のソフトウェアによって、LDAP への接続がクローズされています。

  • 統合リポジトリーでは、レルム内に固有のユーザーが存在することを確認するために、すべてのリポジトリーとレジストリーがチェックされます。 リポジトリーまたはレジストリーの応答が遅い場合は、ユーザーが属しているレジストリーが遅くなくとも、すべての呼び出しが遅くなります。 統合リポジトリー内のすべての参加ベース・エントリーが迅速に応答していることを確認します。

LDAP サーバーへのアクセス時に接続例外が発生することがあります。 例えば、「java.net.SocketException: 接続がリセットされました (java.net.SocketException: Connection reset)」です。

ファイアウォールまたはその他のソフトウェアによって、LDAP への接続がクローズされています。 デフォルトで、パフォーマンスを高めるために、LDAP 接続のプールが維持されます。 キャッシュされた接続がリモート側でクローズされると、新しい接続が作成され、コンテキスト・プールにプットバックされます。 このプロセスによって、遅延が発生し、JVM ログに記録されるエラーが発生する可能性があります。 例えば、以下のエラーを受け取ることがあります。

java.net.SocketException: Connection reset

この状態を回避するには、コンテキスト・プールのタイムアウトをリモート接続のクローズ時間よりも小さく設定します。 例えば、ファイアウォールが接続を 10 分で閉じる場合、接続プールのタイムアウトを 9 分に設定できます。 接続障害が発生し、新規接続を作成する代わりに、プールにある接続の有効期限がチェックされ、新しい接続を作成することで、障害ステップをスキップできます。

LDAP サーバーに対する Kerberos 認証のトラブルシューティング

LDAP サーバーに対する Kerberos 認証で問題が発生した場合は、 Open Liberty サーバーからのエラー・メッセージまたは HTTP エラー・メッセージを参照して、問題をデバッグしてください。 以下の情報は、LDAP サーバーに対する Kerberos 認証に関連する一般的な問題およびエラー・メッセージの原因を判別するのに役立ちます。

統合ユーザー・レジストリー用に Kerberos が構成されている場合、パフォーマンスが低下する

federatedRepository エレメントで allowOpIfRepoDown 属性を有効にすると、フェデレーテッド・ユーザー・レジストリー内の 1 つ以上のユーザー・レジストリーが使用できない場合に、障害を回避するのに役立ちます。 ただし、 ccache ファイルで krb5TicketCache 属性を使用して Kerberos 資格情報が指定されている場合、この構成により全体的なパフォーマンスが低下する可能性があります。 Kerberos 資格情報が ccache ファイル内にある場合、 Open Liberty は、有効期限が近づいているか、有効期限が切れている資格情報を自動更新しようとします。 この自動更新の試行により、パフォーマンスが低下する可能性があります。

この問題を回避するには、 keytab ファイルで kerberos エレメントを使用して Kerberos 資格情報を指定します。 keytab ファイル内の資格情報の有効期限が切れないため、自動更新は必要ありません。 詳しくは、 Open Liberty の Kerberos 認証を参照してください。

Kerberos が有効になっていないレジストリーが使用可能な場合でも、ユーザーがログインできない

1 つの Open Liberty サーバーに対して複数のユーザー・レジストリーが構成されている場合、すべての基本ユーザー・レジストリー、カスタム・ユーザー・レジストリー、および LDAP ユーザー・レジストリーが単一の統合ユーザー・レジストリーに結合されます。 デフォルトでは、サーバーは、すべての参加ユーザー・レジストリーでユーザーを正常に検索して、そのユーザーが統合ユーザー・レジストリー内で固有であることを確認する必要があります。 統合レジストリー内の Kerberosが有効になっている LDAP サーバーが Kerberos チケット・キャッシュを使用してユーザー資格情報を保持しており、その資格情報の有効期限が切れている場合、LDAP レジストリーの検索は失敗します。 この問題を解決するには、 Kerberos チケット・キャッシュを更新します。 例えば、 Java kinit ツールを使用して、 Kerberos チケット・キャッシュを更新できます。

ユーザー・レジストリーが使用不可の場合に障害が発生しないようにするには、 server.xml ファイルで allowOpIfRepoDown 属性を構成します。 以下の例に示すように、 federatedRepository エレメントの primaryRealm サブエレメントで allowOpIfRepoDown 属性を true に設定します。

<federatedRepository>
        <primaryRealm name="FederatedRealm" allowOpIfRepoDown="true">
            <participatingBaseEntry name="o=SampleBasicRealm"/>
            <participatingBaseEntry name="o=ibm,c=us"/>
        </primaryRealm>
</federatedRepository>

詳しくは、 Federated User Registry フィーチャーを参照してください。

Kerberos プリンシパル名が Kerberos チケット・キャッシュ・ファイルに含まれていない

Kerberos プリンシパル名が Kerberos チケット・キャッシュ・ファイルに見つからない場合、 Open Liberty は CWIML メッセージ・タイプをログに記録します。 以下の理由で Kerberos プリンシパル名の欠落が生じることがあります。

  • Kerberos プリンシパル名に対してクレデンシャルが生成されなかったため、 Kerberos 構成が正しくありません。

  • Kerberos チケット・キャッシュに、期限切れの資格情報が含まれています。

いずれの場合も、 Kerberos チケット・キャッシュを更新して問題を解決してください。 例えば、 Java kinit ツールを使用して、 Kerberos チケット・キャッシュを更新できます。

Java SDK のタイプに応じて、以下のいずれかの例のようなメッセージを受け取る可能性があります。

CWIML4507E: Kerberos ログインは、 user1@SAMPLE.COM Kerberos プリンシパルおよび C:\krb5\krb5-user1.cc Kerberos 資格情報キャッシュ (ccache) で失敗しました。 javax.security.auth.login.LoginException: ユーザーからパスワードを取得できない

CWIML4520E: LDAP 操作を完了できませんでした。 LDAP 命名例外 javax.naming.AuthenticationException: GSSAPI [Root exception is javax.security.sasl.SaslException: GSS initiate failed [Caused by GSSException: No valid credentials provided (Mechanism level: Ticket expired (32))]] が処理中に発生しました。

CWIML4520E: LDAP 操作を完了できませんでした。 LDAP 命名例外 javax.naming.NamingException: CWIML4507E: Kerberos ログインが、 user1@SAMPLE.COM Kerberos プリンシパルおよび C:\krb5\krb5-user1.cc Kerberos クレデンシャル・キャッシュ (ccache) で失敗しました。 javax.security.auth.login.LoginException: ユーザーからパスワードを取得できない

Kerberos プリンシパル・ユーザーの有効期限を確認するには、 Java klist ツールを実行します。

詳しくは、 LDAP サーバー用の Kerberos 認証の構成を参照してください。

SSO のトラブルシューティング

サーバーが協定世界時とユーザー・レジストリーを共有しないため、 シングル・サインオン (SSO) が機能しない

異なる Open Liberty サーバーは、 webAppSecurity エレメントの LTPA 鍵、パスワード、および ssoCookieName 属性を共有します。 これらのサーバーが協定世界時とユーザー・レジストリーを共有していない場合、SSO が機能しない可能性があります。 これらのサーバーが協定世界時とユーザー・レジストリーを共有していることを確認してください。

資格情報の再入力を求めるプロンプトが出されます。

トークンの有効期限が切れると、SSO が機能しない可能性があります。 また、ユーザー・レジストリーを不整合な方法で変更した後に Web ブラウザーから Cookie が送信されると、SSO が失敗する可能性があります。 例えば、レルムを変更したり、Cookie が表すユーザーを削除したりします。 資格情報の再入力を求めるプロンプトが表示される場合があります。 トークンの有効期限が切れていないこと、およびユーザー・レジストリーに一貫性のある変更を行っていることを確認してください。

SSL および TLS のトラブルシューティング

TLS プロトコルが広く採用される前は、SSL は Web 通信を保護するための標準プロトコルでした。 時間の経過とともに、SSL に関する多くのセキュリティーの脆弱性が特定され、それが広く使用されることはなくなりました。 ただし、過去の理由から、TLS が使用されている場合でも、暗号化されたネットワーク接続を指すために SSL という用語が使用されることがよくあります。 Open Libertyでは、TLS が基礎となるプロトコルであっても、フィーチャー名および構成名では引き続き SSL という用語が使用されます。 事実上、SSL は TLS の同義語になりました。

Java EE から Jakarta EE に、または ssl-1.0 フィーチャーから transportSecurity-1.0にマイグレーションした後、証明書が見つかりません。

Secure Sockets Layer (ssl-1.0) フィーチャーが使用可能になっている場合は、 SSLContext.getDefault() メソッドを使用して Liberty のデフォルトの SSLContext クラスを取得します。 ただし、 「トランスポート・セキュリティー」 フィーチャーが有効になっている場合、 SSLContext.getDefault() メソッドは Java Secure Socket Extension (JSSE) SSLContext クラスを返します。 したがって、あるフィーチャーから別のフィーチャーに変更すると、アプリケーション・コードの更新が必要になる場合があります。 トランスポート・セキュリティー・フィーチャーが有効になっている場合にデフォルトの Liberty SSLContext クラスを取得するには、 SSLContext.getDefault() メソッドではなく JSSEHelper.getInstance().getSSLContext(null, null, null) メソッドを使用します。

リダイレクトのための TLS ポートを判別できないという CWWKS9105E メッセージを受け取る

使用不可の TLS ポートにリダイレクトするアプリケーションにアクセスしようとすると、以下のメッセージが表示されることがあります。

 CWWKS9105E: 自動リダイレクト用の TLS ポートを判別できませんでした。

TLS 構成が欠落しているか、TLS 構成定義に何らかのエラーがあるために、ポートが使用不可になっている可能性があります。 TLS 構成が server.xml ファイルに存在し、正しいことを確認してください。

構成内に ID フィールドのない鍵ストア・エレメントが存在し、 FFDC1015I エラーが表示されます。

構成内に ID フィールドのない鍵ストア・エレメントが存在する場合、以下のエラーを受け取ることがあります。

FFDC1015I: FFDC 発生事象が作成されました: "java.lang.IllegalArgumentException:
Unknown, incomplete configuration: missing id field com.ibm.ws.config.internal.cm.ManagedServiceFactoryTracker
aSyncReadNupdate.
Exception thrown while trying to read configuration
and update ManagedServiceFactory. Exception = java.lang.IllegalArgumentException:
Unknown, incomplete configuration: missing id field" ロケーション: ffdc_12.04.18_16.09.42.0.log

このエラーは、構成内に ID フィールドのない鍵ストア・エレメントが存在する場合に発生します。 最小の TLS 構成を使用する場合は、 ID フィールドを defaultKeyStoreに設定します。

その他のトラブルシューティングの問題

匿名ユーザーが認証済みユーザーが所有するセッションにアクセスしようとしたことを示す SESN0008E メッセージを受け取ります。

非認証ユーザーが、認証済みユーザーによって作成されたセッションにアクセスしようとすると、以下のメッセージが表示されることがあります。

SESN0008E: 匿名として認証されたユーザーが、次のユーザーが所有するセッションにアクセスしようとしました:
user:LdapRegistry/cn=steven, o=myCompany, c = US.

このエラーは、 login.jsp ファイルなどの JSP (Jakarta Server Page) をフォーム・ログインに使用し、ブラウザーによって送信された SSO トークンの有効期限が切れている場合に発生する可能性があります。 その後、ユーザーには、フォーム・ログイン用に構成された login.jsp ページを使用して再度ログインするよう求めるプロンプトが出されます。 デフォルトでは、JSP ページは、トークンの有効期限が切れたユーザーによって最初に作成されたセッションを取得しようとします。 そのため、ユーザーは認証されず、このセッションにアクセスしたときに資格情報は確立されず、このエラーが発生します。

このエラーを回避するには、 新規セッションを開始するブラウザーを再始動するか、 デフォルトでセッションを作成しないように login.jsp ファイルを構成します。 login.jsp ファイルの更新を選択した場合は、 <%@ page session="false" %>を設定します。

必要なロールのいずれに対するアクセス権限もユーザーに付与されていないことを示す CWWKS9104A メッセージを受け取ります。

アプリケーションが必要とするロールに対する許可をユーザーが持っていない場合、以下のメッセージが表示されることがあります。

CWWKS9104A: {2} の {1} を呼び出し中にユーザー {0} の許可が失敗しました。 ユーザーは、
必要なロールのいずれかに対するアクセス権限を付与しました: {3}。

ユーザーまたはユーザーが属するグループが、エラー・メッセージに示されている役割の少なくとも 1 つにマップされていることを確認してください。 ibm-application-bnd.xmi/xml ファイルに定義されたユーザーからロールへのマッピングが、server.xml ファイルに定義されたマッピングより優先されます。 両方のリソースを検査して、正しいマッピングが定義されていることを確認してください。 詳しくは、 許可を参照してください。

ユーザーの許可が失敗する

ユーザー許可が失敗した場合、以下のメッセージが表示されることがあります。

CWWKS9104A: ユーザー {0} の許可が失敗しました。

同じコンテキスト・ルートに application と webApplication の両方を指定した場合に、このエラーが発生する可能性があります。 競合した場合、定義された最新構成は無視され、CWWKS9104A などの予期しないエラーが発生します。

アプリケーションのインストールで予期しないセキュリティー動作が発生する

application エレメントと dropins フォルダーの両方を使用して server.xml ファイルにアプリケーションを指定すると、アプリケーションのインストールが 2 回試行され、 server.xml ファイル内のセキュリティー構成が有効になる場合と有効にならない場合があります。

以下のメッセージが表示される場合があります。

CWWKZ0013E: {0}という 2 つのアプリケーションを開始することはできません。

このメッセージの後に、予期しないセキュリティー動作やエラー・メッセージ ( CWWKS9104Aなど) が続く場合があります。 この問題を修正するには、dropins フォルダーからアプリケーションを削除し、別のディレクトリーにコピーする必要があります。 dropins フォルダーに残す必要がある場合は、 server.xml ファイル内の以下のコードを使用して、アプリケーション・モニターを無効にする必要があります。

<applicationMonitor dropinsEnabled="false"/>
