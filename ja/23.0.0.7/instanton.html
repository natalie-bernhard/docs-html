Open Liberty InstantOn を使用した、コンテナー化されたアプリケーションの始動の高速化

Open Liberty InstantOn は、 MicroProfile および Jakarta EE アプリケーションの起動時間を短縮します。 InstantOnを使用すると、アプリケーションをミリ秒単位で開始することができ、スループット、メモリー、開発/実動パリティー、または Java 言語機能を損なうことはありません。

InstantOn は、 Linux カーネルの Checkpoint/Restore In Userspace (CRIU) 機能を使用して、後でリストアできる JVM のチェックポイントを取得します。 InstantOnを使用すると、アプリケーションは通常どおりに開発され、アプリケーション・コンテナー・イメージのビルド時に InstantOn によってアプリケーション・プロセスのチェックポイントが作成されます。 アプリケーションが復元されると、そのアプリケーションは同じ JVM で実行されます。これにより、開発と実動の間の完全なパリティーが提供されます。 チェックポイント処理には数秒しかかからないため、CI/CD 処理はほとんど影響を受けません。

アプリケーション・コンテナー・イメージをビルドすると、 InstantOn は、 Open Liberty アプリケーション・プロセスのチェックポイントを含む追加のイメージ層を作成します。 アプリケーション・イメージが開始されると、 Open Liberty ランタイムは InstantOn レイヤーを認識し、アプリケーション・コンテナー・イメージのビルド中に作成されたチェックポイントにアプリケーション・プロセスをリストアします。

Open Liberty および OpenJ9 には、チェックポイントおよびリストア・プロセスに参加するフックが含まれています。 これらのフックにより、 Open Liberty ランタイムは、コンテナー・イメージ内のチェックポイント・プロセスの永続化が安全であるように、チェックポイントのプロセスを準備することができます。 この準備ステップにより、コンテナー・イメージ内の同じ永続状態から、アプリケーションの複数の (場合によっては並行) インスタンスにプロセスをリストアできるようになります。 OpenJ9 および IBM Semeru JVM での CRIU サポートについて詳しくは、Java CRIU Support 資料を参照してください。

InstantOn は、コンテナー・イメージ・ビルドの外部では使用できません。 アプリケーション・コンテナー・イメージは、 Open Liberty アプリケーション・プロセスの信頼性の高いリストアを確保するために必要な、一貫性のある環境を提供します。 InstantOn コンテナー層は、アプリケーション・コンテナー・イメージの最後の層です。 この構成により、イメージの基礎となるレイヤー内のリソースが、チェックポイントが取られた時点から、 InstantOnを使用してイメージが開始された時点まで変更されないことが保証されます。

以下のセクションでは、 InstantOn アプリケーション・イメージをビルドして実行するための前提条件とプロセスについて説明します。

  • チェックポイントを取る場合: beforeAppStart または afterAppStart

  • ランタイムおよびホスト・ビルド・システムの前提条件

  • InstantOn アプリケーション・イメージのビルド

  • InstantOn アプリケーション・イメージの実行およびデプロイ

  • Open Liberty InstantOn でサポートされるフィーチャー

チェックポイントを作成する場合: beforeAppStart または afterAppStart

InstantOn チェックポイントが発生すると、 Open Liberty ランタイムが開始されます。 この始動時に、ランタイムは構成を処理し、使用可能なすべてのフィーチャーをロードし、構成されたアプリケーションの処理を開始します。 この始動シーケンスは、アプリケーションのサイズと複雑さ、およびアプリケーションが必要とする Open Liberty フィーチャーの数によって異なります。 小規模な単純なアプリケーションの場合、このプロセスにかかる時間は 1 秒未満になる可能性があります。 より一般的なアプリケーションの場合は、数秒かかることがあります。 InstantOn は、 Open Liberty ランタイムが始動を完了した後にチェックポイントを取得します。 アプリケーション自体の開始前または開始後にチェックポイントが発生するかどうかを判別するために、以下の 2 つのオプションを使用できます。どちらのオプションも、 InstantOn RUN 構成の一部として指定できます。

  • afterAppStart -アプリケーションの開始後にチェックポイントを実行します。 このオプションは、最も高速な始動を提供しますが、一部のアプリケーションには適していない場合があります。

  • beforeAppStart -アプリケーション・コードが実行される前にチェックポイントを実行します。 このオプションは、アプリケーション・コードが afterAppStart オプションと互換性がない場合に使用します。

どちらのオプションを選択するかは、アプリケーションで実行する必要があるコードによって異なります。 Jakarta EE および MicroProfile アプリケーションには、以下の例のように、アプリケーションの開始時に実行されるアプリケーション・コードが含まれることがあります。

  • loadOnStartup 属性を使用するサーブレット

  • @Startup アノテーションを使用する EJB

  • @Observes @Initialized(ApplicationScoped.class) アノテーションを使用する CDI Bean

場合によっては、アプリケーションの開始時に実行されるアプリケーション・コードが、 InstantOn チェックポイントの実行に適していないことがあります。 そのような場合は、 beforeAppStart オプションを使用してください。 例えば、以下のアプリケーション・コード・シナリオは、 InstantOn チェックポイントの前に回避する必要があるため、 beforeAppStart オプションに適しています。

  • データベースなどのリモート・リソースへのアクセス。 アプリケーション・コンテナーのビルド中に、正しいデータ・ソースに接続できる可能性はほとんどありません。

  • トランザクションの作成。 現在、トランザクションは InstantOn チェックポイントの前に禁止されています。

  • アプリケーションのデプロイ時に変更されることが予期される読み取り構成 (例えば、 MicroProfile Config からの構成)。

このような場合に beforeAppStart オプションを使用すると、 InstantOn チェックポイント・プロセスの復元後にのみアプリケーション・コードが実行されます。 このオプションでは、アプリケーションが着信要求を処理する準備ができる前に、より多くのコードを実行する必要があるため、リストア時間が遅くなる可能性があります。 アプリケーションの早期開始コードが、チェックポイントにとって安全かつ受け入れ可能であると判断された場合は、 afterAppStart チェックポイント・オプションを使用することができます。 このオプションを選択すると、アプリケーション・プロセスが復元されたときの起動時間が短縮されます。

アプリケーションの開始時に実行されるコードがアプリケーションにない場合、 beforeAppStart チェックポイントと afterAppStart チェックポイントは同等です。 これらの場合、両方のチェックポイント・オプションがプロセスのチェックポイントを実行してから、構成されたポートが要求を処理できるようになります。 この手順により、 InstantOn チェックポイント・プロセスが復元された後でのみ、アプリケーションのトランスポート・プロトコルが使用可能になります。

早期開始コードの制限事項および考えられる回避策について詳しくは、 InstantOn の制限事項および既知の問題を参照してください。

ランタイムおよびホスト・ビルド・システムの前提条件

Open Liberty バージョン 23.0.0.6以降、すべての X86-64/AMD64 UBI Open Liberty コンテナー・イメージ には、 Open Liberty アプリケーション・プロセスのチェックポイントを指定してリストアするための InstantOn の前提条件が含まれています。 Open Liberty Ubuntu コンテナー・イメージは、 InstantOnに対して有効になっていません。

現在、 InstantOn は、 IBM Semeru Java バージョン 11.0.19+ および IBM Semeru Java バージョン 17.0.7+ でサポートされています。 InstantOn は、リリース時に IBM Semeru Java の新規バージョンをサポートすることが期待されています。 現在、 InstantOn は、他の Java ベンダー実装ではサポートされていません。

アプリケーション・コンテナー・イメージのビルド中に InstantOn を使用するには、ホスト・マシンが以下の前提条件を満たしている必要があります。

  • カーネル・バージョン 5.9 以上の Linux オペレーティング・システムを使用します。

  • X86-64/AMD64 プロセッサー・アーキテクチャーを使用します。

  • Podman または Docker を使用した特権コンテナー・ビルドの実行を許可します。

アプリケーション・コンテナー・イメージ・ビルドで特権コンテナーを使用する必要があるのは、 InstantOnを使用してコンテナー・イメージをビルドする場合のみです。 結果の InstantOn アプリケーション・イメージを実行するために特権コンテナーを使用する必要はありません。

Dockerを使用するには、バージョン 23.0 以上が必要です。 以前のバージョンの Docker では、 CHECKPOINT_RESTORE Linux 機能はサポートされていませんでした。これにより、 InstantOn はコンテナー内のプロセスのチェックポイントやリストアを実行できなくなります。

チェックポイントおよびリストアのための Linux 機能の前提条件

非特権 (非 root) ユーザーは、アプリケーション・プロセスのチェックポイント処理および復元のために CRIU によってサポートされます。 チェックポイント処理と復元を行うために、CRIU には異なる Linux 機能が必要です。

アプリケーション・プロセス・チェックポイントを実行するには、CRIU に以下の Linux 機能が必要です。

  • CHECKPOINT_RESTORE -この機能は、チェックポイント/リストア機能を過負荷の SYS_ADMIN 機能から分離するために、 Linux 5.9 で追加されました。

  • SETPCAP -この機能は、後続のリストアに必要です。

  • SYS_PTRACE -CRIU は、この強力な機能を使用して、完全なプロセス状態をキャプチャーして記録します。 これは、CRIU がアプリケーション・プロセスをチェックポイント処理している場合にのみ必要です。

アプリケーション・プロセスのリストアを実行するには、CRIU に以下の Linux 機能が必要です。

  • CHECKPOINT_RESTORE -この機能は、チェックポイント/リストア機能を過負荷の SYS_ADMIN 機能から分離するために、 Linux 5.9 で追加されました。

  • SETPCAP -この機能により、CRIU はリストアされたアプリケーション・プロセスから機能をドロップできます。 CRIU バイナリーには CHECKPOINT_RESTOREなどの追加機能が付与されているため、CRIU が復元する最後のプロセスからそれらの機能をドロップできるようにするために、 SETPCAP が必要です。

InstantOn アプリケーション・イメージのビルド

InstantOnを使用するアプリケーション・コンテナー・イメージをビルドするために、以下の 2 つのオプションを使用できます。

  • コンテナー・イメージのビルド時にアプリケーション・チェックポイントを実行するために、 checkpoint.sh スクリプト を実行する Dockerfile または Containerfile の最後に、特別な RUN 命令を追加します。 このオプションでは、 Podmanを使用する必要があります。

  • 3 ステップのプロセス を使用して、アプリケーション・イメージをビルドし、チェックポイントを実行し、最終結果を InstantOn アプリケーション・コンテナー・イメージにコミットします。 このオプションでは、 Podman または Docker バージョン 23.0 以降を使用する必要があります。

checkpoint.sh スクリプトを実行するには、 Podman を使用してアプリケーション・コンテナー・イメージをビルドする必要があります。 現在、 Docker を使用して InstantOn アプリケーション・コンテナー・イメージをビルドすることはできません。これは、 Docker が、必要な Linux 機能をコンテナー・ビルドに付与する方法を提供していないためです。 Docker を使用して InstantOn アプリケーション・コンテナー・イメージをビルドするには、3 ステップのビルド・プロセスに従う必要があります。

Podman および checkpoint.sh スクリプトを使用した InstantOn イメージのビルド

checkpoint.sh スクリプトを使用して、 Dockerfile ファイルまたは Containerfile ファイルの末尾に RUN checkpoint.sh 命令を追加することにより、アプリケーション・チェックポイントを実行できます。 checkpoint.sh の実行は、コンテナー・イメージ・ビルド中の最後の RUN 命令でなければなりません。 この構成は、アプリケーション・プロセス・チェックポイントを実行し、プロセス・データをアプリケーション・コンテナー・イメージの最後のレイヤーとして保管します。 現在、 Docker は必要な Linux 機能を付与できないため、このスクリプトでは、 Docker ではなく Podman を使用する必要があります。

以下のイメージ・テンプレートの例では、 kernel-slim-java17-openj9-ubi タグを使用して、Java 17 の IBM Semeru ディストリビューションで最新の Open Liberty リリースを使用するイメージをビルドします。 この例では、 afterAppStart チェックポイント・オプションを使用します。

Dockerfile
FROM icr.io/appcafe/open-liberty:kernel-slim-java17-openj9-ubi

# Add a Liberty server configuration that includes all necessary features
COPY --chown=1001:0  server.xml /config/

# This script adds the requested XML snippets to enable Liberty features and grow the image to be fit-for-purpose.
# This option is available only in the 'kernel-slim' image type. The 'full' and 'beta' tags already include all features.
RUN features.sh

# Add interim fixes (optional)
COPY --chown=1001:0  interim-fixes /opt/ol/fixes/

# Add an application
COPY --chown=1001:0  Sample1.war /config/dropins/

# This script adds the requested server configuration, applies any interim fixes, and populates caches to optimize the runtime.
RUN configure.sh

# This script performs an InstantOn checkpoint of the application.
# The application can use beforeAppStart or afterAppStart to do the checkpoint.
# The default is beforeAppStart when not specified
RUN checkpoint.sh afterAppStart

以下の Podman コマンドを使用して、 InstantOn アプリケーション・コンテナー・イメージをビルドします。 必要な Linux 機能をコンテナー・イメージ・ビルドに付与するには、このコマンドを root ユーザーとして実行するか、 sudo ユーティリティーを使用して実行する必要があります。

podman build \
   -t dev.local/liberty-app-instanton \
   --cap-add=CHECKPOINT_RESTORE \
   --cap-add=SYS_PTRACE\
   --cap-add=SETPCAP \
   --security-opt seccomp=unconfined .

3 つの --cap-add オプションは、コンテナー・イメージのビルド中にアプリケーション・プロセス・チェックポイントを実行するために CRIU が必要とする 3 つの Linux 機能を付与します。 --security-opt オプションは、コンテナー・イメージ・ビルドに対するすべての Linux システム呼び出しへのアクセス権限を付与します。

Docker または Podman での 3 ステップ・プロセスを使用した InstantOn イメージのビルド

コンテナー・イメージのビルド中に Podman を使用して checkpoint.sh を実行できない場合は、以下の 3 ステップのプロセスを使用して、 InstantOn アプリケーション・コンテナー・イメージをビルドできます。

  1. InstantOn レイヤーを使用せずにアプリケーション・コンテナー・イメージをビルドする

  2. アプリケーション・コンテナーを実行して、実行中のコンテナー内のアプリケーションのチェックポイントを実行します。

  3. 停止したコンテナーをチェックポイント・プロセス・データとともに InstantOn アプリケーション・コンテナー・イメージにコミットします。

これらのステップを Podman および Docker で使用して、 InstantOn アプリケーション・イメージをビルドできます。 Dockerの場合、バージョン 23.0 以降が必要です。 以下の例では、 Docker を使用して、 liberty-appという名前のアプリケーション・イメージをビルドすることを想定しています。

1. アプリケーション・コンテナー・イメージのビルド

以下の例のように、イメージ・テンプレート (Dockerfile または Containerfile) を設定します。 この例では、 kernel-slim-java17-openj9-ubi タグを使用して、Java 17 の IBM Semeru ディストリビューションで最新の Open Liberty リリースを使用するイメージをビルドします。 このテンプレートでは、 checkpoint.sh スクリプトは実行されません。

Dockerfile
FROM icr.io/appcafe/open-liberty:kernel-slim-java17-openj9-ubi

# Add a Liberty server configuration that includes all necessary features
COPY --chown=1001:0  server.xml /config/

# This script adds the requested XML snippets to enable Liberty features and grow the image to be fit-for-purpose.
# This option is available only in the 'kernel-slim' image type. The 'full' and 'beta' tags already include all features.
RUN features.sh

# Add interim fixes (optional)
COPY --chown=1001:0  interim-fixes /opt/ol/fixes/

# Add an application
COPY --chown=1001:0  Sample1.war /config/dropins/

# This script adds the requested server configuration, applies any interim fixes, and populates caches to optimize the runtime.
RUN configure.sh

Dockerを使用してアプリケーション・コンテナー・イメージをビルドするには、以下のコマンドを実行します。

docker build -t liberty-app .

liberty-appというタグが付けられた結果のアプリケーション・コンテナー・イメージには、 InstantOn チェックポイント・プロセス層は含まれません。

2. チェックポイントを実行するためのアプリケーション・コンテナーの実行

アプリケーション・コンテナー・イメージを実行して、実行中のコンテナー内でアプリケーション・プロセスのチェックポイントを実行します。 以下の例では、 liberty-app アプリケーション・イメージを使用して、 afterAppStart オプションを指定してアプリケーション・プロセスのチェックポイントを実行します。

docker run \
  --name liberty-app-checkpoint-container \
  --privileged \
  --env WLP_CHECKPOINT=afterAppStart \
  liberty-app

このコマンドは、コンテナー内でアプリケーションを実行し、アプリケーション・プロセス・チェックポイントを実行します。 --env オプションは、チェックポイント afterAppStart オプションを指定するために WLP_CHECKPOINT 環境変数を設定します。 アプリケーション・プロセスのチェックポイントが完了すると、 liberty-app-checkpoint-container アプリケーション・コンテナーは停止し、終了します。

3. チェックポイント・プロセス・データを使用して、停止したコンテナーをコミットします。

前のステップで停止した liberty-app-checkpoint-container コンテナーには、 InstantOn チェックポイント・プロセスからのデータが含まれています。 最後に、以下のコミット・コマンドを実行して、このチェックポイント・プロセス・データを取得し、それをアプリケーション・コンテナー・イメージ層にコミットします。

docker commit liberty-app-checkpoint-container liberty-app-instanton
docker rm liberty-app-checkpoint-container

これで、 liberty-app と liberty-app-instantonの 2 つのアプリケーション・イメージが作成されました。 liberty-app-instanton コンテナー・イメージを使用してコンテナーを開始すると、元の liberty-app イメージよりも起動時間が短縮されます。 停止した liberty-app-checkpoint-container コンテナーは不要になったため、安全に削除できます。

InstantOn アプリケーション・イメージの実行およびデプロイ

InstantOn アプリケーション・イメージをローカルで実行する場合、またはパブリック・クラウドにデプロイする場合は、特別な考慮事項が必要です。 InstantOn チェックポイント・プロセスをリストアするには、以下の前提条件が必要です。

  1. コンテナー・イメージを実行するホストは、 Linux カーネル 5.9 以上を使用する必要があります。

  2. Linux 機能の CHECKPOINT_RESTORE および SETPCAP が、実行中のコンテナーに付与されている必要があります。

  3. 必要なシステム呼び出しが、実行中のコンテナーに付与されている必要があります。

  4. ホスト・プロセッサーは X86-64/AMD64 でなければなりません。

ローカルでの InstantOn アプリケーション・イメージの実行

ホスト・システムが X86-64/AMD64 プロセッサーを使用して Linux カーネル 5.9 以上を実行している場合は、 Podman または Docker をローカルで使用して InstantOn アプリケーション・イメージを実行できます。 以下のコマンドは、 Podmanを使用して liberty-app-instanton InstantOn アプリケーション・イメージを実行します。

podman run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=SETPCAP \
  --security-opt seccomp=unconfined \
  -p 9080:9080 \
  liberty-app-instanton

以下のコマンドは、 Dockerを使用して liberty-app-instanton InstantOn アプリケーション・イメージを実行します。

docker run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=SETPCAP \
  --security-opt seccomp=unconfined \
  -p 9080:9080 \
  liberty-app-instanton

どちらの場合も、 --cap-add オプションは CHECKPOINT_RESTORE 機能と SETPCAP 機能を付与します。 InstantOn アプリケーション・コンテナー・イメージを実行するために SYS_PTRACE 機能は必要ありません。

必須の Linux システム・コール

--security-opt オプションは、実行中のコンテナーに、すべての Linux システム・コールへのアクセス権限を付与します。 コンテナー・エンジンのデフォルトによっては、 seccomp-unconfined 設定の --security-opt は必要ない場合があります。 CRIU が InstantOn アプリケーション・プロセスをリストアするには、コンテナーが clone3、 ptrace、およびその他のシステム呼び出しにアクセスできる必要があります。 この要件は、 SYS_PTRACE の昇格された Linux 機能がプロセスのリストアに必要とされない場合でも当てはまります。 コンテナー・エンジンのデフォルトを更新して、必要なすべてのシステム呼び出しを組み込むことができます。

あるいは、コンテナーのポリシーを指定する --security-opt seccomp オプションを使用してファイルを指定することもできます。 以下のコマンドを使用して、 seccompの JSON ポリシー・ファイルを指定します。

podman run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=NET_ADMIN \
  --cap-add=SYS_PTRACE \
  --security-opt seccomp=criuRequiredSysCalls.json \
  -p 9080:9080 \
  liberty-app-instanton

結果の criuRequiredSysCalls.json ファイルにより、CRIU が InstantOn アプリケーション・プロセスをリストアするために必要とするすべての Linux システム・コールへのアクセス権限が付与されます。

失敗した InstantOn リストアからのリカバリー

InstantOn アプリケーション・プロセスのリストアが失敗した場合、 Open Liberty は、 InstantOn チェックポイント・プロセスを使用せずにサーバーを始動します。 このような場合、 Open Liberty アプリケーションは、 InstantOn チェックポイント・プロセス層が存在しないかのように開始します。これは、正常にリストアされた InstantOn プロセスよりも時間がかかります。 失敗した InstantOn リストアからのこのリカバリー起動は、以下の環境変数を設定することで無効にできます。

CRIU_RESTORE_DISABLE_RECOVERY=true

InstantOn アプリケーション・コンテナー・イメージをビルドした後、この環境変数をローカルで実行するように設定することで、リストアが正常に行われたことを確認できます。 例えば、以下の Podman コマンドを実行できます。

podman run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=SETPCAP \
  --security-opt seccomp=unconfined \
  --env CRIU_RESTORE_DISABLE_RECOVERY=true \
  -p 9080:9080 \
  liberty-app-instanton

失敗したアプリケーション・コンテナー・イメージの開始をクラウド環境が継続的に再始動しようとしないようにするため、 CRIU_RESTORE_DISABLE_RECOVERY 変数のデフォルト値は falseです。

Kubernetes サービスへの InstantOn アプリケーションのデプロイ

現在、 Open Liberty InstantOn は、以下のパブリック・クラウド Kubernetes サービスでテストおよびサポートされています。

  • Amazon Elastic Kubernetes Service (EKS)

  • Azure Kubernetes Service (AKS)

InstantOn アプリケーション・プロセスを復元するための 前提条件 がある場合は、他のパブリック・クラウド Kubernetes サービスも機能する可能性があります。

Kubernetesにデプロイする場合は、 InstantOn アプリケーション・プロセスをリストアできるように、コンテナーに CHECKPOINT_RESTORE 機能と SETPCAP Linux 機能を付与する必要があります。 コンテナーに以下の securityContext を指定することで、デプロイメント YAML ファイルでこれらの機能を構成できます。

        securityContext:
          allowPrivilegeEscalation: true
          privileged: false
          runAsNonRoot: true
          capabilities:
            add:
            - CHECKPOINT_RESTORE
            - SETPCAP
            drop:
            - ALL

Open Liberty InstantOn のサポート対象フィーチャー

InstantOn は、 Open Liberty フィーチャーのサブセットをサポートしています。 InstantOn がサポートしていない機能が有効になっている場合、アプリケーション・プロセスのチェックポイントを実行しようとすると障害が発生します。 InstantOn は、以下の Jakarta EE および MicroProfile コンビニエンス・フィーチャーをサポートします。

  • Jakarta EE Web Profile バージョン 8.0 以降

  • MicroProfile バージョン 4.1 以降

アプリケーションのニーズに応じて、 Jakarta EE Web Profile および MicroProfile フィーチャーによって有効にされた Open Liberty パブリック・フィーチャーを個別に有効にすることができます。 このオプションは、コンビニエンス・フィーチャーによって使用可能になっているフィーチャーの完全なセットを使用可能にしないようにします。

InstantOn は、コンビニエンス・フィーチャーで有効になっているフィーチャーに加えて、以下のフィーチャーもサポートしています。

  • audit-1.0

  • bells-1.0

  • distributedMap-1.0

  • federatedRegistry-1.0

  • ldapRegistry-3.0

  • monitor-1.0

  • openidConnectClient-1.0

  • passwordUtilities-1.1

  • restConnector-2.0

  • sessionDatabase-1.0

  • socialLogin-1.0

  • webCache-1.0

制限について詳しくは、 InstantOn の制限と既知の問題を参照してください。
