MicroProfile フォールト・トレランスを使用した非同期プログラミング

MicroProfile Fault Tolerance は、障害の処理に役立つアノテーションを提供します。 @Asynchronous フォールト・トレランス・アノテーションを CompletionStage インターフェースと組み合わせて、フォールト・トレランスを備えた非同期コードを作成することができます。

@Asynchronous アノテーション

@Asynchronous アノテーションが付けられたメソッドは非同期で実行されます。これは、メソッドが呼び出されたときにメイン・スレッドで即時に実行されないことを意味します。 代わりに、メソッドは後で (通常は別のスレッドで) 実行されます。 @Asynchronous アノテーションを使用するには、アノテーションをメソッドに追加し、アノテーションを付けたメソッドが CompletionStage オブジェクトまたは Future オブジェクトのいずれかを返すようにします。 以下の例では、 CompletableFuture.completedFuture() メソッドを使用して、結果が CompletionStage 戻りタイプと互換性を持つようにします。

@Asynchronous
public CompletionStage<String> serviceA() {
    ...
    return CompletableFuture.completedFuture("serviceA");
}

通常、メソッドが呼び出されると、結果が返されます。 ただし、メソッドに @Asynchronous アノテーションを付けると、フォールト・トレランスが介入します。 アノテーション付きメソッドが呼び出されると、フォールト・トレランスは、別のスレッドで後で実行するようにメソッドをスケジュールし、 CompletionStage オブジェクトまたは Future オブジェクトを返します。 このオブジェクトは、メソッドがまだ実行されていないためにメソッドから返される CompletionStage または Future オブジェクトではありません。 フォールト・トレランスが作成するオブジェクトです。 その後、メソッドが実行され、 CompletionStage オブジェクトまたは Future オブジェクトが返されます。

メソッドが Future オブジェクトを返す場合、フォールト・トレランスは、以前にユーザーに返した Future オブジェクトを更新します。 オブジェクトが更新され、メソッドから返された Future オブジェクトに委任されます。これにより、ユーザーはメソッドからの結果を表示できます。 メソッドが CompletionStage オブジェクトを返す場合、フォールト・トレランスは、 CompletionStage オブジェクトが完了するのを待ってから、以前にユーザーに返した CompletionStage オブジェクトを完了します。

@Asynchronous アノテーションのユース・ケース

以下の 2 つのユース・ケースは、 @Asynchronous アノテーションを使用して非同期プログラムの回復力を高める方法を示しています。

  • 非同期 API 呼び出しに適用されるフォールト・トレランス

  • 回復力と並行して実行される複数のアクション

非同期 API 呼び出しに適用されるフォールト・トレランス

@Asynchronous アノテーションを使用して、 CompletionStage オブジェクトを返す非同期 API 呼び出しにフォールト・トレランス・ポリシーを適用できます。 @Asynchronous アノテーションがない場合、 @Retryなどの他のフォールト・トレランス・アノテーションは、期待どおりに動作しません。 以下のユース・ケースでは、JAX-RS クライアント API から .rx() メソッドがリモート REST サービスを非同期に呼び出します。

private Client client = ClientBuilder.newClient();

public CompletionStage<String> clientDemo() {
    CompletionStage<String> response = client.target("http://example.com/resource")
        .request(MediaType.TEXT_PLAIN)
        .rx()
        .get(String.class);
    return response;
}

アノテーションを付けずに clientDemo() メソッドを呼び出すと、 CompletionStage<String> という戻りの型を受け取ります。この例では、 response という名前になります。 次に、 CompletionStage オブジェクトの完了後に実行するアクションを追加できます。

response.thenAccept(System.out::println);
--> responseText

要求が失敗した場合は、 @Retry アノテーションを使用して要求を再試行することができます。 この例では、要求が失敗してもメソッドが例外をスローしないため、 @Retry アノテーションを使用しても機能しません。 このメソッドは、常に CompletionStage オブジェクトを返します。このオブジェクトを介して、要求の成功または失敗が報告されます。 ただし、 @Asynchronous アノテーションを追加すると、メソッドが戻ったときではなく、 CompletionStage オブジェクトが完了したときに @Retry アノテーションが適用されます。

@Asynchronous
@Retry
public CompletionStage<String> clientDemo() {
    ...
}

JAX-RS .rx() メソッドの使用について詳しくは、 Reactive JAX-RS クライアントを使用した RESTful サービスのコンシュームに関するガイドを参照してください。

回復力と並行して実行される複数のアクション

複数のメソッドを並行して実行するには、以下の例に示すように、他のサービスを呼び出すメソッドを作成し、それらに @Asynchronous アノテーションを付け、それらを呼び出すことができます。

@Inject
private RequestScopedClass1 requestScopedBean1;

@Inject
private RequestScopedClass2 requestScopedBean2;

public CompletionStage<String> callServicesAsynchronously()  {
    CompletionStage<String> result1 = requestScopedBean1.serviceA(); // Where serviceA is annotated with @Asynchronous
    CompletionStage<String> result2 = requestScopedBean2.serviceB(); // Where serviceB is annotated with @Asynchronous
    ...
}

この例では、 serviceA() メソッドが呼び出され、次に serviceB() メソッドが呼び出されます。 ただし、両方のサービスに @Asynchronous アノテーションが付けられているため、これらのサービスは順次実行されるのではなく、異なるスレッドで同時に実行されます。

その他のフォールト・トレランス・アノテーションは、 @Asynchronous アノテーションを使用して実装することもできます。 例えば、 @Retry アノテーションを serviceA() メソッドに追加し、 @Timeout アノテーションを serviceB() メソッドに追加することができます。

@RequestScoped
public class RequestScopedClass1 {

    @Retry
    @Asynchronous
    public CompletionStage<String> serviceA() {
        doSomethingWhichMightFail()
        return CompletableFuture.completedFuture("serviceA");
    }
}

@RequestScoped
public class RequestScopedClass2 {

    @Timeout
    @Asynchronous
    public CompletionStage<String> serviceB() {
        doSomethingWhichMightFail()
        return CompletableFuture.completedFuture("serviceB");
    }
}

この場合、 serviceA() メソッドが何回か再試行する必要がある場合、結果を取得するための呼び出し ( CompletionStage.thenAccept() メソッドなど) は、すべての再試行が完了するまで戻りません。

@Asynchronous アノテーションを使用した実行フローの相違点

@Asynchronous アノテーションを使用してメソッドにアノテーションを付けると、実行のフローが変化します。 以下の図は、フォールト・トレランス・アノテーションが @Asynchronous アノテーションなしでどのように連動するかを示しています。

@Asynchronous アノテーションを付けない実行フロー

この図では、フォールト・トレランス・アノテーション付きメソッドの呼び出しは、少なくとも 1 回の試行で構成されます。 再試行が有効になっている場合は、複数回試行される可能性があります。 各試行では、以下のステップが実行されます。

  • 回路ブレーカーが検査されます。 回路ブレーカーが開いている場合は、試行の結果として CircuitBreakerException 例外が発生し、試行の残りのステップはスキップされます。

  • 回路ブレーカーが開いていない場合は、タイムアウト・タイマーが開始され、メソッドはバルクヘッド上にスペースを予約しようとします。

  • バルクヘッドが満杯でない場合は、予約が取得され、メソッドが実行されます。 この試行の結果は、返された値、またはメソッドによってスローされた例外です。 メソッドの実行中にタイムアウトになると、メソッドを実行しているスレッドが中断されます。 メソッドが例外を返すかスローすると、バルクヘッド予約が解放され、バルクヘッド上のスペースが解放されます。

  • ただし、バルクヘッドが満杯の場合、メソッドは実行されず、試行の結果は BulkheadException 例外になります。

  • 次に、タイムアウトが停止します。 タイムアウトが停止する前に期限切れになった場合、試行の結果は TimeoutException 例外に置き換えられます。

  • 試行の結果は、回路ブレーカーによって記録されます。

これで試行が完了し、結果が表示されます。 再試行が必要な場合、結果は破棄され、新しい試行が開始されます。これは、前のステップが繰り返されることを意味します。 試行が完了し、再試行が必要なくなった後で、最後の試行で例外が発生した場合は、フォールバックが必要になることがあります。 フォールバックが必要かどうかは、フォールバック構成によって異なります。 フォールバックが必要な場合は、フォールバックが実行され、フォールバックの結果が前の結果に置き換わります。 最後に、結果がユーザーに返されます。

次の図は、フォールト・トレランス・アノテーションが @Asynchronous アノテーションとどのように連動するかを示しています。 緑色の 「将来または CompletionStageを戻す」 ボックスは、前の図のボックスに加えて、以下のようになります。

@Asynchronous アノテーションが付いた実行フロー

@Asynchronous アノテーションを使用すると、実行フローで以下の変更が行われます。

  • メソッドが実行される前に、 CompletionStage オブジェクトまたは Future オブジェクトが返されます。 メソッドが実行されると、メソッドからの結果が CompletionStage オブジェクトまたは Future オブジェクトに伝搬され、呼び出し元がアクセスできるようになります。

  • バルクヘッドは、実行を受け入れるか拒否するかに加えて、後で実行するために実行をキューに入れることもできます。 このメソッドがバルクヘッドによって受け入れられると、即時ではなく別のスレッドで実行されるようにスケジュールされます。

  • タイムアウトを使用すると、タイムアウトの有効期限が切れた場合にメソッドが中断されます。 タイムアウトの有効期限が切れると、ダイアグラムの 「タイムアウトの有効期限」 ボックスに示されているポイントまで実行がスキップされます。 結果は、メソッドが TimeoutException 例外で終了したかのように処理されます。

  • フォールバックが発生すると、フォールバックも非同期で実行され、別のスレッドで実行するようにスケジュールされます。

他のフォールト・トレランス・アノテーションとの対話

@Asynchronous アノテーションを使用してメソッドにアノテーションを付けると、以下のフォールト・トレランス・アノテーションに影響を与えます。

  • @Bulkhead

  • @Timeout

@Bulkhead アノテーションとの対話

@Asynchronous アノテーションと @Bulkhead アノテーションを一緒に使用すると、最大数の要求が既に実行されている場合に、フォールト・トレランスによって要求をキューに入れるオプションが提供されます。 メソッドの呼び出し時に実行されている同時要求の数が最大数より少ない場合、メソッドは即時に実行されるようにスケジュールされます。

メソッドの 1 回の実行が終了したときに要求がキュー内にある場合は、キューからの最初の要求が開始されます。 キューがいっぱいになると、メソッドは BulkheadException 例外で失敗します。 キューのサイズは、 @Bulkhead アノテーションの waitingTaskQueue パラメーターを使用して構成できます。

@Timeout アノテーションとの対話

@Asynchronous アノテーションと @Timeout アノテーションを一緒に使用すると、呼び出し元に返される CompletionStage オブジェクトまたは Future オブジェクトは、タイムアウトの期限が切れるとすぐに完了できます。 メソッドがまだ実行されている場合でも、別のスレッドで実行されているため、結果が作動可能であることを別のスレッドにシグナル通知できます。 メソッドを実行しているスレッドは、作業を停止してリソースを節約できるように中断されます。

中断されたことに応答しない長期実行操作にタイムアウトを適用する必要がある場合は、 @Asynchronous アノテーションを使用できます。 タイムアウトが期限切れになり、 TimeoutException 例外を受け取った場合でも、操作は完了するまで実行される可能性があります。

Future オブジェクトを返す場合の の制限

@Asynchronous アノテーションは、 Future オブジェクトを返すメソッドを非同期で実行させることができますが、 フォールト・トレランス・ポリシーは、 CompletionStage オブジェクトを返す非同期メソッドにのみ適用できます。 Future オブジェクトには、そのメソッドの結果を取得する 2 つの方法があります。 get() メソッドを使用してブロックして待機するか、 isDone() メソッドを使用してポーリングします。 非同期結果にフォールト・トレランスを実装するには、結果を待機またはポーリングする 2 番目のスレッドを必要としないようにコールバックが必要です。 CompletionStage オブジェクトにより、この必要なコールバックが容易になります。

コールバックがない場合、フォールト・トレランスは、メソッドの結果ではなく、メソッド呼び出しの前後に適用されます。 Future オブジェクトにはコールバックがないため、フォールト・トレランスを実装すると、以下の問題が発生します。

  • Future オブジェクトの結果が例外であっても、 Future オブジェクトが返されるとすぐに、メソッド呼び出しは成功したと見なされます。

  • Future オブジェクトが完全でない場合でも、メソッドが戻るとバルクヘッドが解放されます。

  • Future オブジェクトが完全でない場合でも、メソッドが戻るとタイムアウトは終了します。

このような問題があるため、 Future オブジェクトを返すことは、操作を並行して実行する場合にのみ適しています。 このような状況では、メソッドが return CompletableFuture.completedFuture(result); ステートメントで終了することがよくあります。これは、例外的に完了した Future オブジェクトを返すことができないことを意味します。 メソッドが例外をスローするか、成功した Future オブジェクトを返します。
