Jakarta EE 概述

Jakarta EE (以前称为 Java EE) 是用于构建企业应用程序的开放式源代码 Java 平台。 Open Liberty 实现了 Jakarta EE ，它提供了诸如事务管理，安全性和数据库访问等公共服务的规范。

Jakarta EE 指定应用程序编程接口 (API) 和运行时环境，以开发和运行安全的可扩展应用程序。 API 是允许应用程序使用操作系统或其他应用程序或服务的特定数据或功能的接口。

Jakarta EE 还定义了用于管理应用程序组件的容器模型。 Jakarta EE 应用程序服务器 (例如 Open Liberty) 托管这些容器，并提供为 Jakarta EE 规范开发的任何应用程序都可以使用的标准化系统服务。 这些开源规范通过提供可在不同产品和环境中应用的配置标准，使应用程序更具可移植性和弹性。 这些标准规范帮助开发者专注于编写应用程序，而不是管理低级别基础架构详细信息。 Jakarta EE 广泛用于银行，电子商务，旅行等方面的企业应用。

Java SE 和 Jakarta EE

Jakarta EE 以前由 Oracle 公司拥有，称为 Java Enterprise Edition (Java EE)。 2017 年， Oracle 将 Java EE 移至开放式源代码 Eclipse 基础，此时其名称已更改为 Jakarta EE。 Jakarta EE 提供了社区驱动的开放式源代码模型，其发行版的更新频率高于 Java EE，从而使它能够更快发展以满足现代应用程序的需要。

Jakarta EE 扩展了提供核心 Java 功能的 Java SE。 Java SE 提供了用于运行 Java 应用程序的 Java 运行时环境 (JRE) 和用于开发 Java 应用程序的工具集合的 Java 软件开发包 (SDK)。 JRE 主要由 Java 虚拟机 (JVM) 和 Java 类库 (JCL) 组成。

JVM 是用于解释 Java 应用程序代码的程序。 与 Jakarta EE 一样， JVM 是一个规范。 正如提供了 Jakarta EE 规范的不同实现 (例如 Open Liberty) 一样，也存在 JVM 规范的不同实现，例如 OpenJ9 和 HotSpot。 尽管这些实现中的每个都源自相同的 JVM 规范，但它们之间可能存在性能差异。 例如， Open Liberty 在 OpenJ9 上比在 HotSpot上具有更好的内存占用量和更快的启动时间。

JCL 是一组可动态装入的库， JVM 可以在运行时调用这些库。 这些库通过提供现代操作系统提供的许多公共功能，使 Java 应用程序能够独立于操作系统。

下图显示了 Jakarta EE 与 Java SE 平台之间的关系。 Java SE 平台提供 Java SDK ， JRE ， JVM 和 JCL 的规范。 Jakarta EE 通过指定为企业应用程序开发设计的 API 集合来扩展 Java SE。 Open Liberty 服务器通过实现这些平台指定的功能来支持云本机应用程序。

Jakarta EE Platform

Platform 版本

Java SE 通过一系列版本进行了演变，因为有更多的类添加到 JCL ，引入了新的 API ，并且不推荐使用和替换了一些较旧的函数。 目前，使用长期支持 (LTS) 维护的 Java SE 版本只有 Java SE 8 ， Java SE 11 和 Java SE 17。 在 LTS 发行版之间， Oracle 引入了短期支持发行版，例如 Java 16。 这些短期发行版仅支持 6 个月。 有关在哪些版本的 Java SE Open Liberty 上运行的更多信息，请参阅 Java SE 支持。

Java EE 8 是 Oracle 公司将平台移至 Eclipse 基础之前平台的最后一个发行版，该基础首先将平台作为 Jakarta EE 8 发布。 雅加达 EE 9 于 2020 年抵达。 此发行版中的主要更改是将 Jakarta 软件包名称中的 javax 替换为 jakarta 。 2021 年 2 月， Jakarta EE 9.1 发布。 虽然此发行版未包含来自 V 9.0的任何 API 更改，但它添加了对 Java SE 11 的支持。

Jakarta EE 规范

Jakarta EE 平台的持续开发由 Jakarta 开发者社区通过 Jakarta EE 规范流程 (JESP) 来推进，后者提供了为该平台引入新规范项目的蓝图。 Jakarta EE 兼容运行时 (例如 Open Liberty) 实现规范项目所描述的功能。 每个实现都由社区通过 技术兼容性工具包 (TCK) 进行验证。 TTCK 是用于检查引用实现合规性的一系列测试。 要使规范项目成为正式规范，必须通过开发至少一个满足 TCK 测试的开源引用实现的过程来批准该规范项目。

Open Liberty 是 Jakarta EE 9.1的参考实现，这意味着它实现了由 Jakarta EE Platform 规范定义的所有必需的 Jakarta EE 规范。 在 Open Liberty 上运行的应用程序可以通过在服务器配置中启用相应的 Open Liberty 功能部件 来利用 Jakarta EE API。 下表描述了几个最常用的 Jakarta EE 规范，并列出了它们对应的 Open Liberty 功能部件。 试用 列提供指向 Open Liberty 指南的链接，这些指南演示 Open Liberty 如何实现规范。

Jakarta EE 和 Open Liberty

Open Liberty 团队对 Jakarta EE 的参与可追溯到项目开始时。 该团队在 Jakarta 工作组中活动，在规范委员会和指导委员会等委员会中工作。 我们的开发者是许多 Jakarta EE 规范的落实者，包括 Jakarta Batch， Jakarta Connectors， Jakarta Concurrency和 Jakarta XML Web Services。 他们还在 Jakarta EE Platform 雨伞规范中扮演领导角色。

引入新规范版本后， Open Liberty 将在其 模块化功能部件的新版本中实现这些规范版本。 Open Liberty 零迁移体系结构 使您能够灵活地继续使用同一版本的 API 或根据应用程序的需要升级到更高版本。 在 Open Liberty Beta 发行版 中定期引入新的 Jakarta EE 实现功能部件版本，然后将这些功能部件包括在完整的一般可用性 (GA) 发行版中。

Jakarta EE 完整平台和 Web 概要文件

Jakarta EE 规范定义完整平台 Enterprise Edition。 Web 概要文件是相应定义的完整平台子集。

Web 概要文件

Web 概要文件定义了以现代 Web 应用程序为目标的合理完整堆栈。 此堆栈是完整平台标准 API 的子集，可满足大多数 Web 应用程序的需求。

完整平台

全平台定义了 Jakarta EE 编程模型的全补充。 除 "Web 概要文件" 功能外，完整平台还具有企业所需的高级业务功能规范，例如，连接，企业 bean ，消息传递和应用程序客户机。

安装 Open Liberty 及其可用功能部件时，请考虑这些定义。 每个安装选项都提供 Open Liberty 内核以及支持完整平台或 Web 概要文件的功能部件。 或者，您可以将个别 Jakarta EE 功能部件安装到现有 Open Liberty 内核安装。

此外，在将应用程序部署到 Open Liberty 服务器时，请考虑这些定义。 如果应用程序需要支持企业连接 API ，那么必须向安装添加完整平台功能部件，或者将应用程序部署到具有完整平台功能部件的 Open Liberty 安装。

要快速添加对 Jakarta EE 的支持，请在 server.xml 文件中指定 Web 概要文件 或 Jakarta EE Platform 便利功能。 第三个便利功能部件 Jakarta EE Application Client也可用于快速配置应用程序客户机组件。 有关便利功能的更多信息，请参阅 便利功能。
