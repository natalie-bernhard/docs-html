ACME による自動証明書管理

Automatic Certificate Management Environment (ACME) プロトコルは、Transport Layer Security (TLS) 証明書の発行と検証のプロセスを自動化します。 Open Liberty Automatic Certificate Management Environment (ACME) サポート・フィーチャーを使用して、ACME 認証局 (CA) からサーバー上のドメインのブラウザー・トラステッド TLS 証明書を自動的に取得することができます。

クライアントは、TLS 証明書を使用してサーバー上のドメイン・ネームを認証します。 これらの TLS 証明書は、サーバーがドメインを合法的に制御し、中間者攻撃からクライアントを保護することを検証します。 TLS 証明書は、ドメインに対するサーバー制御が CA によってテストおよび検証されるレコードです。 TLS 証明書を取得するには、サーバーが証明書署名要求 (CSR) を生成し、CA からのいくつかのチャレンジに応答する必要があります。 これらのチャレンジは、検証対象のドメインをサーバーが制御していることを証明します。

以前は、TLS 証明書の発行には、人間がかなり関与する必要がありました。 ACME プロトコル は、サーバー上の自動証明書管理エージェントと ACME CA ( Let 's Encrypt ™など) との間で実行できるように、プロセスを標準化します。

ACME および Open Liberty

Open Liberty サーバーで ACME CA を使用するには、 ACME Support フィーチャー (acmeCA-2.0) を使用可能にして構成する必要があります。 少なくとも、ACME CA の URL と、サーバーが制御する 1 つ以上のドメインの名前を指定する必要があります。 パブリック ACME プロバイダーからコールバックを受け取るには、 HTTP-01 チャレンジ・タイプの標準に準拠するようにポート 80 が開いている必要があります。 多くの内部またはテスト ACME プロバイダーでは、任意のオープン・ポートを使用して ACME チャレンジを完了できます。 ACME テスト・サーバーの使用について詳しくは、 ACME テスト・サーバーの使用を参照してください。

TLS を有効にして Open Liberty サーバーを始動すると、そのサーバー用に自己署名 TLS 証明書が作成されます。 この証明書は、テストおよび開発の目的で使用できます。 ただし、証明書は CA によって署名されていないため、TLS を介してサーバーに接続するクライアントはその証明書を信頼しません。 ブラウザーでは、クライアントがサーバーを信頼するかどうかを確認する必要があります。これにより、テストや開発の際でも、コンテナー化された環境やマイクロサービスに問題が発生する可能性があります。

自己署名証明書の問題を回避するために、Let 's Encrypt などのサービスは ACME プロトコルを使用して、パブリック・インターネット経由で CA 署名 TLS 証明書を無料で提供します。 ACME サポート・フィーチャーが有効になっている場合、新規証明書が必要であれば、 Open Liberty サーバーは始動時に構成済み CA プロバイダーからの証明書を自動的に要求します。 この機能は、CA からのチャレンジに応答し、証明書の更新と取り消しを管理することもできます。

REST API への HTTPS 呼び出しを行うことで、TLS 証明書を表示、更新、または取り消すことができ、アクティブな ACME CA アカウントに関する詳細を取得できます。 REST API を使用するには、実行するアクションに応じて、管理者ロールまたはリーダー・ロールのいずれかを持っている必要があります。 Open Liberty 管理者ロールおよびリーダー・ロールの構成について詳しくは、 アプリケーション・セキュリティー ・フィーチャーを参照してください。

ACME CA アカウント管理

ACME サポート機能には、ACME CA を持つ有効なアカウントが必要です。 アカウントの詳細は、アカウント・キー・ファイルに保管されます。 CSR が送信される前に、このフィーチャーはまず、既存のアカウント・キー・ファイルがあるかどうかを検査します。 アカウント・キー・ファイルが存在しない場合は、新しいファイルが生成されます。 その後、このファイルを使用して、新規アカウントを登録するか、既存のアカウントにログインします。 他のセキュリティー資格情報と同様に、アカウント鍵ファイルをバックアップして保護する必要があります。

REST API を使用して、いつでも新しいアカウント・キーを要求できます。 既存の鍵が漏えいした場合、または定期的なセキュリティー上の予防措置として、アカウント・キーの更新が必要になることがあります。 アカウント・キーを更新するために、管理者役割のユーザーは、以下の HTTPS 要求を送信できます。

curl -kv https://mydomain.com:443/ibm/api/acmeca/account -X post -H "content-type: application/json" -d '{"operation":"renewAccountKeyPair"}'

一部の認証局は、サービスのご利用条件を提供しています。 ACME CA がサービスのご利用条件を変更する場合は、新しいご利用条件に手動で同意する必要がある場合があります。 変更されたサービスのご利用条件に同意しない場合、ACME CA サーバーは、お客様がご使用条件を確認するまで、お客様のアカウントからの今後の要求を拒否する可能性があります。 サービスの中断を回避するには、ACME 構成で、モニター対象の E メール・アドレスを指定するアカウント連絡先を指定します。 詳しくは、 ACME Support フィーチャーを参照してください。

証明書の更新および取り消し

ACME サポート機能は、デフォルトで、有効期限切れまたは取り消された TLS 証明書を自動的に更新します。 renewBeforeExpiration プロパティーは、サーバーが新しい証明書を要求する、証明書の有効期限が切れるまでの時間を指定します。 このプロパティーのデフォルト値は 7 日です。

サーバーは始動時に、CA 署名 TLS 証明書の有効期限を検査します。 有効期限が指定された時間範囲内にある場合、または証明書が既に有効期限切れまたは取り消されている場合、サーバーは新しい TLS 証明書を自動的に要求します。 サーバーは、稼働中に TLS 証明書を毎日検査し、有効期限切れまたは失効した証明書を自動的に更新します。 更新要求が失敗した場合、バックグラウンド・プロセスは、要求が成功するまで 1 時間ごとに新しい TLS 証明書を要求し続けます。 詳しくは、 ACME Support フィーチャーを参照してください。

acmeCA 構成に対するランタイムの変更により、証明書の自動更新要求がトリガーされる可能性があります。 例えば、ディレクトリー URI が別のエンドポイントに変更された場合、サーバーは新しい TLS 証明書を要求します。

また、アクティブな TLS 証明書を手動で更新することも選択できます。 アクティブな証明書を更新するために、管理者ロールのユーザーは、以下の HTTPS 要求を送信できます。

curl -kv https://mydomain.com:443/ibm/api/acmeca/certificate -X POST -u admin:password -H "content-type: application/json" -d '{"operation":"renewCertificate"}'

置き換えられた TLS 証明書は、新しい TLS 証明書のインストール後に自動的に取り消されます。 TLS 証明書が更新されると、新しい接続は新しい TLS 証明書を使用します。 既存の接続は中断されません。

サーバーを廃棄する前、テスト中、または ACME CA から別の CA に移動する前に TLS 証明書を取り消すには、管理者ロールのユーザーが以下の HTTPS 要求を送信できます。

curl -kv https://mydomain.com:443/ibm/api/acmeca/certificate -X POST -u admin:password -H "content-type: application/json" -d '{"operation":"revokeCertificate","reason":"key_compromise"}'

reason キーと値のペアを使用して、取り消し理由を指定できます。 以下の理由値が有効です。

  • unspecified

  • key_compromise

  • ca_compromise

  • affiliation_changed

  • superseded

  • cessation_of_operations

  • certificate_hold

  • remove_from_crl

  • privilege_withdrawn

  • aa_compromise

理由値を指定しない場合、デフォルト値は unspecifiedです。

REST API を使用した ACME CA アカウントおよび証明書モニター

REST API に HTTPS 要求を送信することにより、アクティブな ACME CA アカウントおよびアクティブな TLS 証明書をモニターできます。 アカウントの詳細と連絡先情報を確認したり、現在の CA 署名 TLS 証明書の詳細 (証明書と発行者の有効期間など) を表示したりすることができます。 証明書の更新を要求すると、更新された TLS 証明書情報が表示されます。 ただし、これらの HTTPS エンドポイントから返されるコンテンツは情報提供のみを目的としており、プログラムで使用したり依存したりすることはできません。

アクティブな ACME CA アカウントと TLS 証明書の両方を表示するために、管理者またはリーダーのいずれかの役割を持つユーザーは、以下の HTTPS 要求を送信できます。

curl -kv https://mydomain.com:443/ibm/api/acmeca -X GET -u admin:password

アクティブなアカウントのみを表示するには、管理者ロールまたはリーダー・ロールのいずれかのユーザーが以下の HTTPS 要求を送信できます。

curl -kv https://mydomain.com:443/ibm/api/acmeca/account -X GET -u admin:password

アクティブな TLS 証明書のみを表示するには、管理者ロールまたはリーダー・ロールのいずれかのユーザーが以下の HTTPS 要求を送信できます。

curl -kv https://mydomain.com:443/ibm/api/acmeca/certificate -X GET -u admin:password

以下の例に示すように、ブラウザーを使用し、管理者またはリーダーの役割資格情報を指定して REST エンドポイントにアクセスすることもできます。

https://mydomain.com:443/ibm/api/acmeca
https://mydomain.com:443/ibm/api/acmeca/account
https://mydomain.com:443/ibm/api/acmeca/certificate
