ログおよびトレースの構成

Open Liberty には、アプリケーションとランタイムによって書き込まれたメッセージを処理する統合ロギング・コンポーネントがあり、First Failure Data Capture (FFDC) 機能を提供します。 System.out、 System.err、または java.util.logging.Logger ストリームを使用してアプリケーションによって書き込まれたデータのロギングは、サーバー・ログに結合されます。

サーバーには、以下の 3 つの 1 次ログ・ファイルがあります。

  • console.log -このファイルは、 server start コマンドによって作成されます。 これには、基礎となる JVM プロセスからリダイレクトされた標準出力および標準エラー・ストリームが含まれます。 このコンソール出力は、人間が理解できるようにフォーマットされており、ログ分析の自動化に役立つ情報が不足しています。 console.log ファイルは、サーバーが再始動されるとリセットされます。

  • messages.log - このファイルは、ロギング・コンポーネントによって書き込まれる、または取り込まれるすべてのメッセージを含みます。 このファイルに書き込まれるすべてのメッセージには、メッセージ・タイム・スタンプやメッセージを書き込んだスレッドの ID などの追加情報が含まれます。 このファイルは、自動ログ分析に適しています。 このファイルには、JVM プロセスに直接書き込まれるメッセージは含まれません。

  • trace.log -このファイルは、トレースを有効にした場合にのみ作成されます。 これには、 messages.log ファイルのすべての内容と、使用可能なトレースが含まれます。 このファイルには、JVM プロセスに直接書き込まれるメッセージは含まれません。

ロギング構成

ロギング・コンポーネントは、サーバー構成を通じて制御できます。 ロギング・コンポーネントは、 logging エレメントによって server.xml ファイル内で完全に構成できます。 ただし、ロギングは server.xml ファイルが処理される前に初期化されるため、 server.xml ファイルを使用してロギングを構成すると、早期ログ・エントリーで後続のログ構成とは異なるログ構成が使用される可能性があります。 この問題を回避するには、 boostrap.properties ファイルで、また場合によっては環境変数を使用して、ロギング構成の大部分を提供することができます。 bootstrap.properties ファイルと server.xml ファイルの両方で同じ構成プロパティーに異なる値を指定しないでください。 ロギング構成設定について詳しくは、 ソース別の構成設定 の表を参照してください。

ログ・ファイル・ストレージ管理

console.log ファイルは、プロセス stdout および stderr ストリームをファイルにリダイレクトすることによって作成されます。 結果として、 Open Liberty は、 messages.log ファイルの場合と同じレベルの管理 (ログ・ロールオーバーなど) を提供できません。 console.log ファイルのサイズが心配な場合は、コンソール・ログを OFF に設定し、代わりに messages.log ファイルを使用することができます。 ロギング・コンポーネントによって書き込まれる、または取り込まれるすべてのメッセージは、 messages.log ファイルに書き込まれます。メッセージ・ログでファイル・ロールオーバーを構成することができます。

ただし、コンソール・ログ・レベルを OFFに設定すると、基礎となる JVM からのメッセージは引き続き console.log ファイルに書き込まれます。 ネイティブ標準出力および標準エラー・ストリームに書き込まれるメッセージにも、同じ原則が適用されます。 Open Liberty ロギング・コンポーネントは、 -verbose:gc 出力など、JVM プロセスによって直接書き込まれるメッセージの宛先をキャプチャーしたり制御したりすることはありません。 これらのメッセージは、アクティブな JVM 設定に従ってファイル・システムに経路指定され、 messages.log ファイルには表示されません。

ロギング・コンポーネントがコンソール・ログに書き込まれないようにし、100 Mb で 3 回ロールオーバーするように messages.log ファイルを構成するには、以下の構成を使用します。

com.ibm.ws.logging.max.file.size=100
com.ibm.ws.logging.max.files=3
com.ibm.ws.logging.console.log.level=OFF
com.ibm.ws.logging.copy.system.streams=false

TBASIC ログ・フォーマット

TBASIC ロギング・フォーマットは、 Open Liberty コンソール、メッセージ、およびトレース・ログ・ファイル全体で一貫性のあるログ・フォーマットを提供します。 トレース・ログの BASIC ロギング・フォーマットが存在します。 ただし、トレース・ログとコンソール・ログおよびメッセージ・ログの出力は異なります。 TBASIC フォーマットは、トレース・ログの BASIC オプションと一致するトレース・ログ、コンソール・ログ、およびメッセージ・ログに対して一貫性のあるフォーマット・オプションを提供します。 TBASIC フォーマットは、 BASIC オプションの別名として機能します。

以下の例に示すように、 bootstrap.properties ファイルで TBASIC 形式を指定できます。

com.ibm.ws.logging.message.format= tbasic
com.ibm.ws.logging.console.format= tbasic
com.ibm.ws.logging.trace.format= tbasic

以下の例に示すように、 server.env ファイルで、メッセージおよびコンソール・ログの TBASIC ログ・フォーマットを環境変数として指定することもできます。

WLP_LOGGING_MESSAGE_FORMAT=TBASIC
WLP_LOGGING_CONSOLE_FORMAT=TBASIC

JSON ロギング

JSON 形式でログを生成することにより、ログの構文解析を単純化できます。 JSON は、多くのログ分析ツールがフォーマット固有の構文解析命令を必要とせずに取り込むことができる自己記述型フォーマットです。 bootstrap.properties ファイルを編集するか、環境変数を指定することにより、JSON 形式でログを生成するように Open Liberty ログを構成できます。 以下の 2 つの例は、これらの各オプションの構成を示しています。

  • bootstrap.properties ファイルで JSON ロギングを構成します。

    com.ibm.ws.logging.message.format=json
com.ibm.ws.logging.message.source=message,trace,accessLog,ffdc,audit

  • server.env ファイル内の環境変数を使用して JSON ロギングを構成します。

    WLP_LOGGING_MESSAGE_FORMAT=json
WLP_LOGGING_MESSAGE_SOURCE=message,trace,accessLog,ffdc,audit

構成可能な JSON フィールド名

ログが JSON 形式の場合は、 jsonFieldMappings 属性を使用して、デフォルトのフィールド名を新しいフィールド名に置き換えることができます。 同じロギング構成内の他のサーバーが Open Liberty のデフォルト名とは異なるフィールド名を使用している場合は、デフォルトのフィールド名を置き換える必要が生じることがあります。 例えば、 Open Liberty メッセージは message フィールド名で参照されますが、別のコンテナー内のメッセージは logというフィールドにある場合があります。 この場合、ダッシュボード上のログにメッセージの 2 つの異なる視覚化が表示されます。 Open Liberty 出力フィールド名を変更して他のログと一致させると、それらを同じ視覚化で表示することができます。 以下の例は、JSON フィールドの名前を変更するためのサンプル構成を示しています。

  • 新規フィールド名を構成するには、 server.env ファイルに以下の環境変数を含めることができます。

    WLP_LOGGING_JSON_FIELD_MAPPINGS=loglevel:level

    この例では、 loglevel フィールド名は level フィールド名に置き換えられます。

  • 特定のソースのフィールド名を構成するには、 server.env ファイルに以下の環境変数を含めることができます。

    WLP_LOGGING_JSON_FIELD_MAPPINGS=message:message:log

    この例では、 message フィールド名は、メッセージ・ログ内の log フィールド名に置き換えられます。

ログからフィールドを省略するには、以下の例に示すように、置き換えなしでフィールド名を指定します。

WLP_LOGGING_JSON_FIELD_MAPPINGS=defaultField名前:

複数のフィールドの名前を変更したり、除外したりするには、フィールド名マッピングのコンマ区切りのリストを指定します。

デフォルトの JSON フィールド名の完全なリストについては、 JSON ログ・イベントの参照リストを参照してください。

詳しくは、 構成可能な JSON ログ・フィールド名を参照してください。

カスタム・メッセージおよびトレース JSON フィールド

JSON 形式のメッセージおよびトレース出力にカスタム・フィールドを追加して、特定の問題またはインシデントに関する情報を収集できます。 例えば、特定のユーザーからの要求を確認する場合は、カスタム・フィールドを追加して、そのユーザーの ID でアプリケーション・ログをフィルターに掛けることができます。 セッション ID に別のフィールドを追加して、特定のセッションのアプリケーション・ログを分析およびフィルタリングすることができます。

Open Liberty LogRecordContext API は、ログ・レコードおよびトレース・レコードにカスタム・フィールドを追加できます。 この API は、拡張の値を指定してフィールドを追加します。 LogRecordContext API を使用するには、まず com.ibm.websphere.logging.hpel.LogRecordContext クラスをインポートします。 以下の例は、さまざまな種類のカスタム・フィールドを JSON ログに追加する方法を示しています。

ストリング値フィールドをアプリケーション・ログに追加するには、以下の行をアプリケーションに組み込むことができます。

LogRecordContext.addExtension("userName","bob");

新しく指定されたフィールドは、 addExtension メソッドを実行するスレッドと同じスレッドで作成されるログ項目およびトレース項目に追加されます。 この例では、 bob ユーザー ID 用に userName という名前のカスタム・フィールドが追加されます。

ブール値、浮動小数点値、int 値、または long 値を持つフィールドを追加するには、以下の例に示すように、拡張名に接尾部 _bool、 _float、 _int、または _longを含める必要があります。

LogRecordContext.addExtension("extensionName_bool","true");
LogRecordContext.addExtension("extensionName_int","112233");
LogRecordContext.addExtension("extensionName_float","1.2");
LogRecordContext.addExtension("extensionName_long","132");

これらの接尾部を指定して非ストリング値を追加する場合、結果の JSON フィールド値はログ内で引用符で囲まれません。 ログでは、ストリング値の JSON フィールド値のみが引用符で囲まれます。

ログからカスタム・フィールドを削除するには、以下の方法を使用します。

LogRecordContext.removeExtension(extensionName);

拡張を削除した後、同じスレッドで作成される後続のログおよびトレースの JSON 出力には、そのフィールドは含まれません。

選択可能な HTTP アクセス・ログ JSON フィールド

ログが JSON 形式で、 accessLog ソースが指定されている場合、デフォルトの HTTP アクセス・ログ JSON フィールドを別のフィールド・セットに置き換えることができます。 jsonAccessLogFields 属性を使用して、アクセス・ログでデフォルト・フィールドを使用するか、 logFormat 属性で指定された一連のフィールドを使用するかを指定できます。 置換フィールドは、 httpEndpoint エレメント内の accessLogging エレメントの logFormat 属性で指定します。

この構成では、リモート・ユーザー ID や要求ヘッダーなど、JSON ログでは使用できない情報を受け取ることができます。 これらのログは、Elastic スタックなどのログ分析ツールでサーバーをモニターするために使用できます。 HTTP アクセス・ログのフォーマット・オプションについて詳しくは、 HTTP アクセス・ロギングを参照してください。

以下の例は、デフォルトの HTTP アクセス・ログ・フィールドを logFormat 属性で指定されたフィールドに置き換える、 server.xml ファイル内の構成を示しています。 JSON アクセス・ログを受信するには、 HTTP アクセス・ロギング を有効にする必要があります。

<httpEndpoint httpPort="9080" httpsPort="9443" id="defaultHttpEndpoint">
    <accessLogging logFormat='%R{W} %u %{my_cookie}C %s'/>
</httpEndpoint>
<logging jsonAccessLogFields="logFormat"/>

時間ベースのログ・ロールオーバー

ログ・ロールオーバー開始時刻とログ・ロールオーバー間隔期間を指定することにより、Liberty メッセージおよびトレース・ログの時間ベースの定期的なロールオーバーを有効にすることができます。 指定されたロールオーバー開始時刻は、ログが最初にロールオーバーする時刻です。 ロールオーバー間隔期間は、連続するログ・ロールオーバー間の時間間隔です。

例えば、サーバーのロールオーバー開始時刻が午前 0 時で、ロールオーバー間隔が 1 日の場合、メッセージとトレース・ログは毎日午前 0 時にロールオーバーされます。

Open Liberty メッセージおよびトレース・ログのロールオーバーを構成するには、 bootstrap.properties ファイルを編集して環境変数を指定するか、サーバー構成属性を使用します。 以下の例は、これらの各オプションの構成を示しています。ログ・ロールオーバーの開始時刻は午前 0 時、ログ・ロールオーバーの間隔は 1 日です。

  • 以下の例では、 bootstrap.properties ファイルで時間ベースのログ・ロールオーバーを構成します。

    com.ibm.ws.logging.rollover.start.time=00:00
com.ibm.ws.logging.rollover.interval=1m

  • 以下の例では、 server.env ファイル内の環境変数を使用して、時間ベースのログ・ロールオーバーを構成します。

    WLP_LOGGING_ROLLOVER_START_TIME=00:00
WLP_LOGGING_ROLLOVER_INTERVAL=1m

  • 以下の例では、 server.xml ファイル内のサーバー構成属性を使用して、時間ベースのログ・ロールオーバーを構成します。

    <logging rolloverStartTime="00:00" rolloverInterval="1d"/>

トレース・ログの詳細レベル

server.xml ファイル内のロギング・エレメントに traceSpecification 属性を指定して、トレースを選択的に有効にし、サーバー構成内の特定のコンポーネントのトレース・ログ詳細レベルを設定することができます。 以下の例では、2 つのコンポーネント・グループのトレース仕様を指定しています。

<logging traceSpecification="com.myco.mypackage1.*=fine:com.myco.mypackage2.*=finest"/>

この例では、2 つの詳細レベル指定がコロン (:) で区切られています。

com.myco.mypackage1.*=fine 値は、 com.myco.mypackage1で始まるパッケージ名を持つすべてのクラスに適用されます。

com.myco.mypackage2.*=finest 値は、 com.myco.mypackage2で始まるパッケージ名を持つすべてのクラスに適用されます。
