JDBC を使用したリレーショナル・データベース接続

Java アプリケーションは、Java Database Connectivity (JDBC) を介してリレーショナル・データベースに接続し、リレーショナル・データベースと対話します。 API。 Open Liberty サーバー上で実行されているアプリケーションがリレーショナル・データベースに接続できるように、データ・ソースと JDBC ドライバーを構成することができます。

アプリケーションは、リレーショナル・データベースと通信して、フライト・スケジュール、製品インベントリー、顧客の購入履歴など、さまざまな種類の情報を取得します。 JDBC API は、データベースに関する詳細を標準化された方法でアプリケーションに提供することにより、アプリケーションとリレーショナル・データベースの間のアダプター層を提供します。 Open Libertyでは、 JDBC API との対話は、 Java Database Connectivity フィーチャーによって構成されます。

JDBC ドライバー・ライブラリー構成

リレーショナル・データベースに接続するには、通常はデータベース・ベンダーによって提供される JDBC ドライバーが必要です。 JDBC ドライバーを構成して、データベースへの接続を取得するデータ・ソースを定義することができます。 Open Liberty サーバー構成で JDBC データ・ソースを構成するには、Java Database Connectivity フィーチャーを使用可能にして、 JDBC ドライバーを含むライブラリーを指定する必要があります。 以下の例では、 library エレメントは、 JDBC ドライバー JAR ファイルを含むディレクトリーの場所を指定しています。

<featureManager>
    <feature>jdbc-4.3</feature>
</featureManager>

<library id="jdbcLib">
    <fileset dir="jdbc" includes="*.jar"/>
</library>

Open Liberty は、一般的に使用される JDBC ドライバーのさまざまなデータ・ソース・タイプの実装クラス名を認識します。 ほとんどの場合、 JDBC ドライバーの場所のみを指定する必要があります。

JDBC ドライバーのトレースの構成について詳しくは、 JDBC トレースを参照してください。

Maven または Gradle を使用した JDBC ドライバー構成

Maven または Gradle ビルド・ツールを使用してアプリケーションをビルドする場合は、 JDBC ドライバー・ファイルを ${server.config.dir}/jdbc ディレクトリーにコピーするように依存関係を構成できます。 ドライバー・ファイルは、サーバーが作成された後、アプリケーションがデプロイされる前にコピーする必要があります。これにより、実行時にアプリケーションがデータベース・インスタンスを使用できるようになります。

Maven では、 Open Liberty Maven プラグイン の copyDependencies エレメントを使用して、 JDBC ドライバーの依存関係を作成できます。

Maven を使用してアプリケーションをビルドする場合、以下の例のようなコードを pom.xml ファイルに追加することで、 JDBC ドライバーをダウンロードしてデプロイできます。 この例では、ドライバー・ファイルは、 ${server.config.dir} ディレクトリーにある jdbc ディレクトリーにコピーされます。

<plugin>
<groupId>io.openliberty.tools</groupId>
<artifactId>liberty-maven-plugin</artifactId>
<version>3.3.4</version>
  <configuration>
    <copyDependencies>
      <dependencyGroup>
       	<location>jdbc</location>
     	  <dependency>
            <groupId>com.ibm.db2</groupId>
            <artifactId>jcc</artifactId>
            <version>11.5.6.0</version>
      	  </dependency>
      </dependencyGroup>
     </copyDependencies>
   </configuration>
</plugin>

この例では、Liberty Maven プラグインのバージョン 3.3.4 を使用します。 別のバージョンを使用するには、 version 値を更新して、使用するバージョンを指定します。 詳しくは、 Open Liberty Maven プラグインを使用した依存関係のコピーを参照してください。

Open Liberty Gradle プラグイン は、 JDBC ドライバー・ライブラリーの構成を直接サポートしていません。 Gradleを使用して コピー・タスク を build.gradle ファイルに手動で追加し、 JDBC 依存関係を管理する必要があります。

Gradle を使用してアプリケーションをビルドする場合は、まず copy タスクを build.gradle ファイルに追加します。 次に、そのタスクの依存関係を deploy タスクに追加して、サーバーが作成された後、アプリケーションがデプロイされる前に copy タスクが実行されるようにします。

configurations {
  jdbcLib
}
dependencies {
  jdbcLib 'com.ibm.db2:jcc:11.5.6.0'
}
task copyJDBC(type: Copy) {
  from configurations.jdbcLib
  into '$buildDir/wlp/usr/servers/<YourServerName>/jdbc'
  include '*.jar'
}
deploy.dependsOn 'copyJDBC'

この build.gradle ファイルの例では、 jdbcLib という名前の configurations インスタンスが、ドライバーの依存関係を使用して宣言されています。 copyJDBC タスクは、ドライバー JAR ファイルがサーバー構成ディレクトリー内の jdbc フォルダーにコピーされることを指定します。 最後に、 deploy 依存関係は、アプリケーションをデプロイする前にドライバーをサーバーにコピーして、実行時に使用できるようにする必要があることを指定します。

データ・ソース構成

Open Libertyを使用して任意の JDBC ドライバーを構成できます。これには、多くの一般的なベンダー・データベース用の組み込み構成が含まれます。 以下の例は、 server.xml ファイルでデータ・ソースを構成するための基本パターンを示しています。

<library id="jdbcLib">
    <fileset dir="jdbc" includes="*.jar"/>
</library>

<dataSource jndiName="jdbc/myDB">
    <jdbcDriver libraryRef="jdbcLib"/>
    <properties serverName="localhost" portNumber="5432"
                databaseName="myDB"
                user="exampleUser" password="examplePassword"/>
</dataSource>

この例では、 dataSource エレメントは、 JDBC ドライバー JAR ファイルを含むライブラリーを参照し、 properties 属性を使用していくつかの JDBC ベンダー・プロパティーを指定します。

すべての JDBC ドライバーには、その dataSource 実装クラスで構成できるさまざまなプロパティー・コレクションが用意されています。 JDBC ドライバーのデータ・ソースに、String パラメーターまたはプリミティブ・パラメーターを持つ setter メソッドがある場合、 dataSource エレメントの下に単一の properties サブエレメントまたは properties.{JDBC_VENDOR_TYPE} サブエレメントを指定することで、これらのプロパティーを構成できます。 使用している JDBC ドライバー用に Liberty が properties.{JDBC_VENDOR_TYPE} サブエレメントを提供している場合は、そのサブエレメントを使用します。 それ以外の場合は、 properties サブエレメントを使用します。 Liberty が提供するベンダー固有の properties.{JDBC_VENDOR_TYPE} サブエレメントについて詳しくは、 dataSource エレメントを参照してください。

以下の例は、 properties サブエレメントで JDBC ベンダー・プロパティーを指定するための基本構成を示しています。

<dataSource jndiName="jdbc/myDB" jdbcDriverRef="myDriver"/>
    <properties someProperty="someValue" anotherProperty="5" />
</dataSource>

ベンダー固有のプロパティー・サブエレメントの例として、 Open Liberty が properties.db2.jcc サブエレメントを提供する IBM Db2 JDBC ドライバーを考えてみます。 以下のアプリケーション・コードは、 IBM Db2 JDBC ドライバー・データ・ソース・クラスの currentLockTimeout プロパティーを示しています。

public void setCurrentLockTimeout(int lockTimeout);
public int getCurrentLockTimeout();

以下の例に示すように、 properties.db2.jcc サブエレメントを指定することで、 server.xml ファイルでこの設定を構成できます。

<dataSource jndiName="jdbc/myDB">
    <jdbcDriver libraryRef="jdbcLib"/>
    <properties.db2.jcc
                serverName="localhost" portNumber="50000" databaseName="myDB"
                user="exampleUser" password="examplePassword"
                currentLockTimeout="30s"/>
</dataSource>

データ・ソースを構成した後、構成を素早くテストして、 Open Liberty サーバーがデータベースにアクセスできるかどうかを確認できます。 詳しくは、 データベースへの接続の検証を参照してください。

デフォルト・データ・ソースの構成

Open Libertyで Java EE フィーチャーまたは Jakarta EE フィーチャーを使用可能にする場合、デフォルトのデータ・ソースを構成することができます。 デフォルトのデータ・ソースを構成するには、以下の例に示すように、 dataSource エレメントの ID を DefaultDataSourceに設定します。

<dataSource id="DefaultDataSource">
<jdbcDriver libraryRef="jdbcLib"/>
<properties serverName="localhost" portNumber="5432"
            databaseName="myDB"
            user="exampleUser" password="examplePassword"/>
</dataSource>

Oracle UCP

Oracle Universal Connection Pool (UCP) は、スタンドアロンの JDBC 接続プールです。 Oracle UCP を Open Libertyと共に使用する場合は、 Open Liberty 組み込み接続プール機能の代わりに Oracle UCP 接続プールを使用します。 一部の Oracle 高可用性データベース 機能では、 Oracle UCP を使用する必要があります。 Oracle UCP のサポートは、 Open Liberty バージョン 19.0.0.4で追加されました。

Oracle UCP では、 user や passwordなどの一部のプロパティーを properties.oracle.ucp エレメントで設定する必要がある場合があります。 Open Liberty 接続プールが使用不可であるため、一部の Open Liberty データ・ソースおよび接続マネージャーの構成値は無視されます。 これらのデータ・ソースおよび接続マネージャーのプロパティーのほとんどについて、 Oracle UCP は同等の機能を提供します。 詳しくは、 properties.oracle.ucp エレメントの資料を参照してください。

Oracle UCP JDBC ドライバーを Maven Centralから取得します。 以下の例は、 Oracle UCP のサンプル・データ・ソース構成を示しています。

<dataSource jndiName="jdbc/oracleUCPDS" >
    <jdbcDriver libraryRef="OracleUCPLib" />
    <properties.oracle.ucp URL="jdbc:oracle:thin:@//localhost:1521/SAMPLEDB" />
</dataSource>

<library id="OracleUCPLib">
    <fileset dir="Oracle/Drivers" includes="ojdbcx.jar ucp.jar"/>
</library>

Oracle RAC

Oracle Real Application Clusters (RAC) は、ノードと呼ばれる 2 つ以上のデータベース・インスタンスをまとめて、単一システムとして動作するクラスター・データベースを形成するオプションです。 Oracle JDBC ドライバーを使用すると、 Oracle RAC 環境でフェイルオーバー・サポート、ロード・バランシング、またはその両方を構成できます。 Oracle RAC は、高可用性と柔軟なスケーラビリティーの両方を提供し、通常、 Oracle UCP とともに使用されます。

以下の例は、2 つのノードで構成される Oracle RAC データベースのサンプル・データ・ソース構成を示しています。

<dataSource jndiName="jdbc/oracleUCPDS">
  <jdbcDriver libraryRef="OracleUCPLib" />
  <properties.oracle
  URL="jdbc:oracle:thin:@(DESCRIPTION=(FAILOVER=ON)(LOAD_BALANCE=OFF)(ADDRESS=(PROTOCOL=TCP)(HOST=example-host-1)(PORT=example-port-1))
  (ADDRESS=(PROTOCOL=TCP)(HOST=example-host-2)(PORT=example-port-2))(CONNECT_DATA=(SERVER=DEDICATED)(SERVICE_NAME=name)))"
  user="username" password="password" />
</dataSource>

<library id="OracleUCPLib">
    <fileset dir="Oracle/Drivers" includes="ojdbcx.jar ucp.jar"/>
</library>

この例では、 example-host-1 と example-port-1 は最初のノードのホストとポートの値を表し、 example-host-2 と example-port-2 は 2 番目のノードのホストとポートの値を表します。 FAILOVER パラメーターと LOAD_BALANCE Oracle パラメーターは、両方のノードのグローバル構成を指定します。 Oracle データベース・パラメーターについて詳しくは、 Oracle RAC の資料を参照してください。 Oracle サービスを使用していない場合、 SERVICE_NAME の値はデータベース名です。 Oracle サービスを使用している場合、 SERVICE_NAME の値はサービスの名前になります。 Oracle ログイン資格情報を Oracle プロパティーとして指定する必要はありません。 その他のデータベース認証方式 も機能します。 一部の Oracle RAC 機能では、 Oracle UCP を使用する必要があります。これは、 Open Liberty バージョン 19.0.0.4 以降でのみ使用可能です。

Open Liberty

以下の例は、 Open Liberty がデフォルトで認識しないリレーショナル・データベースのサンプル・データ・ソース構成を示しています。 dataSource エレメントの type 属性を使用して、データ・ソースのタイプを指定します。 type 属性の値は、 「データ・ソース・タイプ」 セクションで説明されているインターフェース・クラス名のいずれかにすることができます。 次に、以下の例に示すように、 jdbcDriver エレメントで、そのクラスのドライバー実装へのインターフェース・クラス名のマッピングを指定します。

<dataSource id="myDB" jndiName="jdbc/myDB" type="javax.sql.XADataSource">
    <jdbcDriver libraryRef="jdbcLib"
               javax.sql.XADataSource="com.example.jdbc.SampleXADataSource"/>
    <properties serverName="localhost" portNumber="1234"
                databaseName="myDB"
                user="exampleUser"
                password="examplePassword"/>
</dataSource>

詳しくは、 Java Database Connectivity フィーチャーを参照してください。

データ・ソース・タイプ

Open Liberty アプリケーションからデータベースにアクセスするには、アプリケーション・コードで javax.sql.DataSource インターフェースを使用する必要があります。 Open Liberty は、このインターフェースの管理対象実装を提供します。この実装は、 JDBC ドライバーが提供するデータ・ソースまたはドライバーの実装によって支えられています。 Open Libertyの場合、 JDBC ドライバーは、 ServiceLoader 機能を使用して、以下のタイプのデータ・ソースまたは java.sql.Driver ドライバー実装の少なくとも 1 つを提供する必要があります。

  • javax.sql.DataSource この基本形式のデータ・ソースには、接続イベント・リスナーやステートメント・イベント・リスナーなど、接続プールを最適化するために Liberty で使用される機能がありません。 さらに、複数のリソースを必要とする 2 フェーズ・トランザクションに参加することもできません。

  • javax.sql.ConnectionPoolDataSource このタイプのデータ・ソースでは、 接続プールが有効になっています。 複数のリソースを必要とするトランザクションで 2 フェーズ対応リソースとして参加できません。

  • javax.sql.XADataSource このタイプのデータ・ソースは、接続プールに対応しており、複数のリソースを必要とするトランザクションに 2 フェーズ対応リソースとして参加することができます。 javax.sql.XADataSource データ・ソース・タイプは、基本的に、 javax.sql.DataSource データ・ソース・タイプおよび javax.sql.ConnectionPoolDataSource データ・ソース・タイプによって提供される機能のスーパーセットです。 ただし、 JDBC ベンダーによっては、 JDBC 仕様に記載されていない動作や制限に微妙な違いがある場合があります。

  • java.sql.Driver java.sql.Driver ドライバー実装は、データベースに接続するための基本的な方法を提供します。 この実装には URL が必要であり、通常は Java SE アプリケーションで使用されます。 javax.sql.DataSourceと同様に、接続プーリングを強化するインターオペラビリティーを提供しないため、複数のリソースを必要とするトランザクションに 2 フェーズ対応リソースとして参加することはできません。 Open Libertyで作業するには、この実装で ServiceLoader 機能を提供する必要があります。この機能は、URL の JDBC ドライバー実装をディスカバーするために Open Liberty で使用されます。

type 属性が指定されていない場合、 Open Liberty は条件付き順序でデータ・ソース・タイプを探し、使用可能な最初のタイプを選択します。

Java Database Connectivity フィーチャー 4.3 以降を使用している場合、または デフォルトのデータ・ソースを参照している場合、 Open Liberty は、以下の順序でデータ・ソース・タイプを探します。

  1. javax.sql.XADataSource

  2. javax.sql.ConnectionPoolDataSource

  3. javax.sql.DataSource

Java Database Connectivity フィーチャー 4.2 以前を使用していて、デフォルトのデータ・ソースを参照していない場合、 Open Liberty は以下の順序でデータ・ソース・タイプを探します。

  1. javax.sql.ConnectionPoolDataSource

  2. javax.sql.DataSource

  3. javax.sql.XADataSource

リレーショナル・データベース接続用の アプリケーション構成

server.xml ファイルで構成されているデータ・ソースを使用するには、データ・ソースを注入するか、アプリケーション・コードでルックアップを指定します。 以下の例では、 server.xml ファイルの dataSource エレメントに jdbc/myDB の jndiName 値が指定されていることを前提としています。

Web コンポーネントまたはエンタープライズ Bean コンポーネントでは、以下の例のようなアプリケーション・コードを使用してデータ・ソースを注入できます。

@Resource(lookup = "jdbc/myDB")
DataSource myDB;

myDB 値が デフォルト・データ・ソースとして構成されている場合は、以下の例に示すように、 lookup オブジェクトを省略できます。

@Resource
DataSource myDB;

Java Naming and Directory Interface フィーチャー が使用可能になっている場合、Java Naming and Directory Interface (JNDI) 検索によってアプリケーションからデータ・ソースを参照できます。 以下の例は、 myDB データ・ソース値の JNDI 検索を示しています。

DataSource myDB = InitialContext.doLookup("jdbc/myDB");

myDB 値がデフォルトのデータ・ソースとして構成されている場合、JNDI 検索では、以下の例に示すように、JNDI 名の代わりに java:comp/DefaultDataSource 値を指定できます。

DataSource myDB = InitialContext.doLookup("java:comp/DefaultDataSource");

も参照してください。
