使用 Open Liberty InstantOn 更快启动容器化应用程序

Open Liberty InstantOn 为 MicroProfile 和 Jakarta EE 应用程序提供快速启动时间。 使用 InstantOn，应用程序可以以毫秒为单位启动，而不会影响吞吐量，内存，开发生产奇偶性校验或 Java 语言功能。

InstantOn 使用 Linux 内核的 "检查点/在用户空间中复原" (CRIU) 功能部件来获取稍后可复原的 JVM 的检查点。 使用 InstantOn，可正常开发应用程序，然后在构建应用程序容器映像时， InstantOn 会生成应用程序进程的检查点。 在复原应用程序时，它将在同一 JVM 中运行，这将提供开发与生产之间的完全奇偶性校验。 因为检查点进程仅需要几秒钟，所以您的 CI/CD 进程几乎没有受到影响。

构建应用程序容器映像时， InstantOn 会创建额外的映像层，其中包含 Open Liberty 应用程序进程的检查点。 应用程序映像启动时， Open Liberty 运行时将识别 InstantOn 层，并将应用程序进程复原到应用程序容器映像构建期间创建的检查点。

Open Liberty 和 OpenJ9 包含参与检查点和复原过程的挂钩。 这些挂钩允许 Open Liberty 运行时为检查点准备进程，以便将检查点进程保存在容器映像中是安全的。 此准备步骤确保可以将进程从容器映像中的同一持久状态复原到应用程序的多个可能并发实例中。 有关 OpenJ9 和 IBM Semeru JVM 中的 CRIU 支持的更多信息，请参阅 Java CRIU 支持 文档。

无法在容器映像构建外部使用 InstantOn 。 应用程序容器映像提供了一致的环境，这是确保 Open Liberty 应用程序进程的可靠复原所必需的环境。 InstantOn 容器层是应用程序容器映像的最后一层。 此配置可确保映像的底层层中的资源不会从采用检查点的时间更改为使用 InstantOn启动映像的时间。

以下部分描述了构建和运行 InstantOn 应用程序映像的先决条件和过程:

  • 何时获取检查点: beforeAppStart 或 afterAppStart

  • 运行时和主机构建系统先决条件

  • 构建 InstantOn 应用程序映像

  • 运行和部署 InstantOn 应用程序映像

  • Open Liberty InstantOn 支持的功能部件

何时创建检查点: beforeAppStart 或 afterAppStart

当发生 InstantOn 检查点时，将启动 Open Liberty 运行时。 在此启动期间，运行时将处理配置，装入所有已启用的功能部件，并开始处理已配置的应用程序。 此启动顺序根据应用程序的大小和复杂性以及它需要的 Open Liberty 功能部件数而有所不同。 对于小型简单应用程序，此过程可能需要少于 1 秒。 对于更典型的应用程序，可能需要几秒钟。 InstantOn 在 Open Liberty 运行时完成启动后采用检查点。 有两个选项可用于确定检查点是在应用程序本身启动之前还是之后发生，可以将其指定为 InstantOn RUN 配置的一部分:

  • afterAppStart -在应用程序启动后执行检查点。 此选项提供了最快的启动，但可能不适用于某些应用程序。

  • beforeAppStart -在运行任何应用程序代码之前执行检查点。 如果应用程序代码与 afterAppStart 选项不兼容，请使用此选项。

您选择的选项取决于应用程序必须运行的代码。 Jakarta EE 和 MicroProfile 应用程序可能包含在应用程序启动时运行的应用程序代码，例如以下示例:

  • 使用 loadOnStartup 属性的 servlet

  • 使用 @Startup 注释的 EJB

  • 使用 @Observes @Initialized(ApplicationScoped.class) 注释的 CDI Bean

有时，在应用程序启动时运行的应用程序代码可能不适合执行 InstantOn 检查点。 对于此类情况，请使用 beforeAppStart 选项。 例如，必须在 InstantOn 检查点之前避免以下应用程序代码场景，因此更适合 beforeAppStart 选项:

  • 访问远程资源，例如数据库。 在应用程序容器构建期间，不太可能有正确的数据源可用于连接。

  • 正在创建事务。 目前，在 InstantOn 检查点之前禁止事务。

  • 读取在部署应用程序时预期会更改的配置，例如 MicroProfile Config 中的配置。

在这些情况下使用 beforeAppStart 选项可确保仅在复原 InstantOn 检查点进程之后运行应用程序代码。 此选项可能会导致较慢的复原时间，因为在应用程序准备好为任何入局请求提供服务之前，它必须运行更多代码。 如果确定应用程序早期启动代码是安全的并且对于检查点是可接受的，那么可以使用 afterAppStart 检查点选项。 此选项提供了恢复应用程序进程时最快的启动时间。

如果应用程序没有在应用程序启动时运行的代码，那么 beforeAppStart 和 afterAppStart 检查点等效。 在这些情况下，两个检查点选项都将先执行进程的检查点，然后再启用已配置的端口来处理请求。 此序列确保仅在复原 InstantOn 检查点进程之后才启用应用程序的传输协议。

有关早期启动代码的限制以及可能的变通方法的更多信息，请参阅 InstantOn 限制和已知问题。

运行时和主机构建系统先决条件

从 Open Liberty V 23.0.0.6开始，所有 X86-64/AMD64 UBI Open Liberty 容器映像 都包含 InstantOn 用于检查点和复原 Open Liberty 应用程序进程的先决条件。 Open Liberty Ubuntu 容器映像未针对 InstantOn启用。

当前， IBM Semeru Java V 11.0.19+ 和 IBM Semeru Java V 17.0.7+ 支持 InstantOn 。 InstantOn 预期在发布时支持新版本的 IBM Semeru Java。 目前， InstantOn 在其他 Java 供应商实现上不受支持。

要在应用程序容器映像构建期间使用 InstantOn ，主机必须满足以下先决条件:

  • 将 Linux 操作系统与内核版本 5.9 或更高版本配合使用

  • 使用 X86-64/AMD64 处理器体系结构

  • 允许使用 Podman 或 Docker 执行特权容器构建

仅当使用 InstantOn构建容器映像时，才需要在应用程序容器映像构建中使用特权容器。 运行生成的 InstantOn 应用程序映像不需要使用特权容器。

要使用 Docker，需要版本 23.0 或更高版本。 较早版本的 Docker 不支持 CHECKPOINT_RESTORE Linux 功能，这将阻止 InstantOn 在容器中执行进程的检查点或复原。

用于检查点和复原的 Linux 功能先决条件

CRIU 支持非特权 (非 root) 用户进行检查点和复原应用程序进程。 要检查点和复原， CRIU 需要不同的 Linux 功能。

要执行应用程序进程检查点， CRIU 需要以下 Linux 功能:

  • CHECKPOINT_RESTORE -此功能是在 Linux 5.9 中添加的，用于将检查点/复原功能与重载的 SYS_ADMIN 功能分开。

  • SETPCAP -此功能是后续复原所必需的。

  • SYS_PTRACE -CRIU 使用此强大功能来捕获和记录完整进程状态。 仅当 CRIU 正在检查应用程序进程时，才有必要执行此操作。

要执行应用程序进程复原， CRIU 需要以下 Linux 功能:

  • CHECKPOINT_RESTORE -此功能是在 Linux 5.9 中添加的，用于将检查点/复原功能与重载的 SYS_ADMIN 功能分开。

  • SETPCAP -此功能允许 CRIU 从复原的应用程序进程中删除功能。 由于 CRIU 二进制文件被授予额外的功能 (如 CHECKPOINT_RESTORE) ，因此它需要 SETPCAP ，以便可以从 CRIU 复原的最终进程中删除这些功能。

构建 InstantOn 应用程序映像

有两个选项可用于构建使用 InstantOn的应用程序容器映像:

  • 在运行 checkpoint.sh 脚本 的 Dockerfile 或 Containerfile 末尾添加特殊 RUN 指令，以在容器映像构建时执行应用程序检查点。 此选项要求您使用 Podman。

  • 使用 三步流程 来构建应用程序映像，运行检查点，并将最终结果落实到 InstantOn 应用程序容器映像中。 此选项要求您使用 Podman 或 Docker V 23.0 或更高版本。

要运行 checkpoint.sh 脚本，必须使用 Podman 来构建应用程序容器映像。 目前，您无法使用 Docker 来构建 InstantOn 应用程序容器映像，因为 Docker 未提供授予容器构建必需的 Linux 功能的方法。 要使用 Docker 来构建 InstantOn 应用程序容器映像，必须遵循三步构建过程。

使用 Podman 和 checkpoint.sh 脚本构建 InstantOn 映像

通过将 RUN checkpoint.sh 指令添加到 Dockerfile 或 Containerfile 文件的末尾，可以使用 checkpoint.sh 脚本来执行应用程序检查点。 checkpoint.sh 的执行必须是容器映像构建期间的最后一条 RUN 指令。 此配置执行应用程序进程检查点并将进程数据存储为应用程序容器映像的最后一层。 当前，此脚本要求您使用 Podman 而不是 Docker ，因为 Docker 无法授予必要的 Linux 功能。

以下映像模板示例使用 kernel-slim-java17-openj9-ubi 标记来构建一个映像，该映像使用最新的 Open Liberty 发行版以及 Java 17 的 IBM Semeru 分发版。 此示例使用 afterAppStart 检查点选项。

Docker 文件
FROM icr.io/appcafe/open-liberty:kernel-slim-java17-openj9-ubi

# Add a Liberty server configuration that includes all necessary features
COPY --chown=1001:0  server.xml /config/

# This script adds the requested XML snippets to enable Liberty features and grow the image to be fit-for-purpose.
# This option is available only in the 'kernel-slim' image type. The 'full' and 'beta' tags already include all features.
RUN features.sh

# Add interim fixes (optional)
COPY --chown=1001:0  interim-fixes /opt/ol/fixes/

# Add an application
COPY --chown=1001:0  Sample1.war /config/dropins/

# This script adds the requested server configuration, applies any interim fixes, and populates caches to optimize the runtime.
RUN configure.sh

# This script performs an InstantOn checkpoint of the application.
# The application can use beforeAppStart or afterAppStart to do the checkpoint.
# The default is beforeAppStart when not specified
RUN checkpoint.sh afterAppStart

使用以下 Podman 命令来构建 InstantOn 应用程序容器映像。 要将必需的 Linux 功能授予容器映像构建，必须以 root 用户身份或通过使用 sudo 实用程序来运行此命令。

podman build \
   -t dev.local/liberty-app-instanton \
   --cap-add=CHECKPOINT_RESTORE \
   --cap-add=SYS_PTRACE\
   --cap-add=SETPCAP \
   --security-opt seccomp=unconfined .

三个 --cap-add 选项授予 CRIU 在容器映像构建期间执行应用程序进程检查点所需的三个 Linux 功能。 --security-opt 选项授予对容器映像构建的所有 Linux 系统调用的访问权。

通过将三步过程与 Docker 或 Podman 配合使用来构建 InstantOn 映像

如果在容器映像构建期间无法使用 Podman 来运行 checkpoint.sh ，那么可以使用以下三步过程来构建 InstantOn 应用程序容器映像:

  1. 在没有 InstantOn 层的情况下构建应用程序容器映像

  2. 运行应用程序容器以在正在运行的容器中执行应用程序的检查点

  3. 将包含检查点流程数据的已停止容器落实到 InstantOn 应用程序容器映像中

您可以将这些步骤与 Podman 和 Docker 配合使用，以构建 InstantOn 应用程序映像。 对于 Docker，需要 V 23.0 或更高版本。 以下示例假定您正在使用 Docker 来构建名为 liberty-app的应用程序映像。

1。 构建应用程序容器映像

设置类似于以下示例的图像模板 (Dockerfile 或 Containerfile)。 此示例使用 kernel-slim-java17-openj9-ubi 标记来构建一个映像，该映像将最新的 Open Liberty 发行版与 Java 17 的 IBM Semeru 发行版配合使用。 此模板不会运行 checkpoint.sh 脚本。

Docker 文件
FROM icr.io/appcafe/open-liberty:kernel-slim-java17-openj9-ubi

# Add a Liberty server configuration that includes all necessary features
COPY --chown=1001:0  server.xml /config/

# This script adds the requested XML snippets to enable Liberty features and grow the image to be fit-for-purpose.
# This option is available only in the 'kernel-slim' image type. The 'full' and 'beta' tags already include all features.
RUN features.sh

# Add interim fixes (optional)
COPY --chown=1001:0  interim-fixes /opt/ol/fixes/

# Add an application
COPY --chown=1001:0  Sample1.war /config/dropins/

# This script adds the requested server configuration, applies any interim fixes, and populates caches to optimize the runtime.
RUN configure.sh

要使用 Docker构建应用程序容器映像，请运行以下命令:

docker build -t liberty-app .

生成的标记为 liberty-app的应用程序容器映像不包含 InstantOn 检查点进程层。

2。 运行应用程序容器以执行检查点

运行应用程序容器映像以在正在运行的容器中执行应用程序进程的检查点。 以下示例使用 liberty-app 应用程序映像通过 afterAppStart 选项运行应用程序进程的检查点:

docker run \
  --name liberty-app-checkpoint-container \
  --privileged \
  --env WLP_CHECKPOINT=afterAppStart \
  liberty-app

此命令在容器中运行应用程序并执行应用程序进程检查点。 --env 选项设置 WLP_CHECKPOINT 环境变量以指定检查点 afterAppStart 选项。 当应用程序进程检查点完成时， liberty-app-checkpoint-container 应用程序容器将停止并退出。

3。 使用检查点流程数据落实已停止的容器

先前步骤中停止的 liberty-app-checkpoint-container 容器包含来自 InstantOn 检查点进程的数据。 最后，通过运行以下落实命令，获取此检查点进程数据并将其落实到应用程序容器映像层:

docker commit liberty-app-checkpoint-container liberty-app-instanton
docker rm liberty-app-checkpoint-container

现在有两个应用程序映像: liberty-app 和 liberty-app-instanton。 使用 liberty-app-instanton 容器映像启动容器会显示比原始 liberty-app 映像更快的启动时间。 不再需要 liberty-app-checkpoint-container 已停止的容器，可以安全地将其移除。

运行和部署 InstantOn 应用程序映像

在本地运行 InstantOn 应用程序映像或将其部署到公共云时，需要特殊注意事项。 复原 InstantOn 检查点进程需要以下先决条件。

  1. 运行容器映像的主机必须使用 Linux 内核 5.9 或更高版本

  2. 必须将 Linux 功能 CHECKPOINT_RESTORE 和 SETPCAP 授予正在运行的容器

  3. 必须将必需的系统调用授予正在运行的容器

  4. 主机处理器必须为 X86-64/AMD64

在本地运行 InstantOn 应用程序映像

如果主机系统使用 X86-64/AMD64 处理器运行 Linux 内核 5.9 或更高版本，那么可以在本地使用 Podman 或 Docker 来运行 InstantOn 应用程序映像。 以下命令使用 Podman运行 liberty-app-instanton InstantOn 应用程序映像:

podman run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=SETPCAP \
  --security-opt seccomp=unconfined \
  -p 9080:9080 \
  liberty-app-instanton

以下命令使用 Docker运行 liberty-app-instanton InstantOn 应用程序映像:

docker run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=SETPCAP \
  --security-opt seccomp=unconfined \
  -p 9080:9080 \
  liberty-app-instanton

在这两种情况下， --cap-add 选项都会授予 CHECKPOINT_RESTORE 和 SETPCAP 功能。 不需要 SYS_PTRACE 功能即可运行 InstantOn 应用程序容器映像。

必需的 Linux 系统调用

--security-opt 选项授予正在运行的容器对所有 Linux 系统调用的访问权。 根据容器引擎的缺省值，可能不需要具有 seccomp-unconfined 设置的 --security-opt 。 要让 CRIU 复原 InstantOn 应用程序进程，容器必须具有对 clone3， ptrace和其他系统调用的访问权。 即使不需要提升的 Linux 功能 ( SYS_PTRACE ) 来复原进程，也符合此要求。 您可以更新容器引擎的缺省值以包含所有必需的系统调用。

或者，您可以使用 --security-opt seccomp 选项指定用于指定容器策略的文件。 使用以下命令为 seccomp指定 JSON 策略文件:

podman run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=NET_ADMIN \
  --cap-add=SYS_PTRACE \
  --security-opt seccomp=criuRequiredSysCalls.json \
  -p 9080:9080 \
  liberty-app-instanton

生成的 criuRequiredSysCalls.json 文件授予对 CRIU 复原 InstantOn 应用程序进程所需的所有 Linux 系统调用的访问权。

从失败的 InstantOn 复原中恢复

如果 InstantOn 应用程序进程的复原失败，那么 Open Liberty 将在不使用 InstantOn 检查点进程的情况下启动服务器。 在此类情况下， Open Liberty 应用程序会像不存在 InstantOn 检查点进程层一样启动，这需要比成功复原的 InstantOn 进程更长的时间。 可以通过设置以下环境变量来禁用从失败的 InstantOn 复原启动的此恢复:

CRIU_RESTORE_DISABLE_RECOVERY=true

构建 InstantOn 应用程序容器映像后，可以通过将此环境变量设置为在本地运行来验证复原是否成功。 例如，可以运行以下 Podman 命令:

podman run \
  --rm \
  --cap-add=CHECKPOINT_RESTORE \
  --cap-add=SETPCAP \
  --security-opt seccomp=unconfined \
  --env CRIU_RESTORE_DISABLE_RECOVERY=true \
  -p 9080:9080 \
  liberty-app-instanton

为避免云环境持续尝试重新启动应用程序容器映像的失败启动， CRIU_RESTORE_DISABLE_RECOVERY 变量的缺省值为 false。

将 InstantOn 应用程序部署到 Kubernetes 服务

目前，已在以下公共云 Kubernetes 服务上测试并支持 Open Liberty InstantOn :

  • Amazon Elastic Kubernetes Service (EKS)

  • Azure Kubernetes 服务 (AKS)

如果其他公共云 Kubernetes 服务具有允许 InstantOn 应用程序进程复原的 先决条件 ，那么这些服务也可能工作。

部署到 Kubernetes时，必须向容器授予 CHECKPOINT_RESTORE 和 SETPCAP Linux 功能，以允许复原 InstantOn 应用程序进程。 您可以在部署 YAML 文件中通过为容器指定以下 securityContext 来配置这些功能:

        securityContext:
          allowPrivilegeEscalation: true
          privileged: false
          runAsNonRoot: true
          capabilities:
            add:
            - CHECKPOINT_RESTORE
            - SETPCAP
            drop:
            - ALL

Open Liberty InstantOn 支持的功能部件

InstantOn 支持 Open Liberty 功能部件的子集。 如果启用了 InstantOn 不支持的功能，那么尝试执行应用程序进程的检查点时将发生故障。 InstantOn 支持以下 Jakarta EE 和 MicroProfile 便利功能部件:

  • Jakarta EE Web Profile V 8.0 和更高版本

  • MicroProfile V 4.1 和更高版本

根据应用程序的需要，您可以单独启用由 Jakarta EE Web Profile 和 MicroProfile 功能部件启用的 Open Liberty 公共功能部件。 此选项可避免启用由方便功能部件启用的完整功能部件集。 但是， InstantOn 当前不支持独立的 MicroProfile 功能部件，这些功能部件是任何便利功能部件都未启用的 MicroProfile 功能部件。

除了在 MicroProfile 和 Jakarta 便利功能中启用的功能外， InstantOn 还支持以下功能:

  • audit-1.0

  • bells-1.0

  • distributedMap-1.0

  • federatedRegistry-1.0

  • ldapRegistry-3.0

  • monitor-1.0

  • openidConnectClient-1.0

  • passwordUtilities-1.1

  • restConnector-2.0

  • sessionDatabase-1.0

  • socialLogin-1.0

  • webCache-1.0

有关限制的更多信息，请参阅 InstantOn 限制和已知问题。
