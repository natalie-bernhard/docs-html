Logstash コレクター・イベントの参照リスト

Logstash コレクター機能を使用してリモート Logstash サーバーにログ・イベントを送信し、 Elasticsearch や Kibana などの製品でそれらのイベントを管理および視覚化することができます。 各タイプのログ・イベントには、Kibana ダッシュボードをカスタマイズするために使用できる独自のフィールド・セットがあります。

Logstash コレクター 機能は、実行時にログ・イベントをキャプチャーし、それらをフィールドに分割し、構成済みの Logstash ログ収集サーバーに安全に転送します。 詳しくは、 Logstash コレクターを使用した Logstash へのログおよびイベントの転送を参照してください。

イベント・タイプ

Logstash コレクター機能は、以下のイベント・タイプを生成します。

  • メッセージ・イベント

  • トレース・イベント

  • HTTP アクセス・イベント

  • First Failure Data Capture (FFDC) イベント

  • IBM JDK の ガーベッジ・コレクション・イベント

  • 監査イベント

デフォルトのログおよびトレース・フレームワークに加えて、Logstash コレクター機能は、バイナリー・ロギングが有効になっている場合にメッセージ・イベントおよびトレース・イベントを転送します。 ログ・イベント・タイプについて詳しくは、 ログ管理を参照してください。

サポートされる監査イベントとその監査データ

Open Liberty 「監査」 機能は、サーバー・ランタイム環境およびアプリケーションから監査可能イベントをキャプチャーします。 監査イベントから生成されたデータを使用して、構成された環境を分析することができます。 監査イベントの例については、 JSON ログ・イベントのリファレンス・リスト: 監査イベントを参照してください。

Open Liberty は、JSON 形式または CADF 形式のいずれかで監査イベントを生成できます。 監査イベントは、以下の JSON 形式のタイプでキャプチャーされ、構成された環境を改善できるさまざまな領域を識別するのに役立ちます。

  • 監査サービスの管理 (SECURITY_AUDIT_MGMT)

  • SCIM 操作/メンバー管理 (SECURITY_MEMBER_MGMT)

  • Servlet 3.0 API: ログイン/認証 (SECURITY_API_AUTHN)

  • Servlet 3.0 API: ログアウト (SECURITY_API_AUTHN_TERMINATE)

  • フォーム・ログアウト (SECURITY_AUTHN_TERMINATE)

  • 基本認証 (SECURITY_AUTHN)

  • クライアント証明書認証 (SECURITY_AUTHN)

  • フォーム・ログイン認証 (SECURITY_AUTHN)

  • サーブレット runAs 委任 (SECURITY_AUTHN_DELEGATION)

  • EJB 委任 (SECURITY_AUTHN_DELEGATION)

  • 基本認証へのフェイルオーバー (SECURITY_AUTHN_FAILOVER)

  • 無保護サーブレット許可 (SECURITY_AUTHZ)

  • JACC Web 許可 (SECURITY_AUTHZ)

  • JACC EJB 許可 (SECURITY_AUTHZ)

  • EJB 許可 (SECURITY_AUTHZ)

  • JMS 認証 (SECURITY_JMS_AUTHN)

  • JMS 許可 (SECURITY_JMS_AUTHZ)

  • SAF 許可サービス API 要求 (SECURITY_SAF_AUTHZ)

  • SAF 許可例外 (SECURITY_SAF_AUTHZ_DETAILS)

  • JMX MBean 登録 (JMX_MBEAN_REGISTER)

  • JMX MBean 操作 (JMX_MBEAN)

  • JMX MBean 属性操作 (JMX_MBEAN_ATTRIBUTES)

  • JMX 通知 (JMX_NOTIFICATION)

仮想化環境での サーバー名およびホスト名

Open Liberty サーバーがコンテナーまたはその他の仮想化環境で実行されている場合、 hostName および serverName イベント・フィールドは、構成内の特定の変数に従って自動的に設定されます。

hostName フィールドは、構成で使用可能な以下の値の最初の値に自動的に設定されます。

  • CONTAINER_HOST 環境変数の値。

  • ${defaultHostName} Open Liberty 構成変数の値。

  • JDK で報告される正規ホスト名

serverName フィールドは、構成で使用可能な以下の値の最初の値に自動的に設定されます。

  • CONTAINER_NAME 環境変数の値。

  • ${wlp.server.name} Open Liberty 構成変数の値。

コンテナーで Logstash コレクター機能を使用する場合、コンテナーの開始時に CONTAINER_HOST 環境変数と CONTAINER_NAME 環境変数を設定できます。 これらの環境変数を設定することにより、Logstash コレクター・フィーチャーは、送信するレコードに適切なホスト名およびコンテナー名のタグを付けることができます。これは、問題判別に役立ちます。 これらの環境変数を設定しないと、複数のコンテナーからのイベントを表示するダッシュボードを使用するときに、イベントの発生元のコンテナーを判別できない場合があります。

コンテナーを開始するときに、以下の例のようなコマンドを使用して、これらの環境変数を設定できます。

docker run -d -e LICENSE=accept -e CONTAINER_NAME=yourContainerName -e CONTAINER_HOST=yourContainerHost --name=yourContainerName yourImageName

Podman を使用してコンテナーを管理する場合は、以下のコマンドを実行します。

podman run -d -e LICENSE=accept -e CONTAINER_NAME=yourContainerName -e CONTAINER_HOST=yourContainerHost --name=yourContainerName yourImageName

Open Liberty サーバーが IBM Cloud® Kubernetes Serviceで実行される場合、 CONTAINER_HOST 環境変数と CONTAINER_NAME 環境変数は既に設定されています。
