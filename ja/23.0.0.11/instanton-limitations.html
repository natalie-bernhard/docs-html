InstantOn の制限および既知の問題

一般的な InstantOn の前提条件に加えて、 Open Liberty InstantOn には特定の制限が適用されます。 例えば、早期開始コードを実行する必要があるアプリケーションや、特定の Liberty フィーチャーに依存するアプリケーションでは、 InstantOnを使用するために変更が必要になる場合があります。

InstantOn の前提条件について詳しくは、 ランタイムおよびホスト・ビルド・システムの前提条件を参照してください。

以下のセクションでは、 Open Liberty InstantOnの使用に関する制限と既知の問題について説明します。

  • Jakarta Transactions 構成の制限

  • チェックポイント前の Jakarta Transaction

  • MicroProfile 構成へのアクセスが早すぎる

  • DataSource の注入が早すぎる

  • InstantOnでサポートされていない製品拡張、ユーザー機能、または機能の使用

  • bootstrap.properties ファイルを使用した構成の更新

  • Java SecurityManager はサポートされていません

  • JVM オプションの更新

  • SELinux の制限

  • 山 Linux セキュリティー・モジュールの制限

  • Linux システム・コールへのアクセス

  • 必要な Linux 機能を使用しない実行

  • サポートされるプロセッサー

Jakarta Transactions 構成の制限

Open Liberty Transaction Manager support for InstantOn には、アプリケーション・プロセスの復元時の構成の更新に関する制限があります。 トランザクション・マネージャー の構成属性は、 InstantOn チェックポイントと復元の間で一定でなければなりません。 この制限は、 transaction サーバー構成エレメント ( recoveryGroup や recoverIdentityなど) で直接指定された構成属性にのみ当てはまります。 これらの構成属性の値は、チェックポイントと復元の間に変更してはなりません。

この制限は、以下の transaction 構成例のように recoverIdentity をパラメーター化できないため、 クラウド環境でのトランザクション・リカバリー が設計どおりに機能できないことを意味します。 この例では、アプリケーションのインスタンスごとに固有の recoverIdentity を指定します。

<transaction
  ...
  recoveryGroup="peer-group-name"
  recoveryIdentity="${HOSTNAME}${wlp.server.name}"
  ...
/>

チェックポイント前の Jakarta Transaction

Open Liberty InstantOn では、アプリケーション・プロセスのチェックポイントが実行される前にトランザクションを開始することはできません。 このシナリオは、トランザクションを開始しようとする 早期開始コード がアプリケーションにある場合に発生する可能性があります。 以下の Servletを検討してください。

@WebServlet(urlPatterns = { "/StartupServlet" }, loadOnStartup = 1)
public class StartupServlet extends HttpServlet {
    @Override
    public void init() {
        UserTransaction ut = UserTransactionFactory.getUserTransaction();
        try {
            ut.begin();
            ...
            ut.commit();
        } catch (Exception e) {
            // something went wrong
        }
    }

}

この Servlet の例では、 loadOnStartup = 1 属性を使用しています。 この属性を afterAppStart オプションと一緒に使用すると、サーブレットはチェックポイントの前に初期化されます。 ランタイムはこの競合を検出し、以下のメッセージをログに記録します。

[WARNING ] WTRN0155W: An application began or required a transaction during the server checkpoint request. The following stack trace for this thread was captured when the transaction was created:

この警告の後に、トランザクションを開始しようとしているアプリケーション・コードを識別するのに役立つスタック・トレースが続きます。 その後、サーバーはチェックポイント処理に失敗し、以下のエラーがログに記録されます。

WTRN0154E: The server checkpoint request failed because the transaction service is unable to begin a transaction.

この障害を回避するには、 beforeAppStart オプションを使用するか、早期開始コードを使用しないようにコンポーネントを変更します。 この例では、その変更は loadOnStartup = 1 属性を削除するためのものです。

MicroProfile Config へのアクセスが早すぎます

アプリケーションに早期開始コードがあり、 afterAppStart オプションを使用している場合、アプリケーション・プロセスのチェックポイントが実行される前に、 MicroProfile Config から構成値が注入される可能性があります。 アプリケーション・イメージ・コンテナーの実行時にこのような構成値が変更された場合、アプリケーションは、アプリケーション・プロセス・チェックポイントが実行されたときに設定された失効した値を使用する可能性があります。

Open Liberty ランタイムは、この状態を検出し、構成値が変更されたことを示すアプリケーション・コンテナー・イメージが実行されると、警告メッセージをログに記録します。 以下の例では、デフォルト値が theDefaultに設定された example_config 構成キーを使用します。 チェックポイントが発生すると、環境構成ソースを使用して MicroProfile 構成値にデータを設定することはできません。 構成のこの @Inject アノテーションが、チェックポイントが実行される前に作成されて使用される CDI Bean に含まれている場合、 theDefault の値が注入されます。

    @Inject
    @ConfigProperty(name = "example_config", defaultValue = "theDefault")
    String exampleConfig;

InstantOn アプリケーション・コンテナー・イメージが実行されると、環境変数 EXAMPLE_CONFIG を使用して、更新された値を指定できます。 ランタイムはこの値を検出し、以下のメッセージをログに記録します。

[WARNING ] CWWKC0651W: The MicroProfile configuration value for the key example_config has changed since the checkpoint action completed on the server. If the value of the key changes after the checkpoint action, the application might not use the updated value.

この状態では、 beforeAppStart チェックポイント・オプションを使用します。 もう 1 つのオプションは、 動的 ConfigSourceを使用することです。 前の例は、 exampleConfig 変数の Provider<String> タイプを使用して、動的 ConfigSource を使用するように変更できます。

    @Inject
    @ConfigProperty(name = "example_config", defaultValue = "theDefault")
    Provider<String> exampleConfig;

Provider<String> の get() メソッドを呼び出すたびに、 ConfigProperty アノテーションの現行値が返されます。 この動作により、アプリケーションは、 InstantOn アプリケーション・コンテナーの実行中にアプリケーション・プロセスが復元されたときに、更新された構成値にアクセスできます。

DataSource の注入が早すぎる

アプリケーションに早期開始コードがあり、 afterAppStart オプションを使用している場合は、アプリケーション・プロセスのチェックポイントが実行される前に、アプリケーションに DataSource が注入される可能性があります。 クラウド環境では、アプリケーション・イメージ・コンテナーの実行時に DataSource の構成を変更する必要がある可能性があります。 以下の Servlet の例を考えてみます。

@WebServlet(urlPatterns = "/ExampleServlet", loadOnStartup = 1)
public class ExampleServlet extends HttpServlet {
    @Resource(shareable = false)
    private DataSource exampleDataSource;
    ...
}

この Servlet の例では、 loadOnStartup = 1 属性を使用しています。 afterAppStart オプションを使用している場合、この属性は、チェックポイントの前にサーブレットを初期化します。 アプリケーションをクラウドにデプロイするときに、 DataSource に関連するデプロイメント情報の構成が必要になる場合があります。 以下の Open Liberty server.xml 構成を検討してください。

  <!-- these values are place holders so we don't have to have the env set before checkpoint -->
  <variable name="DB2_DBNAME" defaultValue="placeholder" />
  <variable name="DB2_HOSTNAME" defaultValue="placeholder" />
  <variable name="DB2_PASS" defaultValue="placeholder" />
  <variable name="DB2_PORT" defaultValue="45000" />
  <variable name="DB2_PORT_SECURE" defaultValue="45001" />
  <variable name="DB2_USER" defaultValue="placeholder" />


  <dataSource id="DefaultDataSource">
    <jdbcDriver libraryRef="DB2Lib"/>
    <properties.db2.jcc
      databaseName="${DB2_DBNAME}" serverName="${DB2_HOSTNAME}" portNumber="${DB2_PORT}"
      downgradeHoldCursorsUnderXa="true"/>
    <containerAuthData user="${DB2_USER}" password="${DB2_PASS}"/>
    <recoveryAuthData user="${DB2_USER}" password="${DB2_PASS}"/>
  </dataSource>

この構成では、データベース名、ホスト名、ポート、ユーザー、パスワードなどにプレースホルダー値を使用します。 この構成により、 Kubernetesで説明されているように、環境変数値またはその他の構成メカニズムを使用して値を更新することができます。 これらの構成は、アプリケーション・イメージにハードコーディングしてはならず、アプリケーションをクラウドにデプロイするときに更新できなければなりません。

チェックポイントの前にアプリケーションに DataSource が注入され、 DataSource の構成が変更された場合、更新された構成で InstantOn アプリケーション・コンテナー・イメージが実行されると、アプリケーションが再始動されます。 このシナリオを回避するには、 beforeAppStart オプションを使用するか、早期開始コードにならないようにコンポーネントを変更します。 この例では、その変更は loadOnStartup = 1 属性を削除するためのものです。

InstantOn

InstantOn は、 Open Liberty InstantOn のサポート対象フィーチャーで説明されているように、 Open Liberty フィーチャーのサブセットのみをサポートします。 InstantOn でサポートされているフィーチャー・セットの外部でパブリック・フィーチャーが有効になっていると、チェックポイントが失敗し、以下の例のようなエラー・メッセージが表示されます。

CWWKC0456E: A checkpoint cannot be taken because the following features configured in the server.xml file are not supported for checkpoint: [usr:exampleFeature-1.0]

このエラーは、 InstantOnでサポートされていない構成済みフィーチャーに対して発生します。 この制限には、Liberty 製品拡張および Liberty ユーザー・フィーチャーが含まれます。

bootstrap.properties ファイルを使用した構成の更新

InstantOn アプリケーション・コンテナー・イメージの実行時に、 bootstrap.properties ファイルが読み取られません。 InstantOn アプリケーション・コンテナー・イメージの実行時に構成可能でなければならない値は、代替ソースから取得する必要があります。 例えば、 Kubernetesで説明されているように、環境変数やその他の構成メカニズムを使用できます。

Java SecurityManager はサポートされていません

Open Liberty が SecurityManagerで実行されるように構成されている場合、 InstantOn は、チェックポイント中にこの構成を検出し、以下のメッセージを出して失敗します。

CWWKE0958E: The server checkpoint request failed because the websphere.java.security property was set in the bootstrap.properties file. This property enables the Java Security Manager and is not valid when a server checkpoint occurs.

JVM オプションの更新

InstantOn は、 InstantOn アプリケーション・プロセスをリストアする際の jvm.options の変更をサポートしていません。 JVM プロセスに設定する必要がある JVM オプションは、 InstantOn コンテナー・イメージのビルド中に定義する必要があります。

IBM Semeru JVM では、 OPENJ9_RESTORE_JAVA_OPTIONS 環境変数を使用してリストア時に JVM オプションを設定するためのサポートは制限されています。 詳しくは、Java Checkpoint/Restore In Userspace (CRIU) support の資料を参照してください。

SELinux の制限

SELinux モードが enforcingに設定されている場合、イメージ・テンプレート Dockerfile または Containerfileで checkpoint.sh スクリプト を使用すると、SELinux は InstantOn がアプリケーション・プロセスのチェックポイントを実行できない可能性があります。 virt_sandbox_use_netlink SELinux 設定が無効になっている場合、必要な netlink Linux システム・コールがブロックされます。 このブロックは、コンテナー・イメージのビルド中に InstantOn がアプリケーション・プロセスのチェックポイントを実行しないようにします。 Open Liberty InstantOn は、この制限を検出し、以下のメッセージをログに記録します。

CWWKE0962E: The server checkpoint request failed. The following output is from the CRIU /logs/checkpoint/checkpoint.log file that contains details on why the checkpoint failed.
Warn  (criu/kerndat.c:1103): $XDG_RUNTIME_DIR not set. Cannot find location for kerndat file
Error (criu/libnetlink.c:84): Can't send request message: Permission denied
..
Error (criu/cr-dump.c:2099): Dumping FAILED.
CWWKE0963E: The server checkpoint request failed because netlink system calls were unsuccessful. If SELinux is enabled in enforcing mode, netlink system calls might be blocked by the SELinux "virt_sandbox_use_netlink" policy setting. Either disable SELinux or enable the netlink system calls with the "setsebool virt_sandbox_use_netlink 1" command.

この制限を回避するには、 setsebool virt_sandbox_use_netlink 1 コマンドを使用して virt_sandbox_use_netlink SELinux 設定を有効にするか、SELinux enforcing モードを無効にします。 この問題を回避するためのもう 1 つのオプションは、 3 ステップ・プロセス を使用して InstantOn イメージをビルドすることです。 3 ステップのプロセスでは、アプリケーション・プロセス・チェックポイントを実行する実行中のコンテナーに対する netlink システム呼び出しへのアクセス権限を付与する --privileged コンテナーを使用する必要があります。

山 Linux セキュリティー・モジュールの制限

Yama が以下のいずれかのモードで構成されている場合、 InstantOn は、実行中のコンテナーでアプリケーション・プロセスのチェックポイントを指定することも復元することもできません。

  • 2 -管理者専用接続

  • 3 -接続なし

この構成が存在する場合、 /logs/checkpoint/restore.log には以下のエラーが含まれます。

Error (criu/arch/x86/kerndat.c:178): 32: ptrace(PTRACE_TRACEME) failed: Operation not permitted

InstantOn チェックポイントおよび復元を機能させるには、以下のいずれかのモードで Yama を構成する必要があります。

  • 0 -従来の ptrace 権限

  • 1 -制限付き ptrace

以下のサポートされるパブリック・クラウドの Kubernetes サービスでは、Yama のデフォルトが 1 モードに設定されています。これにより、 InstantOn はデフォルトでチェックポイントを設定し、リストアすることができます。

  • Amazon Elastic Kubernetes Service (EKS)

  • Azure Kubernetes Service (AKS)

Linux システム・コールへのアクセス

必須の Linux システム・コールで説明されているように、CRIU では、アプリケーション・プロセスを復元するためにいくつかの Linux システム・コールが必要です。 この要件では、 InstantOnを使用するときに、実行中のコンテナーに必要なシステム呼び出しを付与するために、追加の構成が必要になる場合があります。

以下の例は、特定のシステム・コールへのアクセスがブロックされたときに /logs/checkpoint/restore.log ファイルに記録されるエラーです。

ブロックされた clone3 システム・コール
Error (criu/kerndat.c:1377): Unexpected error from clone3: Operation not permitted
ブロックから ptrace へのシステム・コール
Error (criu/arch/x86/kerndat.c:178): 28: ptrace(PTRACE_TRACEME) failed: Operation not permitted
ブロックから vmsplice へのシステム・コール
Error (criu/pipes.c:184): 0x4c11a: Error splicing data: Operation not permitted

現在、サポートされているパブリック・クラウドの Kubernetes Service 環境では、CRIU によってデフォルトで使用される必要なシステム・コールが許可されています。 以下のクラウド・プロバイダーを使用する場合、追加の構成は必要ありません。

  • Amazon Elastic Kubernetes Service (EKS)

  • Azure Kubernetes Service (AKS)

必要な Linux 機能を使用しない実行

必要な Linux 機能 が付与されていない場合、チェックポイントおよびリストア中にエラーが発生します。 チェックポイントに必要な機能が付与されていない場合は、 InstantOn コンテナー・イメージのビルド中に以下のエラーが発生します。

Can't exec criu swrk: Operation not permitted
Can't read request: Connection reset by peer
Can't receive response: Invalid argument
[ERROR   ] CWWKC0453E: The server checkpoint request failed with the following message: Could not dump the JVM processes, err=-70

Operation not permitted メッセージは、必要な Linux 機能が付与されていないことを示します。 checkpoint.sh スクリプトを使用している場合、 RUN checkpoint.sh 命令の実行中に以下のエラーが発生します。

Error: building at STEP "RUN checkpoint.sh afterAppStart": while running runtime: exit status 74

このエラーを回避するには、コンテナー・イメージ・ビルドに CHECKPOINT_RESTORE、 SYS_PTRACE、および SETPCAP Linux capabilitiesを付与します。 3 ステップ・プロセス を使用してコンテナー・イメージをビルドする場合は、チェックポイント・ステップを実行しているコンテナーが --privileged コンテナーであることを確認してください。

リストアに必要な機能が付与されていない場合、 InstantOn アプリケーション・コンテナー・イメージを実行しようとすると、以下のエラーが発生します。

/opt/ol/wlp/bin/server: line 1430: /opt/criu/criu: Operation not permitted
CWWKE0961I: Restoring the checkpoint server process failed. Check the /logs/checkpoint/restore.log log to determine why the checkpoint process was not restored. Launching the server without using the checkpoint image.

Operation not permitted メッセージは、リストアに必要な Linux 機能が付与されていないことを示しています。

サポートされるプロセッサー

現在、サポートされているプロセッサーは X86-64/AMD64のみです。 その他のプロセッサーは、 Open Liberty InstantOnの以降のリリースでサポートされる予定です。
