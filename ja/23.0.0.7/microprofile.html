MicroProfile

マイクロサービス・アーキテクチャーは、各機能が独立したサービスとして開発されるクラウド・ネイティブ・アプリケーションを構築するための一般的なアプローチです。 これにより、小規模の自律型チームは、それぞれのサービスを個別に開発、デプロイ、および拡張することができます。

MicroProfile は、クラウド・ネイティブ Java™ マイクロサービスを作成できるように設計されたテクノロジーのモジュラー・セットです。 この概要では、クラウド・ネイティブ・マイクロサービスの開発と管理に MicroProfile がどのように役立つかを説明します。 次に、 Open Liberty MicroProfile ガイドに従って MicroProfile の実地体験を行い、 Open Libertyを使用してマイクロサービスをビルドできるようにします。

クラウド・ネイティブ Java マイクロサービスを大規模に迅速に導入

クラウド・ネイティブは、アプリケーションを開発してクラウドに大規模に迅速にデプロイするための業界全体のアプローチです。 クラウド・ネイティブ・アプリケーションは、チームに合わせて調整されたマイクロサービスを中心に設計されており、アジャイル・プラクティスと継続的統合/継続的デリバリー (CI/CD) を使用して開発され、導入を合理化します。 クラウド・プラットフォームを提供するさまざまなベンダーにより、ベンダーの囲い込みを回避するためには、オープン・ソースとオープン・スタンダードが不可欠です。

MicroProfile を使用すると、クラウド・ネイティブ Java アプリケーションを、それぞれが 1 つの固有のビジネス機能を表す、疎結合の軽量サービスとして開発およびデプロイすることができます。 このアプローチはモジュール式であり、アプリケーションを理解しやすく、開発しやすく、テストしやすく、保守しやすくします。 アプリケーションでさらに多くの機能が必要な場合は、 Java EE フィーチャーも追加できます。 Open Libertyで MicroProfile 2.0 以降を使用している場合は Java EE 8 フィーチャーを追加し、 Open Libertyで MicroProfile 1.x を使用している場合は Java EE 7 フィーチャーを追加します。

MicroProfile 5.0 は Jakarta EE 9.1 に対応しています。 このリリースでは、アプリケーションは Jakarta EE 9.1 API とともに MicroProfile API を使用することができます。 MicroProfile 5.0 は、javax.* から jakarta.*への Jakarta API パッケージ名の変更を考慮した依存関係の更新を除き、MicroProfile 4.1 と同じ機能を持っています。

MicroProfile 6.0 は Jakarta EE 10 と連携します。 これには Jakarta EE 10 Core Profile が含まれ、 MicroProfile OpenTracing を MicroProfile Telemetry に置き換えます。 したがって、 MicroProfile OpenTracing はアンブレラ・リリースから移動し、スタンドアロン仕様になります。 Jakarta EE 10 Core Profile について詳しくは、 Core Profile specificationを参照してください。

マイクロサービス・アーキテクチャーの実動の利点

MicroProfile が一般的な理由は、コードをより迅速に作成することだけではありません。 また、各サービスは個別に作成および管理されるため、アプリケーションをより詳細なレベルで拡張することもできます。 トラフィックの多いサービスは、システム全体をスケールアップするのではなく、リソースを効率的に使用するように個別にスケーリングすることができます。 もう 1 つの利点は、1 つのサービスの障害をシステムの他の部分から分離できることです。 サービスに障害が発生した場合、独立しているサービスは影響を受けませんが、従属サービスはフォールト・トレランス・ストラテジーを使用して、障害が他のサービスにカスケードしないようにすることができます。

マイクロサービス・アーキテクチャーには多くの利点がありますが、従来のモノリシック・アプリケーションでは明らかになっていない新しい課題も生じています。 MicroProfile は、クラウド・ネイティブ・アプリケーションを容易に開発できるように、これらの課題に対処します。 これらの課題には、サービス障害の許容、一連のマイクロサービスを介して流れる認証済みユーザー要求のエンドツーエンド・セキュリティー、および多数のサービスにわたる要求の問題判別が含まれます。

MicroProfile は、クラウド・ネイティブ Java マイクロサービスの開発を簡素化します。

ほとんどのクラウド・ネイティブ・マイクロサービスは REST API に基づいており、最下部の層が最も重要になっています。 MicroProfile は、その基盤として、REST API の開発と使用を容易にする一連のテクノロジーを提供します。 MicroProfile は、 Java EE API (JAX-RS、CDI、JSON-B、および JSON-P) の小さなセットを受け取り、それらを単純なタイプ・セーフな REST クライアント API で拡張して、REST サービスの取り込みを容易にします。 MicroProfile Reactive Streams Operators は、マイクロサービス用の MicroProfile リアクティブ・プログラミングのためのイネーブリング・テクノロジーです。 この機能により、ストリームに適用される一連の標準オペレーターとともに Reactive Streams API をバンドルすることで、リアクティブ・ストリームの作成と操作が可能になります。

ビルド REST サービス

JAX-RS、CDI、JSON-B、および JSON-P Java EE テクノロジーは、 MicroProfileのベースを提供します。 Java EE および MicroProfileを初めて使用する場合は、これを開始することをお勧めします。 JAX-RS は、リソース・クラスを作成し、適切なアノテーションを追加して必要な Web エンドポイントを作成することにより、REST API を作成できるようにする Java API です。 Context and Dependency Injection (CDI) は、オブジェクトを直接作成したりファクトリーを使用して検出したりするのではなく、 @Inject アノテーションを使用して必要な依存関係をオブジェクトに提供します。 JSON-P および JSON-B を使用すると、JSON との間でクラスを自動的にシリアライズおよびデシリアライズすることが容易になります。

今すぐ試す: JAX-RS、JSON-P、および Open Liberty を使用して REST サービスを作成する方法については、 RESTful Web サービスの作成のガイドを参照してください。

タイプ・セーフ Java を使用した REST サービスのコンシューム

MicroProfile Rest Client は、HTTP を介して REST サービスを呼び出すためのタイプ・セーフなアプローチを提供します。 この API は、JAX-RS によって定義されたクライアント・サイド API を大幅に単純化します。 MicroProfile Rest Client は、クライアントとサービスの間の通信を処理します。 定義してアノテーションを付ける必要があるのは、REST リソースに対して実行する必要があるアクションを記述するインターフェースのみです。 CDI を使用してクライアントを従属クラスに注入すると、このインターフェースの実装が自動的に生成されます。

今すぐ試す: テンプレート・インターフェースを使用した RESTful サービスの利用のガイドで、 MicroProfile Rest Client を使用して HTTP を介して RESTful マイクロサービスをタイプ・セーフな方法で呼び出す方法について説明しています。

詳しくは、 REST クライアントを参照してください。

MicroProfile は、組織のスケーリングを簡素化します。

数百の自律型サービス、コラボレーション・サービス、および頻繁に進化するサービスを処理することで、多くの新しい課題が生じます。 これらの課題には、例えば、チーム間での API の文書化と共有、サービス間でのセキュリティーの伝搬、ネットワークまたはサービスの障害の処理、サービス更新の継続的な統合とデプロイなどが含まれます。 MicroProfile フィーチャーの中間層には、これらのタスクを単純化するための多数の API が用意されています。

REST API の文書化

MicroProfile OpenAPI は、REST API の API 文書を公開するために使用できる OpenAPI 仕様用の Java API を提供します。 JAX-RS アプリケーションから OpenAPI 文書をネイティブに作成できます。 OpenAPI は、 Swagger 仕様の標準化です。

今すぐ試す: RESTful API の文書化のガイドで MicroProfile OpenAPI を使用して RESTful API を文書化およびフィルタリングする方法を検討してください。

詳しくは、 OpenAPIを使用した API の資料を参照してください。

マイクロサービスでの予期しない障害の処理

MicroProfile Fault Tolerance は、依存するサービスでの予期しない障害に対処するための堅固な動作を構築するための API およびアノテーションを提供します。 フォールト・トレランスの側面には、タイムアウト、再試行、フォールバック、バルクヘッド処理、回路ブレーカーなどがあります。

今すぐ試す: チームが作成した以下のガイドで、 MicroProfile フォールト・トレランスを使用して障害の影響を管理する方法を検討してください。

  • @Fallback アノテーションを使用したフォールト・トレラント・マイクロサービスのビルド

  • 障害の高速化とエラーからのリカバリー

  • マイクロサービスに対する失敗した呼び出しの繰り返しの防止

  • マイクロサービスへの同時要求数の制限

詳しくは、 マイクロサービスでのフォールト・トレランスを参照してください。

認証および役割ベースのアクセス制御の管理

MicroProfile JSON Web Token (JWT) は、サービスの相互運用可能な認証および役割ベースのアクセス制御を提供します。 これにより、認証済み JWT トークンを、それらのサービスが複数のベンダー実装で実行されている場合でも、複数のマイクロサービス間で共有することができます。 また、JWT トークン内で渡されるユーザーおよび役割情報に基づいて、マイクロサービス操作へのアクセスを制御することもできます。

今すぐお試しください。 JSON Web トークンを使用したマイクロサービスの保護に関するガイドで、 MicroProfile JWT を使用してマイクロサービスへのユーザーおよび役割のアクセスを制御する方法を説明しています。

詳しくは、 シングル・サインオン: JSON Web Tokenを参照してください。

移植性を向上させるための構成の外部化

MicroProfile Config は、アプリケーションの移植性を向上させるために、アプリケーションからの構成を外部化します。 コア・プリンシパルは、システム・プロパティーおよび環境変数を使用して、デプロイメント時に構成をオーバーライドできるようにすることです。 この機能により、デプロイメントごとに構成を変更することで、マイクロサービスを一度作成し、CI/CD パイプラインを介して何度もデプロイすることができます。

今すぐお試しください: MicroProfile Config を使用してマイクロサービスを構成する方法については、以下のチームが作成したガイドを参照してください。

  • マイクロサービスの構成

  • マイクロサービス内のコードからの構成の分離

詳しくは、 マイクロサービスの外部構成を参照してください。

MicroProfile は、観察事項のマイクロサービスを作成するのに役立ちます。

数百のマイクロサービスを処理するには、強力な運用フォーカスが必要です。 システムで問題が発生し始めた場合、要求が数十または数百のサービスに及ぶ可能性がある根本原因をどのように追跡しますか? どのサービスがうまく実行されていないのか、または要求がそれらのマイクロサービスを通過したジャーニーを理解するにはどうすればよいでしょうか? MicroProfile フィーチャー・セットの上層は、これらの質問に答えるのに役立ちます。 これは、サービスの正常性、サービスの実行状況、およびサービスを介して要求がどのように流れているかを理解するのに役立つ API を提供します。

マイクロサービスの可用性の判別

MicroProfile Health Check は、マイクロサービスが正常であるかどうかを判別するための一般的な REST エンドポイント・フォーマットを提供します。 正常性はサービス自体によって判別でき、必要なリソース (例えば、データベース) およびサービスの可用性に基づいている可能性があります。 サービス自体は実行されている可能性がありますが、通常の操作に必要なものが使用できない場合は、正常性がないと見なされます。 また、ヘルス・チェック・エンドポイントは、 Kubernetes の活性プローブおよび作動可能プローブに簡単に統合できるように設計されています。

今すぐ試す: チームが作成した以下のガイドで、 MicroProfile Health を使用してマイクロサービスの正常性を報告および確認する方法を検討してください。

  • マイクロサービスへのヘルス・レポートの追加。

  • Kubernetesでのマイクロサービスの正常性の確認。

詳しくは、 マイクロサービスのヘルス・チェックを参照してください。

マイクロサービスのテレメトリー・データのモニター

MicroProfile Metrics は、実行中のマイクロサービスのテレメトリー・データをモニターするための一般的な REST エンドポイントを提供します。これは、JMX と似ていますが、JAX-RS を使用するはるかに単純な API です。 JSON または Prometheus テキスト・フォーマットの出力を使用して、組み込みメトリックとアプリケーション定義メトリックの両方にアクセスできます。 この API は、 MicroProfile Health によって提供される単純なアップ/ダウン・レポートよりも詳細な情報を提供します。

今すぐお試しください。 マイクロサービスからのメトリックの提供に関するガイドの MicroProfile Metrics を使用して、マイクロサービスからのシステムおよびアプリケーションのメトリックを提供する方法を検討してください。

詳しくは、 Microservice 可観測性とメトリックを参照してください。

マイクロサービスの分散トレースを使用可能にします。

MicroProfile Telemetry は、 OpenTelemetry プロジェクトに基づいています。これは、トレース・データを作成および管理するためのオープン・ソースのベンダーに依存しないツール、API、および SDK のコレクションです。

Open Libertyでは、 MicroPRofile Telemetry は MicroProfile Telemetry フィーチャーによってサポートされます。 この機能で収集したデータを Jaeger や Zipkinなどのトレース・システムにエクスポートできます。 詳しくは、 MicroProfile Telemetry による分散トレースの有効化を参照してください。

MicroProfile Telemetry は、 MicroProfile OpenTracingに代わるものです。 MicroProfile OpenTracing から MicroProfile Telemetry へのアプリケーションのマイグレーションについては、 Differences between MicroProfile Telemetry 1.0 and MicroProfile OpenTracing 3.0を参照してください。

その他の MicroProfile フィーチャー

MicroProfile スタックを構成する MicroProfile フィーチャーの 3 つの層に加えて、これらの追加の MicroProfile フィーチャーは Open Liberty ランタイムに実装されますが、まだ正式な MicroProfile リリースの一部ではありません。 これらのフィーチャーは、 MicroProfile コア機能を拡張して、マイクロサービス・ベースのアプリケーションでの並行性およびリアクティブ・プログラミングの制御を強化します。

マイクロサービスでの並行タスクの管理

並行性 (複数のタスクを同時に調整および追跡する機能) は、マイクロサービス・ベースのアプリケーションにおける重要な機能です。 MicroProfile Context Propagation は、並行タスク間のコンテキスト認識を提供することにより、Java SE および Java EE の並行性サポートを拡張します。これにより、アプリケーション全体での整合性と可視性が向上します。 MicroProfile Context Propagation を使用すると、アプリケーションは、イベントが発生したときに、信頼できるスレッド・コンテキストの下で、Liberty スレッド化のパフォーマンスによって支えられて、イベントに反応します。

マイクロサービス間でのメッセージの送受信

MicroProfile Reactive Messaging は、アプリケーション内のマイクロサービス間でメッセージを送信、受信、および処理するための簡単な方法を提供します。 また、メソッドを外部メッセージング・システムに接続できるようにするコネクター API も提供します。 MicroProfile Reactive Messaging を使用すると、Liberty は、アノテーションが付けられたアプリケーション Bean メソッドをリアクティブ・ストリーム互換のパブリッシャー、サブスクライバー、およびプロセッサーに変換し、それらを相互に接続します。

今すぐ試す: MicroProfile Reactive Messaging を使用したマイクロサービス間のメッセージの送受信 に関するブログ投稿で、マイクロサービスと外部システムとの間のメッセージングを有効にする方法を検討してください。

マイクロサービス用のリアクティブ・プログラミングの実装

MicroProfile Reactive Streams Operators は、非同期データ・ストリームのフロー制御および洗練されたエラー処理を提供するパブリッシュ/サブスクライブ・パイプラインを実装する手段を提供します。 この実装により、パブリッシャーがダウンストリーム・サブスクライバーを圧倒することなく、データ・ストリームの処理が可能になります。 リアクティブ・ストリーム・オペレーターは、パブリッシャーとサブスクライバーの間に「バック・プレッシャー」をもたらすチケット・システムを使用することで、潜在的な過負荷状態下での効率を向上させることができます。 基礎となるテクノロジーに関係なくパブリッシャーとサブスクライバーを直接接続できるため、Reactive Streams Operator は、さまざまなテクノロジーを効率的に結合できる抽象化として機能します。
